# まあみ UIデザインフロー

> **フロー概要:** claude_design でUI案を生成・選定 → COCOSiLプロジェクトへ移管・適用  
> **まあみがやること:** Step 3（案の選定）と Step 5（目視確認）だけ。それ以外はAIパートナーが実行する。

---

## 一本道フロー（全6ステップ）

```
Step 1  AIに「〇〇のUI生成して」と依頼
           ↓ AIが components/ をスキャン → テンプレートに自動充填 → claude_designに送る
Step 2  AIが Gate 1チェックを実行して結果を報告
           ↓ OK → Step 3へ  /  NG → AIが修正案を提示 → 再生成
Step 3  まあみが「どの案が好き？」で選定  ← まあみの判断①（唯一の選定）
           ↓
Step 4  AIが /start-task → コンポーネント単位で移管 → Gate 2チェック（typecheck）を自動実行
           ↓ typecheck通過 → Step 5へ  /  エラー → AIが修正して再実行
Step 5  まあみが pnpm dev で目視確認  ← まあみの判断②（「動いてる？」だけ）
           ↓ OK → Step 6へ  /  NG → AIに「〇〇が変」と伝えて修正依頼
Step 6  AIが /finish-task → PR作成 → えんまさをレビュアーにアサイン（非同期）
           ↓
        DONE — まあみは次のコンポーネントへ
```

**えんまさへの確認はPRレビューで非同期に行う。まあみは確認の返事を待たなくてよい。**

---

## Step 1: 生成依頼

まあみはAIパートナーに自然言語で依頼するだけ:

```
「〇〇コンポーネントのUI案を2つ生成して」
「診断結果カードのデザインをclaudie_designで作って」
```

AIパートナーは以下を自動で実行する:

1. `ls components/` で既存コンポーネント一覧を取得
2. [テンプレート](../input/setup/claude_design_prompt_template.md) を読み込み
3. 「既存コンポーネント一覧」を動的に充填してclaudie_designに送る

**まあみがやること: 依頼を伝えるだけ。テンプレートの更新不要。**

---

## Step 2: Gate 1チェック（AIが実行）

案が返ってきたらまあみはAIに「Gate 1チェックして」と伝えるだけ。

AIパートナーが以下の12項目をすべて確認し、結果を報告する:

### UXシーケンス整合（4項目）

- [ ] 画面・フローが **共感** から始まっているか
- [ ] **安心** が共感の後に来ているか
- [ ] **分析・診断結果** は安心の後に表示しているか（分析が先になっていない）
- [ ] **行動促進（CTA）** は分析の後に来ているか

### 禁止語（5項目）

UIコピー・ボタンラベル・エラーメッセージのすべてで確認:

- [ ] 「占い」「占い師」が含まれていないか
- [ ] 「鑑定」「鑑定士」が含まれていないか
- [ ] 「運勢」「運命」が含まれていないか
- [ ] 「当たる」（精度の意味で）が含まれていないか
- [ ] 「霊感」「霊視」が含まれていないか

代替表現: 性格分析 / パーソナリティ診断 / 統合レポート / 傾向 / 特性 / 一致度

### ビジュアルトーン（3項目）

- [ ] 急かす・焦らせる表現・デザインになっていないか
- [ ] 過度に派手・チャラい・怪しいトーンになっていないか
- [ ] 「根拠のある性格分析」というポジショニングと合致しているか

### AIが報告する形式

```
Gate 1チェック完了

✅ UXシーケンス: クリア
✅ 禁止語: クリア
⚠️ ビジュアルトーン: 「今すぐ診断！」のCTAが急かす印象。→「診断をはじめる」に修正を提案

判定: NG（軽微）
対処: 上記の修正をClaudie_designに依頼してください。
```

### AIの判定3パターン

| 判定 | 条件 | AIの動作 |
|---|---|---|
| **OK** | 12項目すべてクリア | 「Gate 1クリア。Step 3へどうぞ」 |
| **NG（軽微）** | 禁止語・トーンの軽微な問題 | 修正案を提示してclaudie_designへの指示文を生成する |
| **NG（要フラグ）** | UXシーケンス全体への影響・診断結果の見せ方の根本変更 | 「PRに⚠️フラグを立てます。まあみは次に進んで構いません」 |

**⚠️ NG（要フラグ）の場合も、まあみの作業は止まらない。** AIがPRに「えんまさ確認必要」のフラグを自動で記載する。

---

## Step 3: 選定（まあみが実行）

Gate 1 OKの案の中から「どれが好き？」で選ぶ。正解はない。COCOSiLらしさの審美判断をまあみが行う。

---

## Step 4: 移管（AIが実行）

まあみが選定を伝えたらAIパートナーが以下を実行する:

```bash
# AIが実行するステップ（まあみは指示するだけ）
/start-task <コンポーネント名>

# 既存コンポーネントとの重複チェック
ls components/
find components/ -name "*.tsx" | xargs grep -l "<コンポーネント名>" 2>/dev/null

# コンポーネントファイルの配置（components/配下、コンポーネント単位）
# → Tailwind v4記法チェック（v3記法を自動修正）
# → importパスチェック（src/削除・zod/v4変換）

# 型安全性確認
pnpm typecheck
```

**移管単位: 必ずコンポーネント単位（画面まるごとは禁止）。**  
1コンポーネント = 1コミットが理想。

### AIが自動修正するTailwind v4記法

| v3（NG） | v4（OK） |
|---|---|
| `bg-opacity-50` | `bg-black/50` |
| `text-opacity-75` | `text-black/75` |
| `border-opacity-*` | `border-black/*` |
| `ring-opacity-*` | `ring-black/*` |
| `divide-opacity-*` | `divide-black/*` |

---

## Step 5: 目視確認（まあみが実行）

```bash
pnpm dev
# → http://localhost:3000 でデザイン意図通りに表示されているか確認
```

確認項目はシンプルに3つだけ:

- デスクトップ表示でレイアウトが崩れていないか
- モバイル表示（375px幅）でレイアウトが崩れていないか
- Clerkコンポーネントとのスタイル競合がないか

問題があればAIに「〇〇が変」と伝えて修正依頼。問題なければ Step 6 へ。

---

## Step 6: PR作成（AIが実行）

```bash
/finish-task
```

AIが以下のPRを自動生成して、えんまさをレビュアーにアサインする:

```
## 変更内容
<コンポーネント名> を追加

## Gate 1チェック結果
（AIが自動記入）

## スクリーンショット
（デスクトップ / モバイル）

## ⚠️ えんまさ確認必要（該当時のみ）
<理由>
```

**まあみはPR作成後、えんまさの返事を待たずに次のコンポーネントへ進んでよい。**

---

## えんまさレビューが来たら

えんまさからPRへのコメントが来たら対応する。`⚠️ えんまさ確認必要` フラグがついたPRはえんまさのApprove後にマージする。フラグなしのPRはえんまさのApprove次第でマージ（または任意）。
