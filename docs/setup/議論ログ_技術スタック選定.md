## Turn 1｜現状診断：V2が今持っている「見えない負債」

**河野：** まず現状を整理します。create-next-appのボイラープレートにNext.js 16.2.1、React 19、Tailwind 4、TypeScript 5。依存関係はほぼゼロ。一方でdocs/setupには4ファイル・合計1,700行以上の設計指針がある。

ここで**具体的な矛盾を3つ**挙げます。

1. **パッケージマネージャの不整合**: CI設計（COCOSiL_github_prompt.md）はpnpmキャッシュ前提で書かれているのに、プロジェクトはnpmで初期化されている。
2. **context7の矛盾**: プロジェクトのCLAUDE.mdには「Use context7 for library/framework documentation」と明記されているのに、`plugin_setup.md`では「context7はスキップ、Month 1+で再検討」と書かれている。
3. **プラグインの時間経過**: setup文書は2026年初頭の執筆。2026年5月現在、推奨プラグインの一部は統合・非推奨化している可能性がある。

**安藤：** 問題の本質はバージョンの新しさではない。V1が0.5GBで破綻した根本原因は「データ量の見積もりなしにチャット履歴を保存し続けた」設計判断です。

今V2が直面している本当の問題は「**設計指針ドキュメントと実装環境の同期が取れていない**」こと。1,700行の設計指針は優秀ですが、実際のpackage.jsonやsettings.jsonに反映されていない。ドキュメントと現実の乖離こそが、2人チームを最も消耗させます。

**みさき：** え待って、つまり設計書はめっちゃ充実してるのに、実際のプロジェクトはcreate-next-appのまんまってこと？それってレシピ本だけ集めてキッチンが空っぽみたいな状態じゃないですか〜？

## Turn 2｜評価フレームワーク：選定基準を立てる

**安藤：** 2人チーム・MVP・Supabase無料枠という制約下で、**3軸評価**を使います。

| 評価軸 | 定義 | COCOSiLでの重み |
|---|---|---|
| **Survivability（生存性）** | 6ヶ月後にメンテ不能にならないか | 🔴 最重要 |
| **Velocity（速度）** | 2人で4週間スプリントを回せるか | 🔴 最重要 |
| **Ceiling（天井）** | スケール時に技術が足かせにならないか | 🟡 中程度 |

**河野：** これにToken Economy（トークン経済性）を加えます。7つの外部プラグインをフル接続すると約8,500トークンが固定オーバーヘッドとして消費される。

**追加必須ライブラリの評価:**

| 技術 | 推奨 |
|---|---|
| Zod | 🔴必須（API-First方針の要） |
| @supabase/ssr | 🔴必須（Next.js SSR対応） |
| @clerk/nextjs | 🔴必須（F4要件） |
| openai SDK | 🔴必須（F3要件、JIT追加） |

## Turn 3｜プラグイン選定：本当に7つ必要か

**河野：** **7つの外部プラグインは2人チームには過剰です。**

理由：feature-devとsuperpowersの機能重複、7プラグインのメンテコスト、context7問題（CLAUDE.mdとの矛盾）。

推奨は**3+1構成**:
- context7（接続済みMCP）
- commit-commands（Day 1）
- supabase MCP（要認証）
- code-review（Week 2）

**安藤：** playwrightをスキップリストから外すべき。F3（共感チャット）の3フェーズ会話フローはE2Eテストが必須。ただし外部プラグインとしてのインストールは不要 — 既に組み込みMCPとして利用可能。

**みさき：** え、じゃあplaywrightって最初から入ってるってこと？setup文書が「スキップ」って書いてるのは外部プラグインとしては要らないけど機能自体は使えるってことですね！なるほど〜

## Turn 4｜Hooks・SKILL・パッケージマネージャ：運用設計の法則

**河野：** **5つ全部のHooksをDay 1から入れるのは過剰です。**

「Hook地獄」回避のルール: 1つのHookを2週間運用して信頼を確立してから次を追加する。Day 1はPreToolUseの1つのみ。

**安藤：** パッケージマネージャは**今すぐpnpmに移行すべき**。CI設計がpnpm前提。移行手順：`rm -rf node_modules package-lock.json && pnpm install`。

**河野：** 自作SKILL 6つのうち、**Day 1に必要なのはlanguage-designの1つだけ**です。

```
SKILL導入の法則：Just-In-Time Authoring
SKILLは「使う直前に書く」。先回りして6つ一気に作ると、
実装時の学びが反映されず、メンテ負債になる。
```

**みさき：** あ〜それめっちゃわかります！V1でドキュメント先に全部書いて、結局実装したら全然違うものになって…ってなったんですよ〜。language-designだけ最初に必要ってことは、「占い」って言わないルールがClaude Codeにも染み込んでないとダメってことですよね？

## Turn 5｜設計原則と明日からのアクション

**安藤：** COCOSiL V2のセットアップにおける設計原則を3つに集約します。

### COCOSiL V2 セットアップ3原則

**① Align Before Add（追加する前に整合させる）**
ドキュメントと実装環境の矛盾を先に解消する。npm→pnpm移行、context7の矛盾解消、プラグイン数の見直し。

**② Minimal Viable Toolchain（最小実行可能ツールチェーン）**
Day 1: 外部プラグイン1個 + Hook 1個 + SKILL 1個。残りはJIT追加。

**③ Contract Before Code（コードの前に型契約）**
Zodスキーマ + Supabase型生成（`supabase gen types`）をSingle Source of Truthにする。

**河野：** Day 1 アクションリスト:

| # | アクション |
|---|---|
| 1 | pnpm移行 |
| 2 | `pnpm add zod @supabase/supabase-js @supabase/ssr` |
| 3 | PreToolUse Hook設定（秘密情報検出） |
| 4 | `supabase init` |
| 5 | language-design SKILL.md作成 |
| 6 | `lib/env.ts` 環境変数バリデーション作成 |

**みさき：** なんか最初は「設計書1,700行もあるのに何もできてない」って不安だったんですけど、結局Day 1にやることって6個だけなんですね！しかもプラグイン7つ→1つ、Hook 5つ→1つ、SKILL 6つ→1つ。

てか「追加する前に整合させる」ってマジ名言じゃないですか？あとcontext7とplaywrightが実はもう入ってたの、今日一のサプライズでした。
