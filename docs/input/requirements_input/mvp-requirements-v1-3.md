# COCOSiL MVP 要件定義書 v1.3

## L0
**APIファーストとモック並列開発を採用し、意味設計・見た目設計・構造設計のレイヤー別分業を明確化したCOCOSiLのMVP要件定義書。**

## L1
- **MVPの目的**: 共感AI×性格分析の組み合わせにより、7日以内再訪率30%以上というPMF仮説を検証する。
- **新開発体制(v1.3)**: えんまさ(意味/言語/AI)、まあみ(見た目/UI)、ヒラメ(構造/API)のレイヤー別垂直分業へ移行。
- **開発アプローチ**: APIインターフェースを先行共有し、モックJSONで全画面UIを完成させてから接続する「APIファースト×モック並列開発」を採用。
- **言語設計の徹底**: ターゲットの抵抗感を減らすため、「占い」用語の使用を回避する言語ルールを全実装レイヤーに適用。
- **リスク対策**: まあみのUI実装ブロックを防ぐため、ヒラメによる骨格の先渡しとサポート体制、えんまさの実装介入制限をルール化。

## 1. エグゼクティブサマリー

### 1.1 MVPの目的

COCOSiL MVPは、**「共感AI × 多角的性格分析で、ユーザーは継続的に利用するか」** という仮説を検証する。同時にBOOST第8期コンテストのデモとしても機能する。

### 1.2 検証仮説

| 優先度 | 仮説 | 成功指標 |
|--------|------|----------|
| ★ 最重要 | 性格分析を踏まえた共感AIチャットに、ユーザーは繰り返し戻ってくる | 7日以内の再訪率 **30%以上** |
| ★ 重要 | MBTI×星座×動物占い×六星占術の統合レポートに「自分を理解された」と感じる | レポート満足度アンケート **4.0/5.0以上** |
| ★ 重要 | SNSシェアカードが拡散され、新規ユーザーを獲得する | シェア率 **20%以上**、シェア経由登録 **5人以上** |
| 参考 | 共感チャット後のアクションプランを実行するユーザーがいる | アクションプラン「やってみた」報告率 10%以上 |

### 1.3 スコープ概要

| 項目 | 内容 |
|------|------|
| ターゲット | 25-35歳の若手社会人、対人関係に繰り返し消耗している層 |
| プラットフォーム | Webアプリ（スマホブラウザ対応、モバイルファースト） |
| 技術スタック | Next.js (App Router) / Supabase / Clerk / OpenAI Agent Builder / Gamma API |
| UI実装方針 | Claude Designでプロトタイプ→shadcn/ui + Tailwind CSSベースで実装 |
| **開発体制** | **APIファースト。まあみ=UI、えんまさ=言語/AI設計、ヒラメ=API/バックエンド** |
| 初期ユーザー | SNS小規模告知で50-100人想定 |
| 開発期間 | 4週間（4スプリント） |

## 2. ユーザー体験フロー

（v1.2から変更なし。UXシーケンス「共感→安心→分析→行動」の順序は非交渉で維持）

```
① 簡易登録（Clerk認証 + 生年月日入力）[30秒]
    ↓
② MBTI簡易診断（選択式8-12問 or スキップ）[3-5分]
    ↓
③ 統合レポート表示（MBTI × 星座 × 動物占い × 六星占術）
    └── ③-b SNSシェアカード（独立機能）
    ↓
④ 共感AIチャット（3フェーズ設計）← MVPの心臓部
    ↓
⑤ アクション記録（「できたことを記録する」形式）
```

### 2.1 アクション記録UIの設計方針（v1.3追加）

「できたことを記録する」フォーマットを採用。以下の原則を守る。

| 設計原則 | 実装指示 |
|---------|---------|
| **プレッシャーゼロ** | 達成率・残件数を前面に出さない |
| **否定形を見せない** | 「まだできていない」「未達成」は表示しない |
| **小さな行動を肯定する** | ✅ タップ一つで記録できる設計 |
| **継続的強化** | 「よくできました」より「また来たね」のトーン |

## 5. 開発アプローチ（v1.3新規追加）

### 5.1 APIファーストアプローチ

**開発順序：** API定義 → バックエンド実装（ヒラメ） → フロントエンド実装（まあみ）

従来の「機能別水平分業（同時並走）」から、「レイヤー別垂直分業（APIが先行）」へ変更。

**採用理由：**
- フロントエンド担当（まあみ）がUI未経験のため、APIの形が見えていれば迷走しにくい
- API先行により「モックJSONで動作確認→本番接続」の2段階が可能になる
- UI先行による手戻り（APIの仕様変更でUIを作り直し）を防ぐ

### 5.2 モックファースト並列開発フロー

```
ヒラメ（API）:   [Interface定義] ──────────────► [API実装完了]
                       │                                │
                       ↓ 先行共有                      ↓ 接続フェーズ
まあみ（UI）:    [プロトタイプ生成] → [モックで全画面確認] → [API接続]
                                          ↑
                               モックJSONを使って
                               APIなしで動かす
```

**ポイント：** ヒラメがAPI実装を完了する前に、まあみはモックデータで全画面の見た目と遷移を完成させる。APIが出来た瞬間に接続するだけで済む状態を目指す。

## 9. 言語設計方針（v1.3新規追加）

### 9.1 「占い」使用回避の方針

ターゲットユーザー（25〜35歳・25歳以下の若年層）の中に「占いへの抵抗感」を持つ層が存在する。
一方でMBTI・星座・性格診断は同じ層でも自然に受け入れられている。
COCOSiLは「占いツール」ではなく「自己理解・性格分析ツール」として認知されることを目指す。

### 9.2 言語置換テーブル（一覧）

| 使用禁止 | 代替表現 | 用途 |
|---------|---------|------|
| 占い | 性格分析 / 自己理解 / パーソナリティ診断 | 機能説明 |
| 占い師 | カウンセラー / セラピスト / AIパートナー | 役割説明 |
| 鑑定 | 診断 / 分析 / 読み解き | 動詞 |
| 運勢 | 傾向 / パターン / 特性 | 特徴描写 |
| 占星術 | 星座分析 / 星座ベースの性格傾向 | 機能説明 |
| 六星占術 | （使う場合は「六星占術」そのまま or 「生年月日ベースの性格診断」） | 機能説明 |
| 動物占い | 動物性格診断 / 60アニマル診断 | 機能説明 |
| 当たる | 自分らしい / 腑に落ちる / ぴったりくる | 体験表現 |

### 9.3 適用範囲

この言語設計は以下すべてに適用する：
- UIコピーライティング（タイトル・説明文・ボタンラベル）
- AIプロンプト（チャット・レポート生成・SNSシェアカード）
- エラーメッセージ・空状態テキスト
- SNSシェアカードのテキスト
- LP・告知文
