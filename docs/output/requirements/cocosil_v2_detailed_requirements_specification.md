# COCOSiL V2 詳細要件定義書

> **自分を知って、ラクになる。**
> 共感AIチャット × 多角的自己理解 — V1（v1.3 MVP）の設計思想レベル進化版

---

## 1. プロジェクト概要

> **設計中枢（Layer 0）** — `docs/input/concepts/COCOSiL設計中枢.md`
> 本要件定義書の機能追加・優先度判断・コピー・プロンプトはすべて設計中枢を上位基準として参照する。新機能は5問のリトマス試験紙（Must: Q1無明を晴らすか / Q2三毒を増幅しないか / Q3順序を守るか）を必ず通過していること。

COCOSiL V2は、現代人が対人関係（職場・恋愛・家族）で繰り返し消耗する根本原因——「**現代の無明**（自分も相手も、なぜそうなるかが分からない状態）」——を解消するWebサービスの進化版である。V1（v1.3 MVP）のデプロイ・クローズドテスト完了を踏まえ、ユーザーFBを元に設計思想レベルでアップデートした次バージョン。

### 設計の哲学的根拠（Why → How → So What）

| 層 | 根拠 | V2への適用 |
|---|---|---|
| **Why** | 無明（仏陀）/ 自知者明（老子） | 4体系統合は「なぜ自分はこうなのか」という構造的理解を生む唯一の設計 |
| **How** | 三毒・五蘊・パンチャ構造（4+1=5）・止観 | 「共感→安心→分析→行動」は止観の往還運動の現代的実装。F3.2「安心」フェーズは五蓋（心を覆う蓋）の除去に相当 |
| **So What** | ハーバード成人発達研究（80年） | 自己理解は終点でなく良い人間関係への入口。F4チャットの最終評価軸はQ5「良い人間関係に寄与するか」 |

V2では2つの主軸を貫通させる。**①プロダクト哲学の体系化**——各機能を「東洋哲学×西洋心理学×AI」の哲学に紐づけて根拠化する。**②UXシーケンスの絶対化**——「共感→安心→分析→行動」を非交渉の設計制約として全機能・全プロンプト・全UIに貫通させる。副軸として品質向上（B）・体験深化（C）・体制運用進化（D）を取り込む。

スコープ拡張（A）として、V1にはなかった**Stripe課金（4プラン構造）**・**データオーナーシップ機能**・**5フェーズ共感AIチャット**・**「安心」フェーズ**・**腑落ちマーカー**を追加。ターゲットは25〜35歳の若手社会人。開発期間60日、3名体制（意味/見た目/構造）、APIファースト×モック並列開発。

## 2. ビジネス要件

### 2.1 対象課題

| 課題 | 詳細 | 発現シーン |
|------|------|-----------|
| **本質課題: 現代の無明** | 自分も相手も「なぜこうなるか」が分からず、同じ対人パターンで繰り返し消耗する。東洋哲学が「苦の根源」と定義した「自己への無知」が現代に発現している状態 | 職場の人間関係 / 恋愛 / 家族 / 友人 |
| **既存代替手段の限界** | 共感系AI（感情処理のみ）と診断系（分析のみ）が分断されており、「感情処理→自己理解→行動変容」を一気通貫で提供するサービスが市場に存在しない | — |
| **対面占い師のコスト構造** | 深い共感と洞察を得られるが、月¥1万・非日常・29日間の空白がある。継続的な自己理解を支援できない | — |

### 2.2 提供価値（UVP）

> **「なぜ自分はこうなのか。なぜあの人とうまくいかないのか。その答えが、ここにある。」**

COCOSiLが届けるのは単なる共感でも表面的な自己分析でもなく、**「腑落ち体験」**——なぜ自分はこうなのかが分かって本当にラクになれる体験——である。

### 2.3 競合優位性（7Powers分析）

| Power | COCOSiLへの適用 |
|-------|----------------|
| **Counter-Positioning（カウンターポジショニング）** | 既存サービスは「感情処理」か「診断」の片方しかカバーしない。COCOSiLは「共感→安心→分析→行動」という一気通貫UXで、既存プレイヤーが模倣しにくい設計を持つ。MBTI単体サービスが共感機能を追加しようとすると自社の診断文脈が崩れ、逆も同様 |
| **Cornered Resource（独占資源）** | 4体系統合ナレッジ（MBTI×星座×動物占い60アニマル×六星占術）の構造化知識体系。特に日本発の動物占い60アニマルと六星占術をAI推論基盤に統合したシステムは業界最先端。えんまさが設計した3フェーズ（→5フェーズ）の「根拠ある共感」プロンプト体系 |
| **Switching Costs（スイッチングコスト）** | ユーザーの診断データ・チャット履歴・腑落ちマーカー・アクション記録がCOCOSiLに蓄積され続け、他サービスへの移行障壁を形成。「自分の自己理解の旅路」が記録された状態は他サービスでは再現できない |
| **Process Power（プロセス優位）** | 意味設計（えんまさ）×見た目設計（まあみ）×構造設計（ヒラメ）の垂直分業と、Protected Areas / AGENTS.mdによるガバナンス。「共感→安心→分析→行動」の順序を崩さない運用プロセスを組織的に担保 |

## 3. ユーザー要件

### 3.1 ターゲットユーザー

| セグメント | 属性 | 特徴 |
|-----------|------|------|
| **プライマリー** | 25〜35歳・若手社会人 | 対人関係に繰り返し消耗。年収300〜600万円。自己理解への潜在的関心 |
| **アーリーアダプター** | MBTIオタク・自己分析SNS投稿層 | 診断結果を共有する文化を持ち、シェアカード拡散の担い手 |
| **高LTVセグメント** | 35〜45歳・占いヘビーユーザー | 月¥1万支出層。占い師セッションの補完ニーズ |
| **将来セグメント** | 18〜24歳深夜感情層・法人HR | B2B展開の候補 |

### 3.2 ペルソナ

#### ペルソナA: 佐藤 麻衣（プライマリー）

- **属性**: 28歳・都内IT企業のCSマネージャー・年収420万円
- **行動パターン**: MBTIをSNSで定期投稿、ストレングスファインダーも受けたことがある。夜、仕事のモヤモヤをTwitterに書いてしまいがち
- **課題**: チームメンバーとの関係でいつも同じパターンで消耗する。「なぜ自分はこう感じるのか」がわからない。カウンセリングは敷居が高い
- **期待**: 「占い」とは言いたくないが、MBTIや星座を使って「自分のことが分かる」体験をしたい。AIに「そうそう、それなんだよ」と言いたい
- **COCOSiLへの初回行動**: SNSで友人のシェアカードを見て興味を持ち、「私もやってみる」で登録

#### ペルソナB: 田中 健太（アーリーアダプター）

- **属性**: 31歳・スタートアップ勤務のエンジニア・年収550万円
- **行動パターン**: 週1回自己分析ツールを使う。Notion等に気づきを記録している。「なぜ自分はこう行動するのか」に強い関心
- **課題**: 単発の診断では行動が変わらない。「分析で終わる」体験に飽き飽きしている。共感してくれながら分析してくれるサービスがない
- **期待**: AIとの対話で「腑落ち」したい。チャット履歴が残り、自分の変化が見えると嬉しい
- **COCOSiLへの初回行動**: 技術系SNSでCOCOSiLのアーキテクチャ記事を見て「面白そう」で登録

### 3.3 MVPの体験定義（ユーザー視点）

**V2 MVPを使うと、ユーザーは何ができるか：**

> COCOSiLにアカウント登録した日に、「ようこそ」の共感的対話→4体系統合診断→「安心」フェーズ付きの統合レポート→5フェーズ共感AIチャットという一気通貫の「腑落ち体験」を完了できる。翌日以降は振り返り機能・アクション記録・腑落ちマーカーで継続的な自己理解が蓄積される。課金ユーザーは月1回のレポート再生成・無制限チャットにアクセスできる。

## 4. 機能要件詳細

### 4.1 オンボーディング・登録（F1）

**概要**: ユーザーが30秒以内にアカウントを作成し、UXシーケンス「共感」フェーズの入口へ到達する。

**対応Stage 1要件**: F1.1〜F1.4

**業務フロー（正常系）**:
1. LP/シェアカードリンクからアクセス
2. 「自分を知って、ラクになる。」のLP表示（200文字以内の哲学提示: F1.4）
3. Clerkによるサインアップ（メール or SNS認証）— 30秒以内完了
4. 生年月日入力フォーム（4体系自動計算のため）
5. **「ようこそ」共感的初回対話開始**（F1.3）— AIが「今日はどんな気持ちで来ましたか？」と1〜2往復

**例外系**:
- Clerk認証失敗: エラーメッセージを言語設計文書準拠のトーンで表示（「もう一度試してみてください」）
- 生年月日が未来日付/異常値: インラインバリデーションで即時フィードバック
- SNS認証後の生年月日入力スキップ: 後から入力可能。スキップ時は4体系計算を「未設定」として保持

**特殊要件**:
- F1.3（共感対話）はActiveモード。ユーザーが語る設計
- F1.4の哲学提示は初回のみ表示。再訪時はスキップ

---

### 4.2 性格分析・自動診断（F2）

**概要**: 生年月日から星座/動物性格診断（60アニマル）/六星占術を自動算出し、ユーザーが選択式でMBTI簡易診断を完了する。4体系データが後続の全機能の基盤となる。

**対応Stage 1要件**: F2.1〜F2.4

**業務フロー（正常系）**:
1. 生年月日入力完了後、バックグラウンドで3体系（星座・動物・六星）を自動算出
2. MBTI簡易診断ページへ遷移（8〜12問選択式）
3. MBTI診断完了 or スキップ選択
4. 診断完了→統合レポート生成ページへ遷移

**例外系**:
- MBTIスキップ: 「MBTI未設定」状態のままレポート生成へ。後から診断可能
- 自動算出エラー（生年月日の境界値など）: 「未診断」状態として保持、手動補正導線を設置

**F2.4 哲学的根拠表示の設計**:
- 診断結果画面の各体系名横に「？」アイコン
- タップでアコーディオン展開（例：「なぜMBTIを使うのか」「星座分析が自己理解に使われる理由」）
- えんまさが200〜300文字の根拠テキストを設計

---

### 4.3 統合レポート（F3）⭐ V2の核心

**概要**: 4体系診断結果をAIで統合し、「安心」フェーズを経てリッチなレポートをユーザーに届ける。COCOSiLのUXシーケンス「安心→分析」の中核実装。

**対応Stage 1要件**: F3.1〜F3.5

**業務フロー（正常系）**:
1. MBTI診断完了後、バックグラウンドでLLM（OpenAI）によるレポートコンテンツ生成を開始
2. 生成中に**「安心」フェーズ表示**（F3.2）— 最大15秒
   - 呼吸アニメーション（4〜8秒）
   - テキスト: 「あなたの結果が揃いました。読む前に、ひとつだけ。」（60字以内）
   - AIサブテキスト: 承認・脱判定化・招待の3要素（100字以内、えんまさ設計）
   - CTA「読んでみる→」（Thumb Zone内）
3. CTAタップまたは15秒経過後にレポート本体に遷移
4. Gamma APIにコンテンツを渡してリッチレポートとして整形表示（LLM生成は並行実行）
5. 各セクションに「腑落ち」マーカーボタン（F3.3）— タップで記録
6. レポート末尾に満足度アンケート（5段階+自由記述: F3.4）
7. 再生成ボタン（前回生成から30日経過後に有効化: F3.5）

**例外系**:
- Gamma API障害時: Markdown+CSS静的レポートに自動フォールバック（並行開発済み）
- OpenAI API障害/タイムアウト（30秒超過）: 「少し時間がかかっています」のフレンドリーエラー表示、リトライ導線
- 課金未加入でのレポート再生成試行: 「有料プランで月1回再生成できます」の誘導

**F3.2「安心」フェーズ — 設計三原則**:
1. **Zero-Effort Reassurance**: ユーザーにコストを払わせない。受動的に体験
2. **Active→Receptive Handoff**: F1.3（共感:語る）からF3.2（安心:受け取る）へのモード切替
3. **De-judging through Metaphor**: 「結果=評価」を「結果=地図」に認知シフト

---

### 4.4 共感AIチャット（F4）⭐ MVPの心臓部

**概要**: 性格分析データに基づく「根拠のある共感」を提供する5フェーズ設計のAIチャット。UXシーケンス「共感→分析→行動」をチャット内で体現する。

> **2026-05-05 実装技術確定**：F4の実装基盤は **Vercel AI SDK (`ai`) + OpenAI Chat Completions API** に確定。`streamText()` + Edge Runtime でストリーミング応答を実現。フェーズ状態（`ChatPhase`）はSupabase `chat_sessions.phase` に永続化し、クライアント側に持たせない。詳細は §6.3 参照。

**対応Stage 1要件**: F4.1〜F4.5

**5フェーズ設計**:

| フェーズ | 名称 | AIの役割 | ユーザーの状態 |
|---------|------|---------|--------------|
| Phase 1 | **傾聴** | 話を聞く・反射する | 感情を開く |
| Phase 2 | **共感** | 感情を言語化して返す | 「わかってもらえた」 |
| Phase 3 | **深掘り** | 「なぜ？」を一緒に探る | 根本に気づく |
| Phase 4 | **洞察** | 性格分析と結びつける | 「腑落ちる」 |
| Phase 5 | **行動提案** | 具体的な一歩を提案 | 行動できる気がする |

**業務フロー（正常系）**:
1. レポート閲覧後または直接チャット画面へ遷移
2. AIが Phase 1（傾聴）で起動: 「今日、誰かとの関係でモヤっとしたこと、ありますか？」（例）
3. ユーザーが語る → AIが4体系診断データをコンテキストに注入して応答
4. 会話の流れからAIが自律的にフェーズ遷移を判定（Phase 1→2→3→4→5）
5. Phase 4（洞察）では腑落ちマーカーをチャット内でも表示可能
6. Phase 5（行動提案）の提案をワンタップでアクション記録（F5）に追加
7. セッション終了時にチャット履歴を自動保存

**業務フロー（再訪時）**:
1. 過去チャット履歴・アクション記録・腑落ちマーカーをAIがコンテキストに取り込む（F4.3）
2. 「また来てくれましたね。前回〇〇のことを話していましたが、その後いかがでしたか？」のトーンで起動
3. 継続性のある対話を提供

**例外系**:
- OpenAI APIエラー: 「少し時間をおいて再送してみてください」を言語設計文書準拠のトーンで表示
- チャット中断（タブ切替等）: セッションはサーバー側で保持（30分以内は再開可能）
- 無料プラン月5回上限到達: 「今月の共感チャットをすべて使い切りました。有料プランで続けられます」

**特殊要件**:
- 5フェーズのプロンプト設計はえんまさが担当（意味設計レイヤー）
- Phase移行の判定ロジックはプロンプト内のシステム指示で制御
- F4.2「根拠のある共感」: 診断データ（4体系 + 過去会話要約）をシステムプロンプトに常時注入

---

### 4.5 アクション記録（F5）

**概要**: 共感AIチャットの行動提案フェーズから生まれた行動を「できたこと」形式で記録する。UXシーケンス「行動」フェーズの実装。

**対応Stage 1要件**: F5.1〜F5.3

**設計原則（プレッシャーゼロ）**:
- 達成率・残件数・進捗バーを表示しない
- 「未達成」「できていない」等の否定形は使用しない
- 「また来たね」のトーンで継続を歓迎する

**業務フロー（正常系）**:
1. F4チャットのPhase 5（行動提案）でAIが提案 → 「記録する」ボタン1タップで自動追加
2. または手動でアクション記録画面から追加（自由入力）
3. 記録一覧表示（タイムライン形式、日付で整理）
4. 「できた！」ボタンを1タップで完了マーク（完了後はアーカイブへ）
5. F5.3 振り返り機能: 過去のアクション記録と気づきを「〇〇日前はこんなことを試みていましたね」のトーンで表示

**例外系**:
- アクション記録0件時: 「まだ記録がありません。チャットで話してみることから始めましょう」の誘導（否定形を使わない）

---

### 4.6 SNSシェアカード（F6）

**対応Stage 1要件**: F6.1〜F6.4

**業務フロー**:
1. 統合レポート完了画面に「シェアする」ボタン（Thumb Zone）
2. えんまさ設計プロンプトで4体系統合の独自タイプ名を生成（例: 「夜に強い内省型 — 壬戌の哲学者」）
3. シェアカード画像生成（COCOSiLブランドカラー: `#5B21B6`ベース）
4. X / Instagram Stories / LINE へのワンタップシェア
5. OGP対応のシェアURL生成（`cocosil.ai/share/{uuid}`）

**例外系**:
- シェア先アプリ未インストール時: URLコピーのフォールバック
- タイプ名生成エラー: デフォルトタイプ名（「4体系統合タイプ」等）でフォールバック

### 4.7 課金・サブスクリプション（F7）— Stripe

**概要**: Stripeを決済基盤として4プラン構造を実装。透明性を重視した自己管理型サブスクリプション。

**対応Stage 1要件**: F7.1〜F7.4

**業務フロー（サブスクリプション開始）**:
1. 無料プランユーザーが制限（月5回チャット等）に到達
2. プランアップグレードを促すモーダル表示（誘導はCOCOSiLのトーンで）
3. Stripe Payment Element（日本語UI）で決済情報入力
4. 決済完了 → Stripe Webhookでプランフラグを即時更新（Supabase `profiles.plan`）
5. 機能制限を即時解除

**業務フロー（プラン変更）**:
1. Stripe Billing Portal（ユーザー自己管理）へリダイレクト
2. プラン変更 → プロレーション（日割り精算）を自動計算
3. Webhook受信 → Supabaseを即時更新

**業務フロー（解約）**:
1. Stripe Billing Portalで自己解約（COCOSiLサイドの操作不要）
2. 解約確認後、現在の請求期間終了まで機能維持
3. 期間終了時にWebhookで無料プランへ自動ダウングレード

**認証-課金-DB 3点同期設計**:
```
Clerk User（認証）
    ↕ stripe_customer_id を profiles に保存
Stripe Customer（課金状態）
    ↕ Webhook (idempotency_key付き)
Supabase profiles.plan（DB）
```

**例外系**:
- 決済失敗（カード残高不足等）: Stripeが自動でリトライ（3日間）。ユーザーへメール通知
- Webhook処理失敗: idempotency_keyで重複防止。StripeダッシュボードでWebhook履歴を確認
- 法人格設立前の運用: test_modeで全機能実装完了後、法人設立時にAPIキー入れ替えのみでlive_modeへ

## 6. インテグレーション要件

### 6.1 Clerk（認証）

- **用途**: サインアップ・サインイン・セッション管理・ロールベースアクセス制御（管理者画面）
- **連携方式**: Next.js App Router の `@clerk/nextjs` SDK、middleware.tsでルート保護
- **データ連携**: Clerk `user.id` を Supabase `profiles.clerk_user_id` に保存、Stripe Customer metadataにも記録
- **コスト**: 無料枠（MAU 10,000まで）。超過時 $0.02/MAU

### 6.2 Supabase（データ永続化）

- **用途**: ユーザープロファイル・診断データ・チャット履歴・アクション記録・分析イベントの永続化
- **連携方式**: Next.js Server Components から `@supabase/ssr` SDK でサーバーサイドクエリ
- **認可**: 全テーブルにRLS適用。ユーザーは自分のデータのみ読み書き可能
- **型定義**: `pnpm db:types` でリモートプロジェクトから自動生成（手書き禁止）
- **制約**: 無料枠 0.5GB。チャット履歴の定期アーカイブ戦略を実装

主要テーブル構成:

```
profiles          — Clerk user ID、生年月日、診断データ、Stripe顧客ID、プランフラグ
diagnoses         — 4体系診断結果（MBTI / 星座 / 動物 / 六星）
chat_sessions     — チャットセッション（フェーズ状態を含む）
chat_messages     — チャットメッセージ（role: user/assistant）
action_plans      — アクション記録
share_cards       — シェアカード生成履歴
analytics_events  — PMF KPI計測用イベントログ
```

### 6.3 Vercel AI SDK + OpenAI（共感AIチャット・レポート生成）

> **2026-05-05 技術選定確定**：当初「OpenAI Agent Builder」として仮置きしていたが、議論ログ（`docs/discussions/議論ログ_AIチャット技術スタック選定.md`）による検討の結果、**Vercel AI SDK + OpenAI Chat Completions API** に確定。Dify・OpenAI Assistants APIはいずれも不採用（理由は議論ログ参照）。

- **用途**: 5フェーズ共感AIチャット、統合レポートコンテンツ生成、シェアカードタイプ名生成
- **連携方式**: `ai` + `@ai-sdk/openai` パッケージ。`streamText()` + Edge Runtimeでストリーミング応答。APIキーはサーバー専用（`getServerEnv()` 経由）
- **フェーズ管理**: セッション状態（`ChatPhase`）はSupabase `chat_sessions.phase` に永続化。クライアント側にフェーズ状態を持たせない
- **マルチLLM切り替え**: Vercel AI SDKの統一コネクター（`@ai-sdk/openai` / `@ai-sdk/anthropic` / `@ai-sdk/google`）で1行切り替え可能
- **プロンプト設計**: えんまさが意味設計レイヤーで担当。`lib/prompts/` 配下にTypeScriptで配置（Protected Areas Layer 2）。GUIツールによる管理は禁止
- **コスト**: (仮定) 月額 $50〜200（MAU規模・1ユーザーあたりチャット頻度による）。Vercel AI SDK自体は無料
- **制約**: 課金枠上限に達した場合の切り替え基準を事前決定

### 6.4 Gamma API（統合レポート整形）

- **用途**: LLMが生成したレポートコンテンツをリッチな視覚表現に変換
- **連携方式**: Next.js API Route からREST呼び出し。バックグラウンド処理（MBTI診断完了後に非同期起動）
- **フォールバック**: Gamma API障害時はMarkdown+CSS静的レポートに自動切替（並行開発済み）
- **リスク**: レート制限・コスト構造は Phase 1内に早期PoC必須。Sprint 3前に確認
- **コスト**: (要確認) 未確認。PoC後に月額コスト推定を更新

### 6.5 Stripe（課金）

- **用途**: 4プラン構造のサブスクリプション管理、決済処理、ユーザー自己管理（Billing Portal）
- **連携方式**: `stripe` Node SDK（サーバーサイド）+ Stripe.js（クライアントサイド Payment Element）
- **Webhook**: `app/api/stripe/webhook/route.ts`。署名検証必須、idempotency_key付きで冪等処理
- **法人格対応**: test_modeで全機能実装完了後、法人設立時にAPIキー（`STRIPE_SECRET_KEY`）入れ替えのみでlive_modeへ
- **コスト**: 手数料3.6%（日本カード）+ 決済手数料¥30/件。月¥12万売上時 手数料約¥4,600

### 6.6 Vercel（インフラ）

- **用途**: Next.js 16アプリケーションのホスティング、エッジ配信、Previewデプロイ
- **連携方式**: GitHubリポジトリと連携、mainブランチへのマージで本番自動デプロイ

### 7.1 技術スタック

| レイヤー | 技術 | 選定理由 |
|---------|------|----------|
| 基盤 / インフラ | **Vercel** | Next.js 16との最高親和性、エッジ配信、Previewデプロイ |

| UIフレームワーク | **Tailwind CSS 4 + shadcn/ui** | ブランドカラー（#5B21B6）に合わせたCSSカスタマイズ容易、まあみのプロトタイプ→実装の橋渡し |
| バックエンド | **Next.js 16 API Routes** | フロントと同一プロジェクト、ヒラメ単独でAPIファーストに集中可能 |
| データ永続化 | **Supabase（PostgreSQL + RLS）** | 認可をDB層で担保、型自動生成、無料枠でMVP規模を満たす |
| 認証 | **Clerk** | サインアップ30秒以内、SNS認証統合、Next.js公式統合、Stripe公式連携 |
| AI / チャット | **Vercel AI SDK (`ai`) + OpenAI Chat Completions API** | `streamText()` + Edge Runtimeでストリーミング。フェーズ遷移はアプリ側で管理。マルチLLM切り替えは統一コネクター（`@ai-sdk/openai` / `@ai-sdk/anthropic`等）で対応 |
| AI / レポート | **Gamma API** | 構造化レポートを自動生成。フォールバックはMarkdown+CSS |
| 決済 | **Stripe** | Clerk公式パートナー、プロレーション自動、Billing Portal自己管理、test_mode法人化対応 |
| 言語 | **TypeScript 5 + Zod（`zod/v4`）** | API型安全性、AGENTS.md準拠 |
| パッケージ管理 | **pnpm** | AGENTS.md確定。npm/yarn禁止 |
| ビルド / CI | **GitHub Actions + Vercel** | PR毎Previewデプロイ、`pnpm typecheck` / `pnpm lint` / `pnpm build`をCI実行 |
| 監視 | **Sentry（予定）** | エラー収集とパフォーマンスモニタリング |
| ガバナンス | **`.claude/hooks/prevent-destructive-command.js`** | Protected Areas Layer 1のhook自動ブロック |

### 7.2 アーキテクチャ方針

**データフロー**:
```
クライアント（Next.js RSC / Client Component）
    ↕ HTTPS / Server Actions / API Routes
Next.js API Routes（ヒラメ担当）
    ├── 認証チェック（Clerk middleware）
    ├── 診断計算（`lib/diagnostics/`）
    ├── チャット処理（→ OpenAI API）
    ├── レポート生成（→ OpenAI → Gamma API）
    └── 課金処理（→ Stripe SDK）
    ↕
Supabase（PostgreSQL + RLS）
    └── ユーザーデータはRLSで完全所有者分離
```

**認証・認可の方針**:
- Clerk: セッション管理・JWT発行
- Supabase RLS: DB層でユーザーデータの所有権を強制。APIレイヤーではなくDB層で認可
- Stripe: Webhookの署名検証で不正リクエストを排除

**レイヤー分業の境界**:
- `lib/prompts/**`: えんまさ（意味設計）専任。AIエージェントによる変更はえんまさ承認必須
- `components/**`, `app/(auth|public)/**`: まあみ（見た目設計）担当
- `app/api/**`, `lib/diagnostics/**`: ヒラメ（構造設計）担当

**`src/`ディレクトリは存在しない**: `@/*` は `./`（プロジェクトルート）にマップ。`app/`・`lib/`・`public/`に直接配置

| 5フェーズチャットのPhase遷移が不自然 | 「AIが機械的」という体験 | えんまさのプロンプト設計でフェーズ遷移を自然な会話の流れとして埋め込む。Phase 1完了後に3名でQAレビュー |
