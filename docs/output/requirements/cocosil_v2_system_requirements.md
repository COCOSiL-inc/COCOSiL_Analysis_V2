# COCOSiL V2 システム要件定義書

> **自分を知って、ラクになる。**
> 共感AIチャット × 多角的自己理解 — V1（v1.3 MVP）の設計思想レベル進化版

---

## システム概要

COCOSiL V2は、現代人が対人関係で繰り返し消耗する「**現代の無明**（自分も相手も、なぜそうなるかが分からない状態）」を解消するための、**共感AI × 多角的自己理解**Webサービスの進化版である。

V1（v1.3 MVP）の10機能をベースに、**設計思想レベルの大幅アップデート**を施す。具体的には、(1) **プロダクト哲学の体系化** — 各機能を「東洋哲学×西洋心理学×AI」の哲学に紐づけて根拠化する、(2) **UXシーケンスの絶対化** — 「共感 → 安心 → 分析 → 行動」を非交渉の設計制約として全機能・全プロンプト・全UIに貫通させる、の2軸を主軸とする。副軸として品質向上・体験深化・体制運用進化を組み込む。

ターゲットは**25〜35歳の若手社会人**。技術スタックはNext.js 16 / Supabase / Clerk / OpenAI / Gamma APIを採用し、APIファースト×モック並列開発・3名分業（意味/見た目/構造）・Protected Areasによる安全な開発を採る。**PMF基準は7日以内再訪率30%以上**、3年目MAU 10,000人を目指す。

### 置いた仮定

- **(仮定) V2の対象期間**: V1（v1.3 MVP）の4スプリント完了後、もしくは並行的に進化を反映する次バージョン。具体的なリリース時期は未確定（要確認）。
- **(仮定) スコープ拡張の中身**: ユーザー回答「Aが主軸」「設計思想レベルでアップデート」「①+②が中心」から、UXシーケンス各フェーズの**機能化・明示化**（特に「安心」フェーズの新設）を中心としたスコープ拡張と解釈。
- **(仮定) 課金実装の組み込み**: コンセプトv2.0の4プラン構造（無料/¥500/¥1,500/¥3,000-5,000）をV2要件に含める前提とした（要確認）。V1（v1.3）には含まれていない。
- **(仮定) データオーナーシップ機能**: 「自己理解データはユーザーのもの」という哲学的立場から、エクスポート・削除機能を含める前提とした（要確認）。
- **(確定) 開発体制**: えんまさ（意味/言語/AI）、まあみ（UI）、ヒラメ（API/構造）の3名分業はV2でも継承（v1.3で確定済み・AGENTS.mdで再確認）。
- **(確定) 言語設計ルール**: 「占い」禁止用語と代替表現テーブルは全機能・全プロンプトに適用（言語設計文書 v1.0）。

## 機能要件

> **凡例**: 各要件の「UXシーケンス」列は、共感 / 安心 / 分析 / 行動 / 横断 のいずれにマッピングされるかを示す（V2の設計思想②に基づく）。
> **V2マーク**: `[V2新設]` = V1にない新機能 / `[V2拡張]` = V1の同名機能を拡張 / `[V1継承]` = V1から変更なし

## 非機能要件

| カテゴリ | 要件 | 目標値 | 根拠 |

## 開発ロードマップ

### Phase 1: V1 MVP の完了（v1.3スコープ完成）

**目的**: V1（v1.3 MVP要件）の10機能を完成させ、PMF仮説検証の最低限の素材を作る。V2の進化はこの上に積む。

- F1.1〜F1.2 認証・生年月日入力
- F2.1〜F2.3 自動診断 + MBTI簡易診断
- F3.1, F3.4 統合レポート + 満足度アンケート
- F4.1〜F4.2, F4.4 共感AIチャット 3フェーズ + 履歴保存
- F5.1〜F5.2 アクション記録（できたこと形式）
- F6.1〜F6.3 SNSシェアカード
- F9.1, F9.2, F9.5 アナリティクス + 管理者画面 + エラー通知
- F10.1, F10.2, F10.4 言語設計準拠（手動チェック）

### Phase 2: V2 設計思想の機能化（主軸①+②の実装）

**目的**: V2の主軸であるプロダクト哲学の体系化（①）とUXシーケンス絶対化（②）を、機能として実装する。最も「V2らしい」差分。

- **F1.3** 「ようこそ」共感的初回対話
- **F1.4** プロダクト哲学の簡潔な提示
- **F2.4** 4体系の哲学的根拠表示
- **F3.2** 「安心」フェーズの先出し導入文 ⭐ V2の核心
- **F3.3** 「腑落ち」マーカー機能
- **F4.3** 過去対話・アクション記録の参照
- **F5.3** 振り返り機能
- **F9.3** PMF KPI追跡ダッシュボード

### Phase 3: スコープ拡張（A型・収益化と権利保護）

**目的**: V2のA型スコープ拡張部分を実装。MVP検証後の収益化と、ユーザーデータ主権原則の制度化。

- **F7.1〜F7.4** 課金・サブスクリプション 4プラン構造（決済プロバイダ選定後）
- **F8.1〜F8.2** データエクスポート + アカウント削除
- **F8.3** プライバシーダッシュボード（任意）

### Phase 4: 運用・品質・体制の進化（副軸B+D）

**目的**: 副軸の品質向上（B）と体制・運用進化（D）を反映。長期運用に耐える基盤を整える。

- **F9.4** Protected Areas監視
- **F10.3** 禁止語の自動検査（pre-commit hook / CI）
- 型安全性の総点検（API型定義の網羅、`pnpm typecheck`の常時グリーン化）
- E2Eテスト整備（playwright活用、現状未整備）
- Supabaseアーカイブ戦略の自動化
- ブランチ運用ルール定着（AGENTS.md §8）

## リスク分析

### 技術的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| Gamma API の品質・速度・コスト | 統合レポート（F3）の体験品質を直撃。Phase 1のクリティカルパス | **Sprint 3（Phase 1内）早期にPoC実施。Markdown+CSS静的レポートを並行開発（フォールバック）** |
| OpenAI API の課金枠超過 | 共感AIチャット（F4）が止まる | **課金プラン切替基準を事前決定、F7課金導入で原資確保** |
| Supabase 0.5GB 無料枠超過 | チャット履歴・アクション記録の蓄積で抵触可能性 | **初日からアーカイブ戦略実装、F8データエクスポート完備、運用開始時点で容量監視ダッシュボード** |
| Next.js 16 の破壊的変更 | App Router仕様変更でAPI設計が手戻り | **`node_modules/next/dist/docs/` を必ず読む（AGENTS.md §0）、非推奨警告に従う** |

### ビジネス的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| PMF基準（7日以内再訪率30%）未達 | プロダクト存続の根本リスク | **F9.3 KPIダッシュボードで常時可視化、Phase 2のV2機能で再訪率テコ入れ** |
| 課金移行率が低い | 収益化シナリオ崩壊 | **(仮定) 無料プランの月5回チャット制限で適度な摩擦を作る、F7.2で段階的アンロック** |
| 競合（チャッピー等）の機能拡充 | 「根拠のある共感」の差別化が消える | **4体系統合（特に動物占い60・六星占術）の独自性を強化、F2.4で根拠を可視化** |
| BOOST第8期コンテストの結果 | 資金調達・露出への影響 | (仮定) コンテスト依存度を下げる収益化路線をPhase 3で前倒し |

### 運用的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| Protected Areas 違反による事故（DB破壊・RLS無効化等） | データ損失・セキュリティ事故 | **hook自動ブロック（Layer 1）+ F9.4監視。AI委任範囲はLayer 3に限定（AGENTS.md §7）** |
| 「占い」用語のUI/プロンプト混入 | ターゲットの抵抗感を生み信頼毀損 | **F10.3 禁止語自動検査、PRレビュー時の必須チェック項目化** |
| まあみUI実装のブロック（v1.3継続リスク） | Phase 1スケジュール遅延 | **APIファースト + モックJSON並列開発（v1.3 §5.2継承）、ヒラメによる骨格先渡し** |
| 開発体制ルールの形骸化 | レイヤー越境の頻発、品質劣化 | **副軸D。F10.3自動化、コードレビュー時のLayer境界チェック、AGENTS.md定期参照** |

### UX / プロダクトリスク

| リスク | 影響 | 対策 |
|--------|------|------|
| UXシーケンス順序破綻（共感→安心→分析→行動が崩れる） | 「腑落ち」体験が消え、PMF未達直結 | **設計思想②の絶対化。F3.2「安心」フェーズ必須実装、機能要件のUXシーケンス列で全機能をマッピング** |
| 「分析を先に出して共感を飛ばす」誘惑 | UXシーケンス破綻の典型例 | **AGENTS.md §0非交渉ルール、えんまさ承認なしの順序変更禁止** |
| アクション記録のプレッシャー化（達成率表示等） | プロダクトの優しさが失われリテンション低下 | **F5.1否定形非表示原則、振り返り機能（F5.3）の「また来たね」トーン徹底** |
| 「腑落ち」マーカー（F3.3）が押されない | V2新機能の価値検証不能 | **マーカー利用率をF9.3 KPIに含め、利用率<10%なら3スプリント以内に再設計** |

### 法的 / コンプライアンスリスク

| リスク | 影響 | 対策 |
|--------|------|------|
| 動物占い60アニマルの著作権 | 公開停止・損害賠償リスク | **公開情報を参照して自主構成中（v1.3継承）、株式会社COCOSIL設立後に法的確認** |
| 個人情報保護法 / GDPR対応 | 海外展開時の法令違反リスク | **F8.1〜F8.2のデータエクスポート・削除を初期実装、プライバシーポリシー整備** |
| 決済プロバイダの利用規約・PCI DSS | 違反時の決済停止 | **F7.3で選定したプロバイダ（Stripe等）のSDK経由でのみ実装、カード番号自前保持禁止** |

## 追加情報リクエスト

> より精度の高い要件定義書（Stage 2）のために、以下の情報を追加で確認したい。優先度の高いものから対応すること。

### 🔴 高優先度（要件の根幹に関わる）

1. **V2の対象期間とリリース時期**: V1（v1.3 MVP）と並行して開発するか、MVP完了後に着手するか。Phase 1〜4のスケジュールが大きく変わる。
2. **F7課金実装をV2スコープに含めるか**: 含める場合、決済プロバイダの選定（Stripe/Paddle/Pay.jp等）。MVP検証後に分離する選択肢もある。
3. **F8データオーナーシップ機能の優先度**: 設計思想からは「初期実装すべき」と判断したが、ビジネス優先度的に Phase 3 で問題ないか。
4. **F3.2「安心」フェーズの具体的な体験設計**: えんまさが意味設計レイヤーで担当するが、要件として何分の体験／どんな文言／視覚要素を含むか。

### 🟡 中優先度（仕様の精度に関わる）

1. **F4.1共感AIチャットの3フェーズを5フェーズ等に拡張するか**: 副軸C（体験深化）の候補だが今回採用は見送ったか。
2. **F3.5レポート再生成機能の頻度・無料枠**: Gamma APIコストとの兼ね合い。月1回 or プラン依存。
3. **F9.3 PMF KPI ダッシュボードの実装範囲**: 自前実装か外部ツール（Mixpanel等）連携か。
4. **副軸⑤（4体系のナレッジグラフ化）の採否**: 今回は副軸候補から外したが、V2.5以降の検討対象とするか。

### 🟢 低優先度（あれば書ける）

1. **B2B展開（HR・法人）の準備時期**: コンセプトv2.0で「将来セグメント」とされている。
2. **言語ガバナンス自動検査（F10.3）の実装方式**: pre-commit hook / CI / Lint pluginのいずれか。
3. **管理者ダッシュボード（F9.2）のアクセス制御**: Clerkのrole管理 or 別実装。
