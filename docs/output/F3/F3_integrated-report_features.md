# F3 統合レポート フィーチャー要件（5サブフィーチャー）

> **本書の位置づけ**: F3「統合レポート」の5サブフィーチャー（F3.1〜F3.5）を Feature 層で精緻化したもの。requirements-grill（Feature層 F-1〜F-8）により1問ずつ確定した。受け入れ基準の EARS 形式化は後続の Eval 層で行う。

---

## 0. 概要

### 0.1 上位 Capability

F3 の5サブフィーチャーは、いずれも単一の Capability の上に立つ。

> **CAP「4体系を統合した立体的な自己像（識）をユーザーに届ける」**

設計中枢 §2.3「パンチャ構造（4+1=5）」の論理的必然に対応する。

```
星座 ── 動物 ── 六星 ── MBTI     ← 4つの部分（異なる角度の切り取り）
              ↓ AIが統合（+1）
        1つの立体的な自己像 ＝ 識（しき）   ← "あなたという人の全体像"
```

単一フレームは「ラベル」しか生まない。4視点の統合があって初めて「なぜ自分はこうなのか」の構造的理解が生まれる（設計中枢 原則①「Dispel, Don't Decorate.」）。

### 0.2 5サブフィーチャーの役割分担

| Feature | 名称 | Capability上の役割 | Phase |
|---|---|---|---|
| **F3.1** | レポート生成エンジン | 「+1の統合」そのもの — 4体系→1つの自己像を生成 | Phase 1 先頭 |
| **F3.2** | 「安心」フェーズ | 統合像を受け取る準備（Receptive状態）をつくる | Phase 1 |
| **F3.3** | 「しっくりきた」マーカー | 統合像が当てはまったかをユーザー自身が検証・記録 | Phase 1（検証はPhase 2） |
| **F3.4** | 満足度アンケート | 統合像の品質を運営が計測（PMFフィードバックループ） | Phase 1 |
| **F3.5** | レポート再生成 | 統合像を継続利用の蓄積で更新（生きた地図に育てる） | Phase 3 |

### 0.3 代表ペルソナ

- **ペルソナA 佐藤 麻衣（プライマリー）** — 28歳・都内IT企業のCSマネージャー。「占いとは言いたくないが、MBTIや星座で"自分のことが分かる"体験をしたい」層。F3.1〜F3.4 の中心ペルソナ。
- **ペルソナB 田中 健太（アーリーアダプター）** — 31歳・スタートアップのエンジニア。「単発の診断では行動が変わらない」「自分の変化が見えると嬉しい」層。F3.5 の中心ペルソナ。

### 0.4 UXシーケンス上の位置

F3 は非交渉のUXシーケンス「共感 → **安心 → 分析** → 行動」のうち「安心→分析」の中核実装。設計中枢 §2.5 の止観構造では、F3.2＝「安心（五蓋の除去）」、F3.1/F3.3＝「分析（パンチャ構造で自己を解剖）」に対応する。

## 1. F3.1 レポート生成エンジン

4体系診断結果をAIで統合し、Vercel OG (Satori) で縦長（1024×1792）の構造化レポートを即時生成・表示する。

| 項目 | 内容 |
|---|---|
| **Persona（F-1）** | ペルソナA 麻衣。最初に使うのは「F1オンボーディング→F2の4体系診断（生年月日3体系＋MBTI）を完了した直後の新規ユーザー」 |
| **トリガー（F-2）** | **生成トリガー** = MBTI診断完了（またはスキップ確定）→ バックグラウンドでLLM生成開始。**表示トリガー** = F3.2「安心」フェーズのCTA「読んでみる→」タップ／15秒経過の自動遷移。初回が主トリガー |
| **完了状態（F-3）** | 体験面 = 4体系を1つの統合像（"私の地図"）として受け取り、評価ではなく地図と感じている。機械面 = 縦長レポート(1024×1792)が表示完了し、全セクション閲覧可、末尾のF3.4アンケート・次アクション導線に到達できる |
| **上位Capability（F-4）** | CAP「統合像を届ける」の中核 = 「+1の統合」そのもの |
| **実装フェーズ（F-8）** | Phase 1 先頭（F3.2/F3.3が乗る土台）。gpt-image-2による背景・アイコン追加は imager2アーキ フェーズ3 に分離 |

**ユーザーストーリー（F-5）**

> 4体系診断を終えたばかりの麻衣として、星座・動物・六星・MBTIをバラバラのラベルではなく "これが私" という1つの立体的な自己像として受け取りたい。それは「なぜ自分はこうなのか」が構造として腑落ちし、対人関係の消耗から少しラクになるためだ。

**受け入れ基準の核（F-6）**

| # | 受け入れ基準 | 根拠 |
|---|---|---|
| AC-1 統合性 | レポートに星座・動物・六星・MBTIの4体系すべてが反映され、かつそれらを横断する統合考察（1つの自己像の記述）が1セクション以上含まれること。4ラベルの羅列だけのレポートは不合格 | 設計中枢§2.3／原則① |
| AC-2 関係性 | レポートに「大切な人と良い関係を築くヒント」セクションが必ず1つ含まれること | 設計中枢 原則③ |
| AC-3 レンダリング品質 | 縦長（1024×1792）の構造化レイアウトで文字崩れゼロで表示でき、Vercel OG即時生成が成功すること。生成失敗時はMarkdown+CSS静的レポートにフォールバックすること | 詳細要件§4.3例外系／非機能要件 |

**Out of Scope（F-7）**

1. gpt-image-2による背景・アイコン画像生成（imager2アーキ フェーズ3。文字レイヤーは全フェーズ Vercel OG が担保）
2. レポートのユーザー編集・カスタマイズ（セクション並べ替え・文言修正）
3. レポートのPDF/画像エクスポート・印刷（F8 データオーナーシップの責務）
4. 複数レポートの履歴比較・差分表示（V2スコープ外）
5. F4チャットへのレポート内容のコンテキスト注入（F4.2の責務。F3.1は生成・表示まで）

## 2. F3.2「安心」フェーズ

F3.1レポート生成完了後・表示直前に必ず挿入される、判定不安を和らげる15秒以内の儀式。設計三原則は議論ログ（`議論ログ_安心フェーズ体験設計.md`）で確定済み。

| 項目 | 内容 |
|---|---|
| **Persona（F-1）** | ペルソナA 麻衣 — 特に「MBTI好きで"判定不安"が強い層」。診断結果遷移直前3〜5秒でタップ速度が平均37%低下する「無防備ゾーン」が最も強く出る |
| **トリガー（F-2）** | F3.1レポート生成完了→結果画面遷移直前の「無防備ゾーン」。**スキップ不可で必ず挿入**。初回が主、再訪時はアニメ短縮＋テキスト変更 |
| **完了状態（F-3）** | 体験面 = 「結果＝評価」から「結果＝地図」へ認知転換し、判定不安が和らいだ Receptive（受け取る体勢）。レポートを「一緒に読むもの」として身構えずに開ける。機械面 = 15秒以内（レポート並行生成時は最大30秒）に完了しレポート本体へ到達できる |
| **上位Capability（F-4）** | CAP「統合像を届ける」。F3.2は「統合像を受け取る準備＝Receptive状態をつくる」接続点。設計中枢§2.5「安心＝五蓋の除去」に対応 |
| **実装フェーズ（F-8）** | Phase 1（⭐V2の核心）。F3.1直後。再訪時のアニメ短縮も同フェーズ。AIサブテキストのプロンプト設計はえんまさの意味設計（Gate 2対象） |

**ユーザーストーリー（F-5）**

> 4体系診断を終えてレポート表示を待つ麻衣として、結果を "評価" ではなく "自分を知る地図" として受け取れる心理状態になりたい。それは判定される恐怖に身構えず、レポートを「一緒に読むもの」として開けるためだ。

**受け入れ基準の核（F-6）**

| # | 受け入れ基準 | 根拠 |
|---|---|---|
| AC-1 挿入の絶対性 | F3.1レポート生成完了後・表示前に必ず挿入され、スキップできないこと。15秒以内（レポート並行生成時は最大30秒）で自動完了またはCTAタップで完了 | 議論ログ F3.2.1/F3.2.2・原則② |
| AC-2 Receptiveモードの純度 | 双方向対話なし・AIがユーザーに質問しない・インタラクションはCTAボタン1つのみ（テキスト入力/選択肢/スワイプなし）。F1.3共感フェーズと心理モードが明確に分離されていること | 議論ログ F3.2.7/F3.2.8 |
| AC-3 脱判定化の達成 | AIサブテキストが承認・脱判定化・招待の3要素を100字以内で含み、「地図」メタファーで「結果＝評価」→「結果＝地図」の認知転換を行っていること。禁止語（占い・鑑定・当たる）0% | 議論ログ F3.2.5/F3.2.9・設計中枢 |

**Out of Scope（F-7）**

1. 双方向対話・ユーザーへの質問・問いかけ選択（F1.3共感フェーズの責務）
2. レポート内容そのものの生成・整形・表示（F3.1の責務）
3. 安心フェーズのスキップ設定・オプトアウト（「スキップ不可」が原則②由来の非交渉要件）
4. 「準備できたら開封」型の儀式演出（議論で不採用 — 2回目以降に摩擦化する）
5. gpt-image-2による待ち時間演出ビジュアル（imager2アーキ フェーズ3。F3.2は呼吸アニメで完結）

## 3. F3.3「しっくりきた」マーカー

レポート閲覧中、当てはまった箇所を「これ自分だ」とワンタップで記録し、後続のF4チャット・F5振り返りに継承する。

> **命名について**: 旧称「腑落ちマーカー」。「腑落ち」は内部用語のため、ユーザー向け正式名称を **「しっくりきた」マーカー** に統一した（2026-05-21・全ドキュメント統一リネーム済み）。要件ID F3.3 は不変。

| 項目 | 内容 |
|---|---|
| **Persona（F-1）** | 麻衣（主）= 「AIに『そうそう、それなんだよ』と言いたい」衝動をワンタップ化。田中健太（副）= 記録が後続体験に効いてくる「記録装置」として価値を見出す層 |
| **トリガー（F-2）** | レポート閲覧中、特定の一文・セクションに「これ自分だ／しっくりきた」と感じた瞬間（主）。F4チャット Phase 4（洞察）でも表示（副）。読みながら反射的にタップ＝超低摩擦が前提 |
| **完了状態（F-3）** | 体験面 = 当てはまった箇所を言葉にする手間なくマークでき「自分が反応した記録」が手元にある。機械面 = 項目（セクションID・該当テキスト等）がDB記録され、F4.3（過去参照）・F5.3（振り返り）から参照可能 |
| **上位Capability（F-4）** | CAP「統合像を届ける」。F3.3は止観の「観＝『これは自分のパターンだ』と命名する」の最小ユーザーアクション。統合像を能動的に受け取り直す検証ループ |
| **実装フェーズ（F-8）** | Phase 1 実装（F3.1完成後）。利用率10%以上の検証は継続性が見える Phase 2。10%未達なら3スプリント以内に再設計。UIテキスト（チェックラベル等）はえんまさ（Gate 2） |

**ユーザーストーリー（F-5）**

> レポートを読む麻衣として、「これ自分だ」と感じた箇所をその場でワンタップで残したい。それは、自分が本当に当てはまると思った部分が記録され、後のチャット（F4）や振り返り（F5）で "自分が選んだ手がかり" として効いてくるためだ。

**受け入れ基準の核（F-6）**

| # | 受け入れ基準 | 根拠 |
|---|---|---|
| AC-1 配置と粒度 | マーカーはレポートの各セクション単位に配置され、ユーザーは「これ自分だ」と感じたセクションを個別にマークできること | システム要件F3.3／F3.1のセクション構造に依存 |
| AC-2 超低摩擦・可逆 | マーク／マーク解除がそれぞれ1タップで完了し、テキスト入力・確認ダイアログがないこと。タップ時に軽い視覚フィードバック（チェック表示等）が出ること | F-2「読みながら反射的に付ける」前提 |
| AC-3 記録と非プレッシャー | マークした項目がDB記録されF4.3/F5.3から参照可能であること。かつ、未マークを責める表示（残件数・「0件です」等の否定形）を一切出さないこと | 設計中枢 原則②／F5.1プレッシャーゼロ設計と一貫 |

**Out of Scope（F-7）**

1. マーカーへのコメント・メモ追記（テキスト注釈機能 — 超低摩擦原則に反する）
2. マーク数の集計表示・ゲーミフィケーション（「10個マーク達成！」等 — 三毒増幅・原則②で排除）
3. マーカーの共有・SNS投稿（F6シェアカードの責務。原則③「マウント材料化」リスク）
4. マーカーデータをF4チャットに注入する処理（F4.3の責務。F3.3は記録まで）
5. F4チャット内のマーカー表示（`KnowMarkerInChat` / TSK-UI-004の責務。F3.3はレポート画面上に限定）

> **運用評価指標（受け入れ基準ではない）**: しっくりきたマーカー利用率 10%以上（タップ数 / レポート表示数）。F9.3 KPIダッシュボードで計測。

## 4. F3.4 満足度アンケート

レポート読了直後に5段階評価＋自由記述を収集し、PMF基準（レポート満足度4.0/5.0）を計測する。V1継承機能。

| 項目 | 内容 |
|---|---|
| **Persona（F-1）** | 麻衣（レポートを読み終えた全ユーザーの代表）。主たる受益者は運営（PMF計測）だが、回答者＝麻衣として体験を定義 |
| **トリガー（F-2）** | F3.1レポートを末尾までスクロールして読了し、レポート末尾に到達した瞬間。アンケートはレポート末尾に配置。再生成後の再閲覧時の再表示頻度は Eval 層で詰める |
| **完了状態（F-3）** | 体験面 = 感想を手早く（5段階タップ＋任意のひとこと）伝え終え「声を届けた」感覚がある。スキップしても責められない。機械面 = 回答（5段階スコア＋自由記述）が `analytics_events` 等に記録され、F9.3 KPIダッシュボードのレポート満足度指標（4.0/5.0目標）に集計される |
| **上位Capability（F-4）** | CAP「統合像を届ける」。F3.4は「届けた統合像の品質を計測しPMF基準を検証するフィードバックループ」。F3.3＝ユーザー個人の検証 / F3.4＝運営の検証 |
| **実装フェーズ（F-8）** | Phase 1（F3.1に同梱）。V1継承・必須。PMF基準の計測手段のため初日から満足度データを蓄積する |

**ユーザーストーリー（F-5）**

> レポートを読み終えた麻衣として、「よかった／しっくりこなかった」を5段階＋ひとことで手早く返したい。それは、自分の声が次のレポートの改善に活きると感じられるためだ。

**受け入れ基準の核（F-6）**

| # | 受け入れ基準 | 根拠 |
|---|---|---|
| AC-1 低摩擦 | レポート末尾に5段階評価が表示され、1タップで選択・送信できること。自由記述は任意（空でも送信可） | V1継承・システム要件F3.4 |
| AC-2 非強制 | アンケート未回答でもレポート・後続機能（F4チャット等）に進めること。回答は完全に任意で、スキップ導線があること | 設計中枢 原則②（プレッシャーを作らない） |
| AC-3 計測連携 | 回答がDB記録され、F9.3 KPIダッシュボードの満足度指標（4.0/5.0目標）に集計可能であること。送信後・空状態テキストも言語設計準拠（否定形なし） | システム要件F3.4・F10.4 |

**Out of Scope（F-7）**

1. アンケート回答の強制（回答しないと先へ進めないゲート — AC-2と矛盾）
2. 多項目・詳細アンケート（NPS・機能別評価・デモグラ設問等 — V2は5段階＋自由記述のみ）
3. アンケート結果の他ユーザーへの開示（平均評価・レビュー一覧 — 口コミ機能は持たない）
4. 自由記述へのAI返信・対話化（F4共感AIチャットの責務）
5. アンケートのA/B出し分け・条件分岐ロジック（V2スコープ外）

## 5. F3.5 レポート再生成

課金プランのユーザーが月1回、継続利用の蓄積を反映した更新版レポートを受け取る。統合像を「生きた地図」として育て直す更新ループ。

| 項目 | 内容 |
|---|---|
| **Persona（F-1）** | 田中健太（主）= 「自分の変化が見えると嬉しい」「単発の診断では行動が変わらない」継続ユーザー。麻衣（副）。登録初日でなく、しばらく使った後に戻ってくる機能 |
| **トリガー（F-2）** | 前回生成から30日経過（システム的）＋ 継続利用の蓄積の末「今の自分でもう一度見たい」と感じた瞬間（心理的）。再生成が実際に変化を生むことが成立条件 |
| **完了状態（F-3）** | 体験面 = この1ヶ月の蓄積が織り込まれた更新版レポートを受け取り「前回より解像度が上がった／変化が見える」と感じる。機械面 = 蓄積データ注入の新レポートを生成・保存、課金状態を検証済み、月次再生成枠を消費（次回有効化は30日後）、旧レポートはDB保持するが履歴比較UIは出さない |
| **上位Capability（F-4）** | CAP「統合像を届ける」。F3.5は統合像を「生きた地図」として育て直す更新ループ。副軸C（体験深化）の中核 |
| **実装フェーズ（F-8）** | Phase 3（Day 43〜55）。課金依存（F7 Stripe実装後）＋蓄積依存（F4/F3.3/F5がPhase 1〜2で蓄積後）。F3で唯一 Phase 3 |

**ユーザーストーリー（F-5）**

> 1ヶ月COCOSiLを使ってきた田中（課金ユーザー）として、この1ヶ月のチャット・気づき・行動が織り込まれた更新版レポートを受け取りたい。それは、自分がどう変わってきたかを "今の地図" で確かめられるためだ。

**受け入れ基準の核（F-6）**

| # | 受け入れ基準 | 根拠 |
|---|---|---|
| AC-1 再生成ゲート | 再生成ボタンは前回生成から30日経過後にのみ有効化され、月1回上限を超えて生成できないこと。未経過時はボタン無効＋次回可能日を表示 | システム要件F3.5 |
| AC-2 課金ゲート | 無料プランからの再生成試行時は生成せず「有料プランで月1回再生成できます」の誘導を表示すること（言語設計準拠・禁止語0%） | D4／詳細要件§4.3例外系 |
| AC-3 蓄積反映 | 再生成レポートは前回生成以降のF4チャット・F3.3マーカー・F5記録を入力に含み、初回と差分のある内容になること。蓄積データが乏しい場合も生成は破綻せず成立すること | D3 |

**Out of Scope（F-7）**

1. 月1回を超える再生成（追加課金での即時再生成等 — 品質安定化・LLMコスト最適化）
2. 旧レポートとの履歴比較・差分表示UI（変化は新レポートの内容で伝える）
3. 再生成の自動実行（定期的な自動更新・プッシュ — 三毒の不安煽りを避ける・原則②）
4. 生年月日の変更・再入力（生年月日は不変前提。再生成は生年月日修正の手段ではない）
5. gpt-image-2による再生成時のビジュアル刷新（imager2アーキ フェーズ3）
