## Turn 1｜症状の解剖 — 「テンプレートとリアルの乖離」は何問題か

**伊藤：** 正直に言います。今回のケース、私が一番気になるのは「`.github/` ディレクトリがゼロ」という事実ではなく、**ドキュメントと現実の間に挟まった前提の亀裂**です。

1shot promptとharness docsは3人チーム・モノレポを前提に書かれている。でも実態は2人・フラットアプリ。この前提ズレのまま実行したら、CODEOWNERSに `@hirame` という存在しないGitHubユーザーを書くことになる。具体的に、harness docsのCODEOWNERS案がこれです：
```
packages/chat/src/prompts/ @enma
packages/diagnosis/          @enma @hirame
apps/api/ supabase/          @hirame
apps/web/                    @maami
```
`packages/` も `apps/` もこのプロジェクトには存在しない。これをそのまま貼ったらCODEOWNERSがエラーになるだけです。

これは「**テンプレートの文脈とプロジェクトの文脈が分離している問題**」。harness docsは「プロジェクトが到達すべき完成形」を書いている。でも今必要なのは「2人のチームが今週入れられるもの」です。

**西村：** 伊藤さんの診断に同意しつつ、もう一層深めます。1shot promptの扱いについては、「全部実行」も「無視」も間違い。1shot promptが生成しようとしている `.claude/skills/`・`.claude/commands/` は「AIへの文脈注入インフラ」です。

現状の `AGENTS.md` には「えんまさはバックエンド担当、遠藤はフロントエンド担当」と書かれているが、`.claude/commands/cocosil-work.md` のような「担当外ディレクトリへの操作をブロックするverb-skill」が存在しない。AIが遠藤さんのセッションで `supabase/` を勝手に触るリスクをゼロにする仕組みがない。

2つの別問題：
1. **GitHub CI問題** → Layer 1を今の構造に合わせて作る
2. **AI文脈注入問題** → 1shot promptの核心部分だけを今の状態に合わせて実行する

**みさき：** え待って〜。「AIが担当外を触る」ってどういうことですか？私がClaude使うとき普通に指示したら動くんですけど〜。

**西村：** たとえば遠藤さんが「F3チャット機能のUIを修正して」とClaude Codeに頼んだとします。Claudeがコードを読んで「あ、チャットのロジックは `supabase/functions/` にもあるな」と判断して、DBの関数も書き換え始める。でも `supabase/` の変更にはRLS設計の知識が必要で、それはえんまさの担当。「AIが親切心から担当外を触った」事故。これを防ぐのがverb-skillの「担当ガード」です。

**みさき：** あ〜〜それマジやばい！知らずにAI使ってたら絶対事故りそう〜。

## Turn 2｜選択肢の発散 — 「Now / Soon / Later」軸で全施策を分類

**西村：** 選択肢を**即効性 × コスト × リスク回避**の3軸で並べます。

| 施策 | 今すぐ価値 | コスト | リスク（入れないと） | 分類 |
|---|---|---|---|---|
| `.github/workflows/ci.yml`（typecheck+lint） | ★★★ | 中60分 | 型エラーのまま通る | 🔴 Now |
| `.claude/skills/cocosil-domain/SKILL.md` | ★★★ | 中45分 | AI担当外干渉 | 🔴 Now |
| `.claude/commands/cocosil-work.md` | ★★ | 小30分 | 担当ガードなし | 🔴 Now |
| `CODEOWNERS`（2人版） | ★★ | 小20分 | レビュー漏れ | 🔴 Now |
| `pull_request_template.md` | ★★ | 小20分 | チェック漏れ | 🔴 Now |
| `.claude/skills/frontend-nextjs/` + `backend-supabase/` | ★★ | 中120分 | 役割別文脈なし | 🟡 Soon |
| Vitest + `test` script | ★ | 大120分 | テストがない状態継続 | 🟡 Soon |
| `security.yml` | ★★★ | 大120分 | シークレット漏洩 | 🟡 Soon |
| `deploy.yml` | ★★★ | 大90分 | 手動デプロイのミス | 🟡 Soon |
| `dependabot.yml` | ★ | 小30分 | 依存関係の古さ | 🟢 Later |

**伊藤：** ci.ymlに `build` ジョブを入れるかどうかは要注意です。`pnpm build` はNext.js 16が環境変数を要求する。`NEXT_PUBLIC_SUPABASE_URL` などがGitHub Secretsに登録されていない状態では `lib/env.ts` のZodバリデーションが通らずビルドが落ちる。

提案：**Day 1のci.ymlは `typecheck + lint` の2ジョブのみ**。buildはSecrets登録後、testはVitest導入後に追加する。最初のCIは「確実に緑になる」範囲だけ。

**西村：** 同意。`build` はSecrets登録後に追加する形で。

**みさき：** typecheck と lint って何が違うんですか〜？なんか同じじゃないですか？

**伊藤：** typecheckはTypeScriptが「型の整合性」を確認するもの、lintはESLintが「コーディングスタイル・潜在的バグ」を確認するもの。「文法チェック」と「作文の質チェック」みたいな感じです。

## Turn 3｜要件の確定 — 2人チームの「本当に必要なもの」とテンプレの呪縛を切り分ける

**みさき：** てか、CODEOWNERSって2人しかいないのに意味あるんですか〜？

**伊藤：** 正直に言います——**2人チームにおけるCODEOWNERSの価値は7割引きです**。でも「将来の3人目への投資 + AIへの文脈提供」として最小版で作る価値はある。

**西村：** Context Layering の抽象パターンで整理します：

```
Layer A（全エージェント共通）: AGENTS.md（既存・維持）
Layer B（役割別コンテキスト）: .claude/skills/（追加）
Layer C（タスク実行プロトコル）: .claude/commands/（追加）
```

1shot promptの `/frontend/CLAUDE.md`・`/backend/CLAUDE.md` は作りません。CLAUDE.mdを増やすと「どれを読めばいいか」が分からなくなる。代わりに `.claude/skills/frontend-nextjs/` と `.claude/skills/backend-supabase/` に吸収させます。

**みさき：** えっと、要するに「CIとAI用のドメイン知識ファイルを先に入れて、テストは後回し」ってことですよね？テストなしのCIって意味あるんですか〜？

## Turn 4｜技術深掘り — 「CI vs AI harness 先行」論争と、フラット構造への翻訳法則

**伊藤：** 明確にします。**typecheck + lint の2ジョブは「テストの代替」ではなく「バグの第一層ガード」**です。

Next.js × TypeScript プロジェクトで発生するバグの過半数は「型エラーが通っていた」か「importが壊れていた」ことが原因。Vitestのunit testは「ロジックの正確性」を守る。typecheckは「コードが繋がっているか」を守る。別の層のガード。「typecheck+lint CI → 機能実装 → Vitest導入 → test CI追加」の順番は正しい。

フラット構造版ci.yml（抜粋）：
```yaml
- uses: pnpm/action-setup@v4
  with:
    version: 10.13.1
- uses: actions/setup-node@v4
  with:
    node-version: '20'
    cache: 'pnpm'  # ~/.pnpm-store を自動キャッシュ
- run: pnpm install --frozen-lockfile
- run: pnpm exec tsc --noEmit
```
モノレポ版との違いは `working-directory: apps/web` がない点。`actions/setup-node@v4` の `cache: 'pnpm'` で2回目以降は30秒以内に完了。

**西村：** フラット構造版CODEOWNERSの翻訳法則：

```
# モノレポ版 → フラット版
packages/chat/src/prompts/  @enma    →    app/api/chat/  @MasakiEndo44
apps/api/                   @hirame  →    supabase/      @MasakiEndo44
```

**翻訳の法則：モノレポのパッケージ境界は、フラットアプリでは `app/` のサブディレクトリ境界に対応する。**

**みさき：** あ〜！「モノレポとフラットって、パスが違うだけで同じ考え方」ってことですね〜。でも遠藤さんのGitHubユーザー名知らないと完成しないじゃないですか〜。

**西村：** その通りです。CODEOWNERSは「GitHubユーザー名が確定してから完成させる」のが鉄則。

## Turn 5｜原則統合 — ハーネス設計の3原則と「今週実行できる」具体アクション

**西村：** COCOSiL V2 ハーネス設計の原則を3つに集約します。

### 【COCOSiL ハーネス設計の3原則】

**① Adapt Before Add（追加より適応を先に）**
テンプレートを「現状に合わせて適応」してから追加する。モノレポ前提のCODEOWNERSをそのまま貼らない。3人前提の1shot promptをそのまま実行しない。

**② CI Credibility First（信頼できる緑から始める）**
最初のCIは「確実に緑になる」範囲だけ。typecheck + lint の2ジョブ。buildはSecrets登録後、testはVitest導入後。「赤いCI」は信頼されない。

**③ Context Layering（AI文脈は層別に注入する）**
`AGENTS.md`（Layer A）→ `.claude/skills/`（Layer B）→ `.claude/commands/`（Layer C）。CLAUDE.mdを増やすのではなく、既存 `AGENTS.md` を維持しながら下位層を追加する。

**伊藤：** 具体アクション：

**Phase 1 — Day 1（2時間・5ファイル）：**
1. `.github/workflows/ci.yml`（typecheck + lint の2ジョブ）
2. `.claude/skills/cocosil-domain/SKILL.md`（ドメイン知識・担当ガード）
3. `.claude/commands/cocosil-work.md`（担当ガード付き実装コマンド）
4. `.github/CODEOWNERS`（@MasakiEndo44 で最小版）
5. `.github/pull_request_template.md`（7項目チェックリスト）

**Phase 2 — Week 1（4時間）：**
6〜7. `.claude/skills/frontend-nextjs/` + `.claude/skills/backend-supabase/`
8〜9. Vitest導入 → ci.ymlに `test` ジョブ追加
10〜11. `security.yml` + `deploy.yml`（Secrets登録後）
12. `dependabot.yml`

**Phase 3 — GitHub UI手動（20分）：**
13. Branch Protection Rules（PR必須 + status check必須）
14. GitHub Secrets登録

**みさき：** なんか最初「テンプレートそのまま使えばいいじゃん」って思ってたんですけど、「今の2人チームに合わせてアダプトしてから使う」ってことなんですね〜。「CIを小さく緑にして育てる」って、植物を育てるみたいで好き〜！先に根っこ張って、段階的に葉っぱ増やしていく感じ〜。

## ✅ ハーネスセットアップ方針まとめ

| 項目 | 方針 |
|---|---|
| 1shot prompt の扱い | 全実行しない。`skills/` + `commands/` の該当部分のみ抽出・適応して実行 |
| CLAUDE.md の扱い | 新規作成しない。既存 `AGENTS.md` を Layer A として維持 |
| CODEOWNERS | フラット構造に翻訳。`@MasakiEndo44` で最小版作成、遠藤さんのユーザー名は後追記 |
| ci.yml Day 1 | typecheck + lint の2ジョブのみ。build・testは後回し |
| Vitest 導入 | Week 1後半（機能実装が始まってから） |
| security.yml | Week 1（Layer 2）。Day 1は既存PreToolUse Hookで代替 |
| deploy.yml | GitHub Secrets登録後に作成 |
| AI文脈注入 | 3層：AGENTS.md → `.claude/skills/` → `.claude/commands/` |
| Day 1 所要時間 | Phase 1 = 2時間（5ファイル） |
