# 議論ログ：GitHub PR「Review required」でマージできない問題の解決方針

## 登場人物

- 🧑‍💼 **小田（GitHub DevOps専門家）**：GitHub Actions・branch protection歴9年。OSSリポジトリ管理とチーム開発フロー設計が専門。
- 👩‍💼 **末松（開発チーム設計専門家）**：スタートアップCTO経験を持つエンジニアリングマネージャー。3〜5人の小規模チームの立ち上げ経験が10件以上。
- 💅 **みさき（ギャル）**：IT素人だけど感性は鋭い。GitHubのPRって「なんか赤バッジが出て通れない感じ」で認識している実際のCOCOSiLユーザー。

---

## Turn 1｜「自分のルールで自分が詰む」構造の正体

**小田：** 正直に言います。これは典型的な「Bootstrapping Paradox」です。ハーネスを整備するためのPRが、そのハーネスのルールによってブロックされる——GitHubのCODEOWNERS設計で一番よくある初期設計ミスです。

具体的に見ると、現在の `.github/CODEOWNERS` はこうなっています：

```
.github/     @MasakiEndo44
.claude/     @MasakiEndo44
AGENTS.md    @MasakiEndo44
```

そして Branch Protection は「Require Code Owners review」オン。

つまり **えんまさが変更したファイルのCode Ownerがえんまさ自身** → GitHubのルール上、PRの作成者は自分のレビューを承認として計上できない。結果、永久に1承認が足りない状態になります。

これは「審判が自分の試合を裁けない」問題で、**役割の分離原則が初期フェーズには重すぎる**という構造的問題です。

**末松：** 小田さんの指摘は正しいんですが、私は少し違う診断をします。「ルールがきつすぎる」のではなく、「**Bootstrapping期専用のバイパス設計が欠けている**」と見ています。

スタートアップで10件チームを立ち上げてきた経験から言うと、初期のハーネスPR（ルール自体を作るPR）と、通常の機能PRは**別のレビュー基準で扱うのが普通**です。Stripe、GitHubそのものも、内部のガバナンスドキュメントの初期投入は管理者バイパスで入れています。

結局これは「プロセスの冷却期間がゼロ状態でルールを有効化した」問題。ルール自体は正しい。**適用タイミングの設計が欠けていただけ**です。

**みさき：** え待って〜、つまり「ドアに鍵かけたら自分が家に入れなくなった」ってこと？それって鍵が悪いの？それとも鍵のかけ方が悪いの？

**小田：** みさきさん、その例え、まさにそれです（笑）。今の状態を正確に言うと「鍵はいいんですが、**合鍵を誰かに渡す前に鍵をかけてしまった**」状態です。

## Turn 2｜4つの選択肢を「速度 × 安全性 × 設計負債」で仕分ける

**末松：** では選択肢を整理します。評価軸は3つ。**速度**（今日マージできるか）、**安全性**（mainが壊れるリスク）、**設計負債**（後で後悔しないか）です。

| 選択肢 | 速度 | 安全性 | 設計負債 |
|---|---|---|---|
| A. ヒラメ/まあみを招待→承認待ち | 🔴 遅い（招待受諾待ち） | ✅ 高い | ✅ なし |
| B. Code Owners required を一時オフ | 🟢 今日OK | 🟡 中（手動戻し忘れリスク） | 🟡 手動戻しが必要 |
| C. 直接main push | 🔴 ブロック済み | ❌ 不可 | ❌ Ruleset破壊 |
| D. Admin bypass でマージ | 🟢 今日OK | ✅ CIは通っている | 🟡 設計上の盲点が残る |

**小田：** ここに「E案」を追加したいです。**CODEOWNERS の bootstrap修正 + Admin bypass の組み合わせ**です。

具体的には：
1. CODEOWNERSを直す（`.github/` 系の初期ハーネスPRはチーム全員からのreviewでよい、もしくはBootstrap専用ownerを設ける）
2. 今回のPR #2 は Admin bypass でマージ
3. 以降は正規フローが機能する

これが「正しい設計負債ゼロ」の選択です。

**みさき：** え、Admin bypassってなんですか？えんまさって「管理者」なんですよね？要するに「マネージャーが例外処理できる機能」みたいな感じ？

**末松：** 正解です。GitHubでは Repository Owner は Ruleset の「Bypass List」に自分を追加できます。これは「緊急時の非常ドア」として設計されていて、使用はGitHubのAudit Logに残ります。**透明性があるバイパス**なので、「こっそり外す」とは違います。

## Turn 3｜初期ハーネスPRの特殊性 — 「通常PR」と同じ基準で裁くべきか？

**みさき：** ちょっと聞いていいですか〜？このPRって「ルールを作るPR」なんですよね？それって普通の機能追加のPRと同じに扱っていいんですかね？なんか根本的に違うような気がして〜。

**末松：** みさきさん、それが今回の議論の核心を突いてます。**ガバナンスPRと機能PRは本質的に異なります**。

実際に私が見てきた小規模チームのパターンを整理すると：

| PRカテゴリ | レビュー基準 |
|---|---|
| 機能PR（F1〜F7実装など） | 担当レイヤーのCode Ownerレビュー必須 |
| インフラPR（DB migration等） | ヒラメレビュー + えんまさ確認 |
| **ガバナンスPR（AGENTS.md, CODEOWNERS等）** | **チームオーナー承認 or Admin bypass** |
| セキュリティ緊急対応 | Admin bypass + 事後レビュー |

初期セットアップPRはこの「ガバナンスPR」カテゴリです。そして **チームオーナーがえんまさ1人しかいない今の段階** では、Admin bypass以外の選択肢がそもそもない。

**小田：** 私はここで1点、末松さんと強調点が違います。「今回は特例でよい」とするのではなく、**この構造自体をCODEOWNERSとRulesetに明示的に組み込む**べきだと思います。

GitHubのRulesetには「Bypass List」という機能があって、「特定のロールはルールをスキップできる」と明示できます。これを使えば「Admin bypassは設計の一部である」という事実をコードで表現できます。

モヤっとした「緊急時の例外」を、**透明な設計上の正当な経路**にする——これが設計負債ゼロの方法です。

**みさき：** あ〜、なんか「非常口」を設計図に書いておく感じ〜！それはめっちゃ納得〜。

## Turn 4｜CODEOWNERSの設計問題 — 「えんまさが自分をブロックする構造」の法則

**小田：** Turn 1で触れた問題を、もう少し法則化します。今のCODEOWNERSには**Self-Blocking Anti-Pattern**という構造的問題があります。

```
# 現状（問題あり）
.github/     @MasakiEndo44  ← えんまさだけ
.claude/     @MasakiEndo44  ← えんまさだけ
AGENTS.md    @MasakiEndo44  ← えんまさだけ
```

このパターンが危険な理由：えんまさが `.github/` を変更する全てのPRで、えんまさ以外のレビュアーが必須になる。まあみやヒラメがまだコラボレーターでない今、**構造的に誰もマージできない**。

修正案は2方向あります：

**方向A（シンプル）：** Bypass Listにえんまさ（Admin）を追加する。CODEOWNERS自体は変えない。
**方向B（設計込み）：** CODEOWNERSの `.github/` 行を全員（or チームAlias）に変更する。

```
# 修正後（方向B）
.github/     @MasakiEndo44 @shuichiro16 @maami415  ← 全員
```

こうすると「誰かが.github/を変えたら他の2人のうち1人がレビュー」という正常な分業になります。

**末松：** 私はここで小田さんに反論があります。方向Bは**チームが揃ってから正しい設計**です。まあみとヒラメがまだコラボレーターでない今に方向Bを採用すると、3人全員を記載しているが**実質1人しかレビューできない**という見かけだけ豪華な構造になります。

今の正直な設計は：
1. **短期**：Bypass Listにえんまさを追加（方向A）
2. **コラボレーター招待後**：CODEOWNERSを方向Bに更新

これが「**現在の現実を設計に正直に反映する**」原則です。2人が招待されていないのに3人のCODEOWNERSを書くのは、設計の虚偽表示です。

**みさき：** あ〜なんか「大家族で住む家を最初から建てちゃったけど、まだ2人しか来てない」みたいな感じですよね。スペースはあるけど人がいない〜。

**末松：** その例えが正確です。そして**空の部屋は管理コストになる**。今の段階はまず「今住んでいる人の設計」が正解。

## Turn 5｜Bootstrapping設計の3原則 → 今日のアクション

**末松：** 議論を通じて見えてきた設計原則を3つに集約します。

### Bootstrapping Governance の3原則

**① Transparent Bypass（透明なバイパス）**
Admin bypassは「こっそり外す」のではなく、Ruleset Bypass Listに明示的に設定する。Audit Logで追跡可能にすることで、「例外」を「設計の一部」に変える。

**② Honest CODEOWNERS（現実に正直な所有者定義）**
招待されていないメンバーをCODEOWNERSに書かない。チームの現状を反映した設計を保ち、人が増えるたびに更新する。

**③ Bootstrap PR ≠ Feature PR（初期ガバナンスPRの別扱い）**
ルールを作るPRと、ルールに従うPRは本質的に異なる。初期段階はAdmin権限者がBootstrap PRをbypassマージし、以降の全PRから正規フローを適用する。

**小田：** 今日から取れる具体アクションを順に並べます：

🔴 **今日（所要20分）**
1. GitHub → Settings → Rulesets → `main-protection` → Bypass List にえんまさ（Admin）を追加
2. PR #2 を Admin bypass でマージ
3. マージ後、Bypass Listがそのままでよいか確認

🟡 **ヒラメ・まあみ招待後（所要30分）**
4. GitHub → Settings → Collaborators → @shuichiro16 / @maami415 を招待
5. 招待受諾確認後、CODEOWNERSを全員記載に更新
6. 更新PRはヒラメかまあみに初レビューをもらう（初の正規フロー検証を兼ねる）

🟢 **今後の設計として**
7. HARNESS_DECISIONS.md に「Bootstrapping期のAdmin bypassを設計に組み込んだ」根拠を追記

**みさき：** なんか最初は「ルールを作ったのにマージできない謎すぎる〜」って思ってたんですけど、「自分で鍵かけて自分が入れなかっただけ」で、鍵は正しかったんですよね。合鍵（バイパス）を設計に入れておくのが答え、めっちゃすっきりしました〜！あとCODEOWNERSって「今いる人だけ書く」のがマナーなの、初めて知りました〜。

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| 今回のPR #2 のマージ方法 | Admin bypass（Ruleset Bypass Listにえんまさを追加後にマージ） |
| CODEOWNERSの修正タイミング | コラボレーター招待後に3人全員記載に更新 |

| 初期ガバナンスPRの扱い | Feature PRとは別カテゴリ。Bypass対象として設計に明示 |
