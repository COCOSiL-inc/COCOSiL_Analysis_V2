## Turn 3｜要件・範囲の確定——実需をパターン化して優先度へ落とす

**💅 みさき（実需持ち込み）：**

てか実際にユーザーとして使うとして、一番怖いのってなんだと思います？私、アプリ開いたときに「診断結果がいつも違う」とか「途中でエラーになった」とかが一番萎えるんですよね〜。あと「入力したのに保存されてなかった」も超やだ。そういうのを確認できる仕組みってある感じですか？

**古川 誠一郎（パターン化）：**

みさきさんの指摘は的確です。「診断結果がいつも違う」「保存されてなかった」——これを抽象化すると、**3種類の壊れ方**に集約されます。

```
壊れ方パターン：
① ロジック破綻：同一入力で違う出力（MBTI境界値・六星占術計算）
② 状態不整合：書き込んだのに読めない（Supabase RLS + Clerk JWT の組み合わせ）
③ UX断絶：APIは動くがUIが期待通りでない（レスポンス構造の仕様ズレ）
```

この3パターンに対応する確認が「完了の定義」になります。機能単位で「この機能は①〜③のどれをリスクに持つか」を事前に宣言するだけで、確認スコープが明確になります。

**望月 さくら（優先度整理）：**

パターン化を受けて、COCOSiL V2のF1〜F5に対する優先度を整理します。

| リスクパターン | 検証手段 | 優先度 |
|---|---|---|
| ① ロジック破綻 | Vitest unit test（lib/diagnostics/） | 🔴 F2実装前に必須 |
| ② 状態不整合 | Vercelプレビュー + Supabase branch + 手動ログ確認 | 🔴 F1認証完了後に必須 |
| ③ UX断絶 | えんまさによるプレビューURL手動確認 + チェックリスト | 🔴 全機能・PR単位で必須 |

**結論：F1（ウェルカム・認証）が完了した現時点で、最初に整備すべきは「Vercelプレビューデプロイ + えんまさ確認チェックリスト」です。** これがゼロコストで②③両方をカバーします。

## Turn 4｜技術深掘り——個別ノウハウから法則・制約を抽出する

**望月 さくら：**

具体的な実装方法を示します。Vercelプレビュー + 確認チェックリストの実装は以下の通りです。

```yaml
# .github/workflows/preview.yml
name: Preview Deploy
on:
  pull_request:
    types: [opened, synchronize]
jobs:
  deploy-preview:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - run: pnpm install
      - uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
```

PRを出すとVercelが自動でプレビューURLを生成し、PRにコメントします。えんまさはそのURLをブラウザで叩くだけ。さらに確認チェックリストをPRテンプレートとして `.github/PULL_REQUEST_TEMPLATE.md` に埋め込めば、ヒラメがPRを出すたびにチェックリストが自動生成されます。

**古川 誠一郎（法則化）：**

望月さんの実装を受けて、**法則として抽出**できることがあります。

ここで見えてきた本質的制約は「**人間ループの設計**」です。自動化できる部分（CIが型チェック・ビルドを実行）と人間がやる部分（えんまさが「これは仕様通りか」を判断）を**明示的に分離する**ことが鍵です。

「**Acceptance Gate**」の導入が有効です。PRにマージする前に、設計者が「✅ 動作確認済み」のラベルを付けることをCIの必須条件にする。GitHubのBranch Protectionの「Required approvals」に「えんまさのApproval」を追加するだけで実現できます。

```
機能完了の法則：
型チェック通過（機械） + ビルド成功（機械） + 動作確認済み（人間） = マージ可能
```

この3つが揃って初めて「実装完了」とすることが、手戻りをゼロに近づける唯一の道です。

## Turn 5｜方針統合・結論——設計原則と具体的次の一手

**古川 誠一郎（設計原則）：**

議論を通じて、COCOSiL V2のアトミック確認ループは以下3原則に集約されます。

```
① Define Before Build（作る前に「完了の定義」を宣言する）
  機能実装を始める前に「ロジック破綻・状態不整合・UX断絶」の
  どれをリスクに持つかを /start-task 時に宣言する。

② Human Gate, Machine Floor（機械は床、人間は門）
  型チェック・ビルド・lintは機械が自動実行（床：最低限）
  「仕様通りか」の判断はえんまさが人間として実行（門：マージ条件）

③ Smallest Loop First（最小のループを先に回す）
  E2E・Vitest導入より先に「プレビューURL + チェックリスト」を整備する。
  3日かかる自動化より、今日から動ける人間ループが優先。
```

**望月 さくら（明日からの具体アクション）：**

実際にやるべきことを優先順に並べます。

```
🔴 今週中（1〜2日）：
1. .github/workflows/preview.yml を作成（Vercelプレビュー自動生成）
2. .github/PULL_REQUEST_TEMPLATE.md を作成
   → 確認チェックリスト3項目を埋め込む：
      - [ ] プレビューURLで動作確認済み（担当：えんまさ）
      - [ ] レスポンス構造が型定義と一致（担当：ヒラメ）
      - [ ] UXシーケンス（共感→安心→分析→行動）を崩していない（担当：えんまさ）
3. HARNESS_HEALTH.md に G9「プレビューデプロイ未整備」を追加し、解消目標を記録

🟡 F2実装開始前（Sprint 2）：
4. Vitest導入 + lib/diagnostics/ のunit test（G1解消）
5. pnpm test を ci.yml に追加

🟢 Sprint 3以降：
6. Supabase branch + ステージング環境（重厚路線は後回し）
```

**💅 みさき：**

なんか最初「テストとかCI/CDとか難しそう〜」って思ってたんですけど、えんまさがURLを開いてチェックリストに✅つけるだけで手戻りが減るなら、マジで明日からできそうですよね〜！「仕様書と動いてるものが違う」ってのが一番やばいと思ってたので、えんまさが「門番」になるっていう役割、めっちゃ理にかなってると思います。COCOSiLって「人間の自己理解」のプロダクトじゃないですか？開発の確認も「人間が最後に判断する」ってのは、なんかプロダクト哲学と一致してる気がして好きです〜。
