## Turn 1｜現状診断：なぜドキュメントは読まれないか

**岡部：** COCOSiL V2のdocs構造を見て、まず感じるのは「設計の圧力は高いが、ナビゲーション構造が意図とズレている」ということです。

現状の問題を診断すると3層構造になっています。

**表層症状**：`docs/` 配下が `input/`・`output/`・`discussions/`・`harness/` に分かれていて、各自がどこを読めばいいか不明。`ONBOARDING.md` は存在するが「困ったときの地図」が汎用的すぎる。

**中層構造**：docs配下のファイル数が60超。discussions配下だけで35ファイル。ヒラメが「DB構造の整理をしたい」と思ったとき、参照すべき要件定義書が `docs/output/requirements/cocosil_v2_detailed_requirements_specification.md` にあることをどうやって知るか——ONBOARDING.mdにそのパスが書いていない。

**根本診断**：これは「ドキュメントの量の問題」でなく、**「目的→場所」のマッピングが欠落している**という問題です。ユーザー（開発者）は「DB設計を始めたい」「F2のUIコンポーネントを設計したい」という目的でdocsに入ってくる。でも今のONBOARDINGは「役割別（ヒラメ/まあみ/AI）」のナビしかない。役割≠目的です。

---

**藤本：** 岡部さんの診断に補足すると、**「役割別ナビは入口に強いが、作業途中に役立たない」**という非対称性があります。

私が8年で見てきたパターンで言うと、ドキュメントが死ぬタイミングは2つです。①着手時に正しい文書を見つけられない ②実装途中で「この仕様はどこに書いてあったか」を探せない。

COCOSiL V2は、TASK管理単位がまだ整備されていない。これが致命的です。GitHub Issueは「何をするか」の管理ツールで、TASKファイルは「どうやるか・今どこまでやったか」の管理単位。この2つが紐づいていないと、**ヒラメが手動でDBを変更したときにどのIssueの何番TASK進捗が更新されたか、誰も追えない**。

特に今回の「手動実装でハーネスを修正する」という要件は、つまり「AIエージェントを通さない実装があったとき、ドキュメントの鮮度をどう保つか」という問題です。これはGitHub IssueとTASKファイルの**機械的な連動設計**で解決できます。

---

**みさき：** え待って、ちょっといいですか〜？私まあみ的な立場で言うと、ONBOARDINGは読みましたけど、「F2のUIコンポーネント設計をしたい」って思ったとき、どこ見たらいいか全然わかんなかったんですよね〜。AGENTS.mdに全部書いてあるって言うけど、あれ読むのだけで1時間かかるし…。

「今の私のタスクに必要なドキュメントだけ」を5秒でたどり着けるやつが欲しいです〜！マジで。

## Turn 2｜設計の選択肢：ONBOARDING vs TASK-INDEX vs ハイブリッド

**藤本：** みさきの指摘が本質的です。「今のタスクに必要な文書だけ」というニーズは、**「役割別」でも「時系列別」でもなく、「機能別×作業フェーズ別」のマトリクス**で解決します。

選択肢を3つに整理します：

| 案 | 設計 | 強み | 弱み |
|---|---|---|---|
| **A: ONBOARDING強化** | 既存ONBOARDINGに「機能別参照マップ」セクションを追加 | 既存ファイル拡張で移行コスト低い | ファイルが肥大化、500行超えると誰も更新しなくなる |
| **B: TASK-INDEX.md新設** | `docs/TASK-INDEX.md` で機能×フェーズのマトリクスを管理 | タスク追跡と文書ナビを統合できる | 管理対象が増える |
| **C: ハイブリッド（推奨）** | ONBOARDING = 「今日動くための地図」（200行以内を死守）、TASK-INDEX = 「機能別×フェーズ別の参照マップ+TASK管理」 | 役割分担が明確、更新しやすい | 2ファイルへの習慣化が必要 |

**私の推奨はC**です。設計原則は **Separation of Concerns**——「今日動き出すための情報」と「特定機能の実装を進めるための情報」は読まれる文脈が違う。混ぜると両方が死ぬ。

実装ロードマップを見ると、F1〜F10の機能群があります。TASK-INDEX.mdは `機能ID（F1, F2...）× フェーズ（設計中/実装中/完了）` の表として作り、各セルに「参照すべきドキュメント」と「GitHub Issue番号」を記載する構造にします。

---

**岡部：** 藤本さんの C 案に同意しつつ、**TASKファイルの粒度**について明確にしておく必要があります。

「TASKファイル」は2つの文脈で使われがちです。①プロジェクト管理ファイル（全体進捗管理）と②個別実装手順書（NEARフェーズ実装計画のようなもの）。COCOSiL V2の `impl-plan-near-soft-feedback-loop.md` を見ると、すでに個別実装手順書の文化がある。

私の提案は、**「TASK管理の単位をGitHub Issueに一本化して、TASKファイルは詳細実装仕様書（Issue#N付き）として位置づける」**ことです。さらにTASK IDは `TSK-[分類]-[番号3桁]` 形式とし、分類は DB/UI/API/PROMPT/DOCS/CHORE で機能横断的に管理できるようにします。

```
GitHub Issue #12: DB構造整理（ヒラメ）
  └── docs/output/tasks/TSK-DB-001-db-schema-cleanup.md
        ├── 参照要件: cocosil_v2_detailed_requirements_specification.md §4.1〜4.3
        ├── 変更予定ファイル: supabase/migrations/**
        └── 完了定義: typecheck通過 + えんまさ確認
```

この構造なら、ヒラメが手動でDBを変更したとき「TSK-DB-001.md の実装状況を更新する」という単一のアクションで追跡できます。

---

**みさき：** あ〜なるほど！つまりGitHub IssueとTSKファイルが1対1で紐づいてて、IssueのURLを貼るだけでどのTSKか分かる感じですよね？

それだったら私みたいなFE担当でも「TSK-UI-001のコンポーネント設計ここまでやった」って書けるし、めっちゃわかりやすい〜！あとF2のUIって要件書のどこ見ればいいですか？それも書いといてほしい〜。

## Turn 3｜具体的な要件：何が変わると嬉しいか

**岡部：** みさきの「F2のUIって要件書のどこを見ればいい？」という質問は、**「TASK-INDEXの機能別参照マップ」が解決すべきユースケースそのもの**です。

COCOSiL V2の機能構成から、必要な参照マップを具体化します：

| 機能ID | 機能名 | 担当 | 要件定義書の場所 | 設計仕様書 | GitHub Issue |
|--------|--------|------|----------------|-----------|-------------|
| **F1** | オンボーディング・登録 | まあみ（UI）/ ヒラメ（API） | 要件定義書 §4.1 | `docs/output/F1/f1_detailed_requirements.md` | #TBD |
| **F2** | 性格分析・自動診断 | まあみ（UI）/ ヒラメ（自動計算） | 要件定義書 §4.2 + §4.2 F2.1a/b/c | なし → 作成必要 | #TBD |
| **F3** | 統合レポート | まあみ（UI）/ えんまさ（プロンプト） | 要件定義書 §4.3 | なし → 作成必要 | #TBD |
| **F4** | 共感AIチャット | まあみ（UI）/ えんまさ（プロンプト）/ ヒラメ（API） | 要件定義書 §4.4 | なし | #TBD |

この表が `docs/TASK-INDEX.md` に入れば、**みさきが「F2のUIコンポーネント定義をしたい」と思ったとき、TASK-INDEXを開いて F2 の行を見れば5秒で正しいドキュメントにたどり着けます**。

**藤本：** 次に「/start-task でIssueと紐づける」という要件の具体設計です。

現在の `/start-task` コマンドは：
1. `git status` チェック
2. タスク理解
3. 設計中枢チェック
4. ブランチ名提案
5. ブランチ作成

ここに **Step 2.6 として「対応GitHub Issueの確認と紐づけ」** を挿入します：

```
Step 2.6（新規）: GitHub Issue紐づけ
─────────────────────────────────────
タスク説明にIssue番号（例: #12）が含まれる場合:
  → `gh issue view 12` で内容確認
  → TSK-**ファイルが存在すれば、そのパスを表示
  → 存在しなければ「TSK-[分類]-NNN を作成しますか？」と確認

Issue番号がない場合:
  → `gh issue list` で未割り当てIssueを表示
  → 「対応するIssueはありますか？（番号を入力 / スキップ）」
```

## Turn 4｜手動実装時のドキュメント更新問題：法則と制約

**藤本：** みさきの疑問が核心です。「AIエージェントを通さない手動実装があったとき、誰がどのドキュメントを更新するか」——これは **「強制力と習慣化のトレードオフ」**という根本的な設計問題です。

まず法則を整理します：

**法則①：強制力が高い仕組みほど裏口ができる**  
`supabase db push` をhookでブロックするのはLayer 1制約として機能しています。同じ発想でPRにチェックリストを入れても、「TSKファイルを更新しましたか？ ☑」はマージ前に機械チェックできません。

**法則②：更新コストを下げた仕組みだけが生き残る**  
TASKファイルを更新する行為のコストが「5秒以内」でなければ、繁忙時には確実にスキップされます。

**法則③：GitHub PRレビュープロセスに組み込むのが最も生き残る**

私の提案する「ハーネス修正」の設計は：

```
PR作成時 → .github/pull_request_template.md に追記:
─────────────────────────────────────────────────
## TASK 更新確認（全PR必須）
- [ ] 対応するTSK-**ファイルの「実装状況」セクションを更新した
- [ ] 参照している要件定義書の該当セクションに変更がある場合は反映した
- [ ] TASK-INDEXの進捗列（設計中/実装中/完了）を更新した
─────────────────────────────────────────────────
```

これなら「ヒラメがDB変更を手動で行った後、PRを出すときにこのチェックリストを見て更新する」というフローが自然に生まれます。**機械チェックではなくPRレビュー文化として埋め込む**設計です。

---

**岡部：** 藤本さんの法則に加えて、**「/finish-task でTSKファイルの存在をチェックする」**という機械的ガードが効きます。

現在の `/finish-task` は typecheck + lint + commit + PR作成をします。ここに：

```
Step 1.5（新規）: TSKファイル存在確認
──────────────────────────────────────
1. 現在ブランチ名からIssue番号を推定（feature/12-xxx → #12）
2. docs/output/tasks/TSK-[分類]-NNN-*.md の存在チェック
3. 存在しなければ:
   ⚠️ TSK ファイルが見つかりません。
   → docs/output/tasks/ から候補を提示 or テンプレートで新規作成を提案
4. 存在する場合: 「実装状況」が更新されているか確認を促す
```

これで **AIエージェント経由でも手動実装でも、PRを出す前にTSKファイルが存在することを保証**できます。手動実装時は「PRを出す」という行為が自然なチェックポイントになります。

---

**みさき：** あー！なるほど〜。つまり「PRを出すときにfinish-taskを呼ぶ」を守れば、TASK更新忘れを機械的にブロックできるってこと？

それだったらちゃんとやれる気がします〜！手動実装してもPRは絶対出すから、そのときに「TSK-DB-001更新した？」って聞いてくれるのは助かる〜。えんまさが言ってた「ハーネス修正」ってそういうことか！てか最初からそれを説明してくれれば良かったじゃないですか〜笑

## Turn 5｜方針統合：設計原則と具体アクション

**岡部：** 今回の議論から、COCOSiL V2開発体制の核心的な設計原則を3つに集約します：

**【3つの設計原則】**

> **① Navigation Beats Documentation**  
> docs自体を増やすのでなく、「目的→場所」のマッピングを充実させる。ONBOARDING.mdは200行死守、TASK-INDEX.mdが機能別ナビを担う。

> **② Issue-Anchored Tasks**  
> TSKファイルはGitHub Issueの「実装手順書」として必ず1対1対応する。タスクの存在はIssueで管理し、実装の詳細はTSK-[分類]-NNNで管理する。

> **③ Friction at PR, Not at Code**  
> ドキュメント更新の強制は「コード変更時」ではなく「PR作成時」に集約する。finish-taskとPRテンプレートを更新ポイントとし、開発中の摩擦をゼロにする。

---

**藤本：** 原則を受けて、**明日から動ける具体アクション**を提示します：

**今すぐやること（現在のタスク：ヒラメのDB整理 + まあみのF2 UI）：**

1. **GitHub Issueを2件立てる**
   - `#N: DB構造整理 - profiles/chat_sessions/user_personality_weights 最終化 [ヒラメ]`
   - `#N+1: F2 フロントUI改良＆UIコンポーネント定義 [まあみ]`

2. **TSKファイルを2件作成**
   - `docs/output/tasks/TSK-DB-001-db-schema-cleanup.md` （参照先: 要件定義書 §4.1〜4.2 + impl-plan-near-soft-feedback-loop.md）
   - `docs/output/tasks/TSK-UI-001-f2-ui-component-design.md` （参照先: 要件定義書 §4.2）

3. **TASK-INDEX.mdを作成** （`docs/TASK-INDEX.md`）
   - F1〜F10 × フェーズ（設計中/実装中/完了）× GitHub Issue × TSKファイル のマトリクス

4. **start-taskコマンドにStep 2.6を追加**（Issue紐づけ確認）

5. **finish-taskコマンドにStep 1.5を追加**（TSKファイル存在チェック）

6. **PRテンプレートにTASK更新チェックリストを追記 + F1-F10正規化**

---

**みさき：** なんか最初は「ONBOARDINGを充実させて」って言われたとき、「また文書増えるのか〜」って思ってたんですけど、結局「読まれる地図を作る」ということですよね？

TASK-INDEXが1枚あって、そこから全部たどれるのはめっちゃ現実的だと思います〜！F2のUIコンポーネント設計のIssueも早く立ててほしい〜。あとPRテンプレートに「TASK更新した？」が入るの、正直めっちゃ助かります。忘れるもん絶対笑

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| **ONBOARDINGの役割** | 「今日動き出すための地図」に絞る（200行死守）。機能別参照は TASK-INDEX.md に移管 |
| **TASK-INDEX.md** | `docs/TASK-INDEX.md` を新設。機能ID × フェーズ × GitHub Issue # × TSKファイル のマトリクス |
| **TASKファイルID形式** | `TSK-[分類]-[番号3桁]`（分類: DB/UI/API/PROMPT/DOCS/CHORE）。GitHub Issueと1対1対応 |
| **ファイルパス** | `docs/output/tasks/TSK-[分類]-[番号3桁]-[slug].md` |
| **start-taskの改修** | Step 2.6 を追加：Issue番号の確認・TSKファイル存在確認・新規作成ガイド |
| **finish-taskの改修** | Step 1.5 を追加：TSKファイル存在チェック + 実装状況更新の確認 |
| **PRテンプレートの改修** | TASK更新確認セクション追記 + 担当機能をF1-F10に正規化 |
| **手動実装時のハーネス** | 「PRを出す」行為をチェックポイントとして活用。コード変更時ではなくPR時に更新を求める |
| **現在のタスク着手順序** | ① GitHub Issue 2件作成 → ② TSKファイル 2件作成 → ③ TASK-INDEX.md作成 → ④ スキル修正 |
