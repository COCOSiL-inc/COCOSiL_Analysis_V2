# 議論ログ：F1要件グリルの取捨選択と進行計画

## 登場人物

- 🧑‍💼 **桐生**（要件定義アーキテクト）：BtoC/BtoBプロダクトの要件定義歴12年。EARS記法と「要件の粒度設計」が専門。over-spec / under-spec を嗅ぎ分ける。
- 👩‍💻 **七海**（オンボーディングUXデザイナー）：SaaS・モバイルアプリのサインアップフロー最適化歴9年。完走率・摩擦削減・フォーム設計が専門。
- 💅 **みさき**（ギャル）：IT素人だけど感性が鋭い。実際のCOCOSiLユーザー（25〜35歳、登録フォームを自分の指で通る人）。

## 議論の前提（確定済み）

- スコープ：F1.1（Clerk認証）・F1.2（生年月日入力）・F1.4（プロダクト哲学提示）の3サブ要件。F1.3ウェルカム対話はプロンプト本文が議論ログで確定済みのため対象外。
- 対象層：C（Capability）→ F（Feature）→ E（EARS受け入れ基準）。V/O/E上位は goal-grill で確定済み。
- 書き出し先：新規 `docs/output/F1/F1_onboarding_features.md`。§4.1 と goal-grill は参照素材。
- 矛盾解消：生年月日はスキップ不可・必須に確定（§4.1「SNS認証後スキップ→未設定保持」例外を廃止）。

---

## Turn 1｜現状認識 — F1グリルで本当に詰めるべきものは何か

桐生：F1の素材は珍しく豊富。goal-grill が V/O/E を確定、議論ログが F1.3 プロンプト本文を確定、§4.1 が業務フロー5ステップを記述。要件定義の現場で一番多い失敗は「グリルの問いを盛りすぎて既に決まっていることを再質問する」こと。診断：**F1グリルは"発明"ではなく"製本"のタスク**——3文書に散った答えを1冊に綴じる作業が主体。

七海：半分賛成、半分反論。製本"だけ"だと危険。§4.1はハッピーパス5ステップにすぎない。完走率を殺すのは常に**分岐の質**。生年月日を認証の前に聞くか後に聞くかだけで、担当した家計簿アプリでは完走率が15ポイント動いた。§4.1にその分岐は書いてない。grillの本当の獲物は**製本の隙間に挟まった「誰も決めていない意思決定点」**。

みさき：え待って、「製本」って…つまり、もう答えはだいたいあるけど3個の書類にバラバラだから1冊にまとめる、ってこと？ で、まとめてる途中で「あれ、これ誰も決めてなくない？」がポロッと出てくる、と。

桐生：まさにそれ。診断を更新：**F1グリルの本質 = 「散在する答えの製本」＋「製本の隙間に落ちた未決定の発見」**。前者を高速に流し、後者を執拗に掘る。この2モード切り替えがグリル品質を決める。

---

## Turn 2｜選択肢の発散 — グリル問いの評価軸を立てる

七海：取捨選択には軸が要る。**軸X：決定の自由度（既出 ⇔ 未決）／軸Y：ユーザー摩擦への影響度（低 ⇔ 高）** の4象限。

| 象限 | 扱い |
|---|---|
| 既出 × 低影響 | 省く（再質問しない） |
| 既出 × 高影響 | 確認のみ（1問にまとめて再掲） |
| 未決 × 低影響 | デフォルト採用 [A]ラベルで軽く流す |
| 未決 × 高影響 | 執拗にグリル（1問ずつ） |

桐生：論点候補を配置：

- 完走率の数値 → 既出×高 → 確認のみ
- F1.3プロンプト本文 → 既出（対象外確定）→ 触れない
- 生年月日スキップ不可 → 既出（確定）→ 触れない
- 生年月日の入力UI方式 → 未決×高 → 執拗グリル
- 認証画面の統合方式（Clerk Hosted / Embedded / Custom）→ 未決×高 → 執拗グリル
- F1全体の画面遷移シーケンス → 未決×高 → 執拗グリル
- SNS認証プロバイダの選定 → 未決×中 → 軽グリル
- F1.4哲学提示の表示制御（初回判定）→ 未決×中 → グリル
- F1.4の哲学コピー本文 → 未決だが別タスク（language-design スキル）→ grill対象外、枠だけ確定
- 30秒の計測起点 → 未決×低 → E層で1問

みさき：私が登録で一番イラッとするやつ——生年月日を「1998年」までプルダウンで延々スクロールさせられるやつ。あれマジ無理〜。生年月日UIを執拗にグリルするの超賛成。

桐生：EARS記法の出番もここで線引き。**E層は全Featureに書くのではなく境界値・例外があるFeatureだけ**。生年月日（未来日付・1900年以前・2/29）はEARS必須。哲学提示は「表示するだけ」でEARS不要、Ubiquitous記法1行で足りる。

## Turn 3｜要件・範囲の確定 — みさきの実需を優先度に落とす

みさき：実ユーザーとして、登録で「もう閉じよ」ってなる瞬間が4つ。①プルダウン地獄 ②「なんで生年月日いるの？」の不信 ③SNSでログインしたのにまた入力させられる ④認証画面で急にデザイン変わって「別サイトに飛ばされた？」ってなるやつ。

七海：4つ、きれいにパターン化できる。F1の摩擦は3類型に集約——**①入力コスト（操作の手数）②不信コスト（なぜ要るのか不明）③文脈断絶コスト（見た目・流れが切れる）**。grillの問いはこの3類型を各Featureで潰せるよう設計すべき。

桐生：優先度マーカーで grill 対象を確定：

🔴 必須グリル（未決×高 → 1問ずつ執拗に）
- 生年月日の入力UI方式（3連select / ネイティブdate / 単一テキスト）
- 認証画面の統合方式（Clerk Hosted Page / Embedded Component）
- F1全体の画面遷移シーケンス（LP → 認証 → 生年月日 の順序、生年月日は認証前/後）
- プライバシー説明文の提示方法（インライン / モーダル / ツールチップ）

🟡 推奨グリル（未決×中 → 推奨回答添えて流し気味に）
- SNS認証プロバイダの選定（Google / Apple / メールのみ）
- F1.4 哲学提示の表示制御（初回のみ判定 = Cookie か DB フラグか）
- 再訪時の挙動（哲学提示スキップ・生年月日入力済みスキップ）
- 30秒の計測起点（lp_viewed 起点 / signup_completed 起点）

🟢 省略（[A]デフォルト採用 or 別タスク）
- パスワードポリシー → Clerkデフォルトに委譲
- 認証失敗のエラーコピー本文 → language-design準拠で§4.1既出
- F1.4 哲学コピーの本文 → language-design スキルの別タスク

## Turn 4｜層ごとの問い数とEARS範囲 — リスクを抑える

桐生：問い数には法則。**グリルの問い数は逆三角形——C層は少なく（3〜4問）、F層が本体（8〜10問）、E層は境界のあるFeatureだけ（5〜7問）**。Capabilityを刻みすぎるとFeatureと重複し同じことを2回聞く。

七海：リスク。F1グリルで一番怖いのは**S層（実装詳細）まで降りすぎること**。S層は対象外確定だが、F層グリル中につい「date pickerはどのライブラリ？」まで聞いてしまう。それはやらない。F層は「3連selectか単一入力か」「年範囲」「デフォルト値の有無」まで。ライブラリ選定は TSK-UI-002 に委ねる。

桐生：EARS対象を明示。EARSを書くのは**生年月日バリデーション**（WHEN 未来日付 / IF 1900年以前 / IF 2/29 の3本）と**認証失敗**（IF Clerk認証エラー THEN … の1本）。哲学提示・SNS選定はEARS不要、Ubiquitous 1行で足りる。

みさき：ん？ 普通の人にEARSって難しくないですか？ 急に専門用語出てきた〜。

桐生：EARSは怖くない。「WHEN〈いつ〉/ IF〈条件〉, THEN システムは〈何をする〉」の穴埋めだけ。「生年月日がおかしかったらエラー」を「WHEN 入力日付が今日より未来, システムは送信前にインラインエラーを表示する」と書く——曖昧さを消す道具。

七海：法則が見える。**Grill Depth = Decision Density（グリルの深さは意思決定の密度に比例する）**。生年月日は「UI方式 × タイミング × バリデーション × プライバシー提示」と意思決定が密集。哲学提示は「表示する／初回だけ」とほぼ決まっている。問い数配分はこの密度に従わせる。

## Turn 5｜方針統合 — グリル進行計画の原則化と最初の一手

### F1要件グリルの設計3原則

**① Bind, Don't Reinvent. — 製本せよ、再発明するな。**
既出の答え（goal-grillのV/O/E、議論ログのF1.3、§4.1の業務フロー）は [A]/[I] ラベルで引用し再質問しない。grillの時間を未決定に集中させる。

**② Grill Where Decisions Are Dense. — 意思決定が密なところを掘れ。**
4象限の「未決 × 高影響」（生年月日UI・認証統合方式・画面遷移）に問いを集中。低密度領域は推奨回答1問で流す。

**③ Stop at Feature, Leave Spec to Tasks. — Feature層で止め、Specはタスクに委ねよ。**
ライブラリ選定・コンポーネント構造などS層詳細には降りない。TSK-UI-002 / TSK-API-001 に委ねる。

### 進行計画（全18問・C→F→E）

| 層 | 問い数 | 内訳 |
|---|---|---|
| C（Capability） | 3問 | C-1 F1の能力分解（3つ）／C-2 画面遷移シーケンス／C-3 Capability間の前後関係・F1.3への橋渡し |
| F（Feature） | 9問 | F1.1認証=3問（認証方式／SNSプロバイダ／認証画面統合）／F1.2生年月日=4問（入力UI方式／入力タイミング／プライバシー提示／年範囲・デフォルト）／F1.4哲学=2問（表示位置と200字枠／初回判定方式） |
| E（EARS基準） | 6問 | E-1〜3 生年月日バリデーション（未来日付／1900年以前／2/29）／E-4 認証失敗／E-5 30秒計測起点／E-6 F1完走の正常系受け入れ基準 |

最初の一手 — **C-1**：「F1が満たすべきCapabilityを3つに分解すると？ 推奨：①30秒以内に認証を完了させる ②4体系自動計算の素地（生年月日）を確実に取得する ③哲学提示で初手離脱を防ぎF1.3共感対話へ動機づける」

みさき：最初は「F1ってもう要件あるんじゃないの、なんでまたグリル？」って思ったけど——3つの書類にバラけてて、その隙間に「誰も決めてない生年月日のUI」がポロッと落ちてた、と。それを拾うグリルなら超納得〜。18問なら多すぎないし、私がイラッとするプルダウン地獄もちゃんと掘ってもらえる。付き合えるやつです。
