## Turn 1｜現状診断：COCOSiLの仕様変更伝播の何が壊れているか

**前田：** 議論ログ「ドキュメントアップデート方針」を読んで、まず構造的問題を言います。COCOSiLには**ドキュメントの「正（single source of truth）」が4か所に分散**しています。

具体的に言うと：
- `docs/input/concepts/COCOSiL設計中枢.md`（哲学・原則）
- `AGENTS.md`（エージェント行動規範）
- `docs/output/requirements/`（要件定義書）
- `.claude/skills/cocosil-domain/SKILL.md`（AI参照マスタースキル）

この4つは**互いを参照しているが、互いを更新しない**。たとえば設計中枢で「Constitution as Code」の方針が決まり `lib/constitution/` が新設されても、要件定義書にも `cocosil-domain/SKILL.md` にも反映されない。これはドキュメント工学でいう「ドリフトの構造的温床」です。

伝播すべき変更の種類を分類します：

| 変更種別 | 発生源 | 波及先（必須） | 波及先（推奨） |
|---|---|---|---|
| 哲学・原則変更 | `docs/input/concepts/` | `AGENTS.md §0`、`cocosil-domain/SKILL.md` | `.cursor/rules/` |
| UXシーケンス変更 | `AGENTS.md` | `lib/constitution/ux-sequence.ts`、`cocosil-domain/SKILL.md` | 要件定義書 |
| 禁止語変更 | `lib/constitution/banned-words.ts` | `language-design/SKILL.md`、`AGENTS.md §6` | `docs/input/concepts/language-design-v1.md` |
| API・DB設計変更 | `app/api/`、`supabase/`、`lib/types/` | 要件定義書、`docs/harness/` | `AGENTS.md` |
| 機能スコープ変更 | 要件定義書 or Issue/PR | `cocosil-domain/SKILL.md`、`docs/harness/DESIGN_FLOW.md` | `AGENTS.md`、`docs/README.md` |

**診断：COCOSiLの問題は「何を更新するか知らない」ではなく、「変更が起きたとき誰がどのファイルを更新するかルールがない」こと。これが根本原因。**

**橋本：** 前田さんの診断は正確です。私が付け加えるなら、COCOSiLは特殊な構造を持っています。一般のプロダクトと違い、**ドキュメントがAIエージェントの「行動仕様書」を兼ねている**。

つまりドキュメントのドリフトは、コードのバグより先にAIエージェントの判断ミスとして現れます。Supabaseのテーブル変更を `database.ts` に反映し忘れた場合は型エラーが出ます。でも `cocosil-domain/SKILL.md` が古いままだと、AIが古い仕様で動き続けて**誰も気づかない**。

`docs/discussions/議論ログ_設計整合性ハーネス再設計.md` で「Constitution as Codeが設計された」という事実があります。しかし今日時点でも `lib/constitution/` は空のまま（`.gitkeep`のみ）で、`AGENTS.md` には「コードが正」と書いてあるのにコードが存在しない。この**「宣言はされたが実装されていない正」**が、最もドリフトを起こしやすい状態です。

**みさき：** ん？ちょっと待って。結局「仕様が変わったら、どのファイルも全部手で直さなきゃいけない」ってこと？それってめっちゃ大変じゃないですか？開発者3人しかいないのに、仕様変更のたびに5〜10ファイル直すの？それ誰かが絶対忘れますよね〜？

**前田：** みさきが核心を突いてくれました。その「誰かが絶対忘れる」問題を解決するのが今日設計するスキルの目的です。仕様変更を起点に「どのドキュメントが影響を受けるか」をAIが自動列挙し、各ファイルの更新箇所まで特定して提案する。これが `spec-sync` スキルの存在意義です。

## Turn 2｜スキルの設計軸：何を自動化し、何を人間に残すか

**橋本：** 設計判断の核心は「AIが何をやって、人間が何をやるか」の境界線です。COCOSiLのLayer構造を踏まえて評価軸を立てます。

**評価軸①：変更の「意味的影響」vs「機械的影響」**

| 影響種別 | AI自動化可能 | 人間判断必要 |
|---|---|---|
| ファイルパスの列挙 | ✅ | — |
| 変更すべきセクションの特定 | ✅（高精度） | — |
| 哲学・UXシーケンスへの影響判断 | ⚠️ 補助のみ | ✅ えんまさ |
| 禁止語の追加・削除 | ⚠️ 提案のみ | ✅ えんまさ |
| 技術スタックの変更反映 | ✅ | ヒラメ確認 |
| 要件定義書の差分更新 | ✅（下書き） | えんまさ/ヒラメ確認 |

**評価軸②：COCOSiLのLayer構造との整合性**

```
Layer 1（保護）: supabase/migrations, .env → スキルは触れない・警告のみ
Layer 2（えんまさ承認必須）: lib/prompts, lib/data, lib/diagnostics → スキルは影響指摘のみ
Layer 3（AI委任可能）: app/**, components/**, app/api/** → スキルは更新案まで提示
```

**評価軸③：伝播の向き（双方向 vs 一方向）**

一般のdocs-syncスキルは「コード変更 → ドキュメント更新」の一方向。COCOSiLは**双方向の伝播**が必要：
- `docs/input/concepts/` の変更 → `AGENTS.md`・スキル・`lib/constitution/` への反映
- `lib/constitution/` の変更 → ドキュメント群への反映
- 要件定義書の変更 → `cocosil-domain/SKILL.md` への反映

**前田：** 橋本さんの軸が揃ったので、設計方針を言います。

スキルの核心動詞は「**列挙する → 差分を示す → 更新案を生成する → 承認者を明示する**」の4ステップです。汎用の `docs-sync` スキルは「影響ドキュメントを列挙して更新する」だけですが、COCOSiL版は **Layer構造・承認フロー・双方向伝播**を内包した重厚版になります。

```
変更種別の判定
    ↓
影響ドキュメントの列挙（必須・推奨を分けて）
    ↓
各ファイルの変更すべきセクション特定
    ↓
Layer判定（誰が承認するか）
    ↓
更新案の生成（Layer 3は本文まで、Layer 2は箇条書きで提案）
    ↓
PRチェックリスト生成
```

**みさき：** ちょっとちょっと！「Layer 1は触れない」とか「Layer 2はえんまさ承認」とか、それって普通のdocs-syncスキルにはないやつですよね？COCOSiL専用ってことですか？なんかカスタマイズが多くて、スキルを読む人は分かるんですか？

**橋本：** みさきの懸念は正しい。スキルを使う人（AIエージェント）が混乱しないよう、**変更種別 → 影響ファイル → 承認者**の3つをテーブルで引けるリファレンスを本文に埋め込む設計にします。AIがメモリに頼らず、スキル本文を参照すれば迷わない構造です。

## Turn 3｜COCOSiL特有の「伝播パターン」の洗い出し

**前田：** `docs/discussions/` の過去の議論ログを横断すると、COCOSiLで実際に起きた伝播ミスのパターンが見えてきます。

**実際に起きた（起きかけた）伝播漏れ3ケース：**

1. **Constitution as Codeの宣言漏れ**（`harness-redesign-proposal-2026-05-05.md`に記載 → `AGENTS.md`に入ったが `lib/constitution/` は未実装のまま）
2. **Gamma API依存の記載漏れ**（`lean-canvas-v3.md`にリスク記載 → `AGENTS.md`未反映）
3. **禁止語ルールの「なぜ」が欠落**（`language-design/SKILL.md`に禁止リストはあるが理由なし → エージェントが守らない）

この3ケースを踏まえると、COCOSiLの伝播パターンは**5種類に分類**できます：

| パターン | トリガー変更 | 必須波及先 | 推奨波及先 |
|---|---|---|---|
| **P1: 哲学更新** | `docs/input/concepts/COCOSiL設計中枢.md` | `AGENTS.md §0`、`cocosil-domain/SKILL.md` | `.cursor/rules/00-core-philosophy.mdc` |
| **P2: 禁止語変更** | `lib/constitution/banned-words.ts` | `language-design/SKILL.md`、`AGENTS.md §6` | `docs/input/concepts/language-design-v1.md` |
| **P3: UXシーケンス変更** | `lib/constitution/ux-sequence.ts` | `AGENTS.md §0`、`cocosil-domain/SKILL.md` | 要件定義書 |
| **P4: 機能スコープ変更** | 要件定義書 or Issue/PR | `cocosil-domain/SKILL.md`、`docs/harness/DESIGN_FLOW.md` | `AGENTS.md`、`docs/README.md` |
| **P5: API/DB設計変更** | `app/api/`、`supabase/`、`lib/types/` | 要件定義書（差分）、`docs/harness/HARNESS_HEALTH.md` | `AGENTS.md §3` |

**橋本：** 前田さんの5パターンに加えて、**「誰が変更を起こすか」で伝播先が変わる**重要なポイントがあります。

COCOSiLには3人の担当者（えんまさ・ヒラメ・まあみ）がおり、変更の発生源によって責任者が異なります：

```
えんまさ → P1/P2/P3（哲学・言語・UX） → Gate 2 required
ヒラメ   → P4/P5（API/DB/機能スコープ） → Gate 1 required
まあみ   → P4（フロントUI変更によるスコープ変更） → Gate 2 optional
```

このパターン × 担当者マトリクスをスキルに内包すれば、AIは「誰に承認を求めるか」まで自動判定できます。

**みさき：** え、5パターンって多いですか？実際に使うとき、「これはP1？P3？」って迷いそうじゃないですか〜？判定が難しいやつってないんですか？

**前田：** 鋭い質問です。実際に判定が難しいのは**P1とP3の境界**（哲学変更なのかUXシーケンス変更なのか）と**P4とP5の境界**（機能スコープ変更なのかAPI変更なのか）です。

解決策：スキルにトリガーワードリストを持たせる。「変更したファイルパス or 変更の一言説明」から自動判定するマッピングテーブルです。たとえば `lib/constitution/ux-sequence.ts` が変更されたら問答無用でP3、`app/api/` 配下が変更されたらP5と判定します。

## Turn 4｜スキルの構造設計：他スキルとの連携と差別化

**橋本：** 既存スキル群との役割分担を明確にします。混乱が起きるのは境界が曖昧なときです。

**既存スキルとの境界マッピング：**

| スキル | 主用途 | spec-syncとの違い |
|---|---|---|
| `docs-sync`（既存） | コード変更後のdocs更新 | COCOSiL固有のLayer/承認フローがない |
| `cocosil-domain` | AIの実装判断基準を提供 | 「変更の伝播」ではなく「参照」スキル |
| `requirements-doc-creator` | 新規要件定義書の作成 | 変更の**追跡・伝播**ではなく**新規作成** |
| `goal-grill` | タスク着手前の目標整理 | 仕様変更の伝播とは無関係 |
| `security-sensitive-change-review` | 認証・DB変更のセキュリティレビュー | P5変更の**前置チェック**として連携 |

**連携フロー（スキルチェーン）の設計原則：**

```
P5（API/DB変更）の場合:
  security-sensitive-change-review → spec-sync
  （セキュリティチェック後に伝播先を列挙）

P1/P2/P3（哲学/言語/UX変更）の場合:
  spec-sync → expert-misaki-discussion（変更の意味的妥当性を議論）
  （伝播先を列挙してから、変更の意図を議論して成文化）
```

**前田：** 橋本さんの連携設計に技術的な詳細を加えます。

スキルの内部処理で最も重要なのは**「変更前の状態を把握する」ステップ**です。汎用 `docs-sync` は変更されたファイルを渡せば動きますが、COCOSiL版は「変更前の仕様がどこに書かれていたか」を逆引きする必要があります。

具体的には：
1. **変更内容の入力**（自然言語 or ファイルパス）
2. **パターン判定**（P1〜P5の自動分類）
3. **影響ドキュメントの逆引き**（パターン別テーブルから列挙）
4. **各ドキュメントの現在の記述を読み込み**（スキル実行時にReadツール使用）
5. **差分生成**（変更前後の記述を比較してdiff形式で提案）
6. **Layer判定と承認者明示**
7. **PRチェックリスト生成**

このステップ4「現在の記述を読み込む」が汎用版との最大の差別化です。AIが実際にファイルを読んで、「今のAGENTS.mdのSection 0は何と書かれているか」を確認してから更新案を生成する。

**みさき：** あ〜、つまり「テンプレートを埋めるだけ」じゃなくて、「今のファイルを読んで、ここを直して」って教えてくれるんですね？それ超助かる！開発者でも「このファイルのどこを直せばいいか」って探すの大変じゃないですか。エラーみたいに「○行目直して」って言ってくれる感じ？

**前田：** まさにその通りです。みさきの「エラーみたいに○行目直して」という表現は核心を突いています。スキルの出力形式を「影響ファイル × 変更すべきセクション × 更新案（diff形式）」で統一することで、AIがコピー&ペーストレベルで更新できるようになります。

ただし一つの制約：**Layer 2ファイルは「更新案を生成するが実行しない」**設計にします。`lib/prompts/` や `lib/constitution/` への変更はえんまさの承認なしに実行できないので、スキルは「ここを直してください（えんまさ確認後）」という形で止まる。

## Turn 5｜設計原則の統合と具体的スキル仕様

**前田：** 5ターンの議論を踏まえ、`spec-sync` スキルの設計原則を集約します。

---

**【COCOSiL spec-sync スキルの3原則】**

**① Propagation over Perfection（伝播の網羅性を精度より優先する）**
漏れなく影響先を列挙し、「更新不要」の判断もAIが明示する。網羅性が高ければ、更新精度が低くても人間が補正できる。逆は成立しない。

**② Layer-Aware Execution（Layer構造に従って実行範囲を変える）**
同じ「仕様変更」でも、Layer 1は警告のみ・Layer 2は提案のみ・Layer 3は自動更新まで実行する。スキルは実行前にLayerを判定し、人間の介入が必要な箇所を明示する。

**③ Reverse Traceability（変更を起点に、影響の上流まで遡る）**
一般の `docs-sync` は「コード → ドキュメント」の一方向。COCOSiL版は「コンセプト → AGENTS.md → コード → ドキュメント」の双方向追跡を行う。ConstitutionとAGENTS.mdが矛盾している場合、コードを正として警告する。

---

**橋本：** 明日から使えるスキル仕様の具体アクション：

1. **スキルファイル構成**
   - `SKILL.md`（本体：手順・パターンテーブル・Layer判定ロジック）
   - `references/propagation-matrix.md`（変更種別 × 影響ファイル × 承認者の完全テーブル）
   - `evals/evals.json`（3ケースの評価テスト：P1/P3/P5をカバー）

2. **スキルのトリガー条件**
   - 「仕様変更をドキュメントに反映して」
   - 「〇〇の変更をどこに書けばいいか」
   - `lib/constitution/` 配下のファイルを変更したとき
   - 要件定義書・`docs/input/concepts/` を変更したとき
   - PRレビュー前に「ドキュメント同期チェック」を依頼されたとき

3. **出力フォーマット**（毎回この構造で返す）
   ```
   ## 変更種別：P[N] - [パターン名]
   
   ### 必須更新（自動実行可能）
   - [ ] ファイルパス：変更すべきセクション → 更新案
   
   ### 必須更新（承認後に実行）
   - [ ] ファイルパス：変更すべきセクション → 更新提案（承認者：えんまさ/ヒラメ）
   
   ### 推奨更新（任意）
   - [ ] ファイルパス：変更すべきセクション → 更新提案
   
   ### 不変確認（変更禁止）
   - [ ] 変更してはいけないファイル or セクションと理由
   
   ### PRチェックリスト追記
   - Gate 1/2 の追加チェック項目
   ```

**みさき：** なんか最初「ドキュメントが多すぎてどこ直すか分からない問題」があって、それが「パターン別に影響先を自動列挙」「Layerで実行範囲を変える」「双方向に追跡する」ってまとまったんですね〜！めっちゃ体系化されてる！私が一番好きなのは「更新不要」もちゃんと言ってくれるとこ。「このファイルは今回関係ないです」って言ってくれる方が逆に安心するじゃないですか〜。

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| スキル名 | `spec-sync`（仕様変更→ドキュメント伝播スキル） |
| 伝播パターン | P1〜P5の5パターン（哲学/禁止語/UX/機能スコープ/API-DB） |
| 実行範囲 | Layer 1：警告のみ / Layer 2：提案のみ / Layer 3：自動更新 |
| 双方向追跡 | コンセプト→コード→ドキュメントの双方向伝播をサポート |
| 出力形式 | 変更種別判定 → 必須/推奨/不変の3分類 → PRチェックリスト |
| ファイル構成 | `SKILL.md` + `references/propagation-matrix.md` + `evals/evals.json` |
| 他スキル連携 | P5はsecurity-sensitive-change-review前置 / P1/P3は議論ログ記録推奨 |
| 設計原則 | Propagation over Perfection / Layer-Aware Execution / Reverse Traceability |
