## Turn 1｜現状認識・根本課題の診断

**橋本：** 今のプロジェクト構造を見ると、`.claude/` ディレクトリにスキル・フック・コマンド・設定がフル整備されています。問題は、Cursorがこれらを一切認識しない点です。

具体的な欠落を列挙すると：

| Claude Codeの資産 | Cursorでの状態 | リスク |
|---|---|---|
| `.claude/skills/*.md` | **認識されない** | スキル呼び出しが無効 |
| `.claude/hooks/prevent-destructive-command.js` | **動かない** | 破壊的操作の保護がゼロ |
| `.claude/commands/*.md`（/start-task等） | **スラッシュコマンド未定義** | ワークフローが機能しない |
| `.claude/settings.json` | **Cursor設定とは別物** | MCP設定・権限が引き継がれない |
| `AGENTS.md`（ルート） | ✅ **自動で読まれる** | 問題なし |

**重要な診断：** これは「ツール移行」ではなく「デュアルツール体制の確立」。Claude CodeとCursorを共存させつつ、同一の品質基準・ガードレールを両環境で担保する設計が必要。

**伊藤：** `AGENTS.md` がルートにあることで基本ルールは読まれるが、**15KB超のファイルは最後まで確実に読まれる保証がない**。解決策は重要ルールを `.cursor/rules/` の専用 `.mdc` ファイルに切り出し、`alwaysApply: true` を付けること。

**みさき：** え待って、つまり今のままCursorで開発すると、「supabaseのDBを消すなよ！」ってフックが効かないってこと？それめっちゃヤバくないですか？

## Turn 3｜スキル・コマンドの移植方針

**橋本：** `.claude/skills/` のスキルはCursorの `AGENTS.md` スキルセクションで**既に参照可能な状態**。プロジェクトスキルはすでにCursorでも使える。

問題はコマンド側。Cursorにはスラッシュコマンドが存在しないので、「トリガーワードに反応するルール」として `.cursor/rules/03-workflow.mdc` に実装する。

**伊藤：** 例：

```
「着手する」「ブランチを切る」が出たら：
1. 担当ガード確認
2. 5問リトマス試験紙チェック
3. feature/<slug> 形式でブランチ名提案
```

これにより「着手したい」と言えば `/start-task` と同等の動作をする。

**みさき：** あ〜、Cursorで「着手したい」って言えば同じことをやってくれる感じ？それめっちゃ現実的〜！

## Turn 4｜保護レイヤーの代替実装と制約

**伊藤：** Claude Codeの `prevent-destructive-command.js` はOSレベルの強制ブロック。Cursorには同等のメカニズムがない。

**3段階の代替防衛策**：

```
防衛レイヤー1: 01-protected-areas.mdc（NEVER句・alwaysApply）
防衛レイヤー2: AGENTS.md §7 の常時参照
防衛レイヤー3: Git保護（pre-commitフック）← 後日対応
```

**法則：ソフトガードとハードガードの使い分け。**
- ハードガード（フック）: Claude Code環境で継続
- ソフトガード（ルール): Cursor環境。AI遵守率95%以上
- 構造ガード（Git hooks）: 両環境共通の最終防衛線

**橋本：** `NEVER` 句を使うとLLMの遵守率が統計的に上がる（約15%差）。`.mcp.json` はCursorでも共用可能で追加作業なし。

**みさき：** 「95%以上は従う」ってちょっと怖くないですか？残りの5%のときにDBが消えそうで…。

## Turn 5｜設計原則の統合と具体アクション

**橋本：**

> **① Rules-as-Source-of-Truth**  
> `AGENTS.md` が真の設計源。`.cursor/rules/` はエッセンスを凝縮したプロキシ。

> **② Layer-Aware Rule Injection**  
> Layer 1（保護）は `alwaysApply: true` で毎ターン注入。注入量を絞ることで確実に読まれる。

> **③ Dual-Tool Parity, Not Symmetry**  
> Claude CodeとCursorは同じルールを守るが、実装手段は異なる。守られる確率を最大化する組み合わせを環境ごとに選ぶ。

**伊藤：** 今日から動ける5ステップ：

1. `.cursor/rules/` 4ファイル作成（00〜03）
2. `AGENTS.md` にCursor向け補記を追加
3. `.mcp.json` の確認（追加作業なし）
4. スキルは現状維持（既に使える）
5. （後日）Git pre-commitフックで構造ガードを追加

**みさき：** なんか最初「全部移植しなきゃ？」って思ってたけど、AGENTS.mdが既に読まれてて、スキルも使えて、あとは `.cursor/rules/` を4本作るだけってことですよね？それめっちゃ現実的〜！
