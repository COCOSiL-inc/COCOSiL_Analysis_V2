# 議論ログ：導入ドキュメント設計

## 登場人物

- 🧑‍💼 橋本 誠（DevExエンジニア）：開発者オンボーディング設計15年。スタートアップの「プロジェクト参加初日体験」最適化が専門。
- 🤖 岸田 藍（AIアーキテクトコンサルタント）：LLMエージェントとのペアプログラミング設計専門。CLAUDE.md / MCP設計コンサルタント。
- 💅 みさき（ギャル）：実際のCOCOSiLユーザー。友達のチームに途中参加した経験あり。

## Turn 1｜「最初に読む」ドキュメントはなぜ読まれないのか

橋本： 正直に言います。オンボーディングドキュメントの失敗例を200件以上見てきて、失敗の9割は同じ原因です。「書き手が『何を知っているか』を起点に書く」こと。

COCOSiLの `AGENTS.md` を初めて見た人は、9セクション・約200行のドキュメントと向き合います。情報量は十分です。でもこれは「知っている人が整理したリファレンス」であって、「知らない人が入口として読む文書」ではない。

抽象的に言うと：**ドキュメントが「在庫管理」になっている**。情報を漏れなく記録することが目的化し、「読者の旅路設計」が抜け落ちている。

---

岸田： このプロジェクトには加重要因があります。読者が「人間」と「AIエージェント」の両方いることです。

人間のヒラメさん・まあみさんは「雰囲気で動き出してから疑問が生まれる」読者です。AIエージェントは逆で、「最初に渡されたコンテキストの外では動けない」。

CLAUDE.md に `pnpm のみ使用` と書いてあっても、「なぜpnpmか」という背景が書かれていないAIは、次のセッションで `npm install` を提案することがある。これは **コンテキストの深さ設計の失敗**です。

抽象化すると：人間のオンボーディングは「安心感の設計」、AIのオンボーディングは「コンテキストの初期充填設計」、で根本的に別問題。

---

みさき： え待って、AIって毎回最初から説明しなきゃいけないんですか？友達に「前も言ったじゃん」って言えない感じ？めっちゃ辛くないですか？

岸田： みさきさんのたとえ、正確です。セッションをまたいだ記憶はない。だからこそ CLAUDE.md が「毎回読み込まれる永続コンテキスト」として機能する設計になっているんです。

橋本： みさきさんの感覚、すごく大事。「毎回説明しなきゃいけない辛さ」がゼロになる設計——それが良い導入ドキュメントです。人間にとっては「読んだら迷わず動ける」、AIにとっては「読んだら間違わず動ける」。ゴールは共通しています。

## Turn 2｜人間向けとAI向けで何が根本的に違うか

岸田： 評価軸を立てましょう。設計の差異は3つの軸で見えます。

| 軸 | 人間（ヒラメ・まあみ） | AIエージェント |
|---|---|---|
| **読み方** | 斜め読み → 気になったら深掘り | 全文を均等にパース、重み付けできない |
| **動機** | 「早く動き出したい」感情がある | 動機ゼロ。与えられたコンテキストで動く |
| **エラー対応** | 失敗 → 調べる → 学ぶ | 失敗 → 同じ失敗を繰り返す（次セッションでリセット） |
| **優先情報** | Why（なぜこのプロジェクトは存在するか） | What+Must-Not（何をすべきか、何をしてはいけないか） |

人間向けには「感情的な入口」が必要で、AI向けには「構造化された制約マップ」が必要。同じドキュメントで両方満たすには、セクションを明示的に分けるか、階層化する必要があります。

---

橋本： 「失敗コスト」の非対称性を加えます。

人間がルールを知らずに `npm install` しても1分で気づいて直せます。でもAIが同じことをすると、そのセッション内で何度も `npm install` を提案し続ける可能性がある。COCOSiLの「`npm install` / `yarn add` は禁止」「`src/` ディレクトリは存在しない」「`process.env` の直接参照禁止」——これらは全部、人間なら「あ、そうか」で済むが、AIには **高コスト失敗**に直結するルールです。

**人間の学習コストはO(n)、AIの失敗コストはO(n×セッション数)**。

---

みさき： 友達のスタートアップに手伝いで入ったとき、「どこに何があるか」が全然分からなくて詰まりました。どれ触っていいか分からないって感じ。

橋本： それがまさに核心。「どこに何があるか」＋「どれを触っていいか」——この2つが初日の全てです。

## Turn 3｜「絶対に入れるべき情報」の優先度を確定する

橋本： 初日に知らないと詰まる情報を4つのカテゴリに分けました。

🔴 **必須（知らないと初日に止まる）**
- pnpm のみ使用（`npm install` 禁止）
- `src/` ディレクトリは存在しない（`@/*` はルートマップ）
- `process.env` 直接参照禁止（`@/lib/env` 経由）
- `git checkout -b` 禁止（`/start-task` 経由）
- Layer 1 の保護ファイルに触ってはいけない（`.env*`、`supabase/migrations/**`）

🟡 **推奨（知っておくと判断が速い）**
- UXシーケンス 共感→安心→分析→行動 の絶対性
- 禁止語（占い、鑑定、運勢）と代替表現
- 担当ガード（Layer 1/2/3 と担当者の対応）
- 主要スキル（`/start-task` `/finish-task` `/cocosil-work`）の用途

🟢 **任意（詳細は AGENTS.md へ）**
- ブランチ命名規則の詳細
- Supabase MCP の設定方法
- 議論ログの保存場所

---

岸田： 🔴の項目の中でも、AIが最もコストの高い失敗をするのは「禁止されているコマンドの実行」と「保護ファイルへの書き込み」です。

`prevent-destructive-command.js` フックは最後の防衛線であって、AIに「最初から知らせる」ことで衝突を避けるのが正しい設計。**フックは「壁」、導入ドキュメントは「地図」**。地図があれば壁に当たらない。

また、AI向けには「**なぜか**」を書くことが重要。「pnpm のみ」と書いても理由がなければAIは例外を許容する判断をする可能性がある。

---

みさき： 全部書いたら長くなって読まれなくなるじゃないですか？

橋本： だから「1画面で読める」を絶対的な制約にする。スクロールが1回発生すると読了率が40%落ちます。詳細は AGENTS.md がある。導入ドキュメントは「AGENTS.md を読む前の30秒」に何を渡すかの設計です。

## Turn 4｜AIエージェント向け設計の深掘り

岸田： AI エージェントに渡すべきコンテキストの構造設計。3層フレームワーク：

```
Layer A：プロダクト哲学（Why）
  └ なぜこのプロジェクトは存在するか
  └ 非交渉のUXシーケンス
  └ PMF成功基準（7日以内再訪率30%）

Layer B：制約マップ（Must-Not）
  └ 触ってはいけないファイル（Layer 1）
  └ 使ってはいけないコマンド・パッケージマネージャ
  └ 禁止語と代替表現

Layer C：操作手順（How-To）
  └ 最初に実行するコマンド
  └ タスク開始・終了のワークフロー
  └ 分からないときはどのスキルを呼ぶか
```

COCOSiLにマッピングすると：
- **Layer A** は `README.md` の冒頭に既に書かれている
- **Layer B** の「Must-Not」が `AGENTS.md` の Section 5・6・7 に分散 → **1箇所に集約が必要**
- **Layer C** は `README.md` のセットアップ + `/start-task` の説明で十分

---

橋本： **禁止事項が散在していてAIが拾えない**パターンが最も多い失敗です。

人間はCtrl+Fで検索できますが、AIは「セッション開始時に読み込んだ内容のどこかにあった」という曖昧な参照しかできない。**禁止事項は1つのセクション「⛔ これだけは守る（AI最重要）」に固めるべき**です。

**人間のドキュメント設計原則は「発見性」、AIのドキュメント設計原則は「局所性」**。

---

みさき： ヒラメさんとまあみさんで「自分が何をすべきか」分からないと動けないですよね？

橋本： **「ヒラメさんが読む版」「まあみさんが読む版」は分けない。ただし冒頭に「あなたへのナビ」を入れる。**

```
→ ヒラメ（バックエンド）: あなたは Layer 3「app/api/**」が主戦場。
  最初にやること: Supabase 型生成（pnpm db:types）
→ まあみ（フロントエンド）: あなたは Layer 3「app/**・components/**」が主戦場。
  最初にやること: pnpm dev で開発サーバー起動
→ AIエージェント: Layer 1 の保護ファイルには絶対に触れない。
  迷ったら /cocosil-work を呼ぶ。
```

## Turn 5｜設計原則の統合と構成への着地

岸田：

**導入ドキュメントの3原則：**

① **Welcome First, Rules Second**  
   最初の3行は感情的な入口（Whyと歓迎メッセージ）。ルールは後。

② **Must-Not Locality**  
   禁止事項は1箇所に集約。AIは散在した禁止事項を確実には拾えない。

③ **Role-Based Fast Track**  
   読者を役割別（ヒラメ／まあみ／AI）にナビする3行を冒頭に入れる。

---

橋本： 具体的な構成：

```
docs/ONBOARDING.md — 推奨構成
────────────────────────────
# COCOSiL V2 へようこそ
> 1行ミッション

## あなたへのナビ（最初にここを読む）
- ヒラメ（バックエンド） → やること1行
- まあみ（フロントエンド） → やること1行
- AIエージェント → やること1行

## ⛔ これだけは守る
（全禁止事項を箇条書きで集約。理由つき）

## プロジェクト地図
（ディレクトリ構造 + 担当ガード早見表）

## 最初の30分でやること
（セットアップ → /start-task → pnpm dev）

## 困ったときの地図
（スキル→どこで何を聞くか一覧）

---
詳細は AGENTS.md へ
```

---

みさき： なんか最初「導入ドキュメントって AGENTS.md の短縮版でしょ？」って思ってたんですけど、全然違いましたね〜！「AIの信頼感爆上がりできるドキュメント」早く見たいです！

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| **ドキュメントの役割** | AGENTS.md の短縮版ではなく「地図の入口」。詳細は AGENTS.md へ委譲 |
| **構成の絶対制約** | 1画面に収まること。スクロール不要 |
| **冒頭** | Welcome First — Why（ミッション）→ 3役割ナビ（ヒラメ/まあみ/AI） |
| **禁止事項の設計** | Must-Not Locality — 1セクション「⛔ これだけは守る」に全集約。理由つき |
| **AI向け設計** | Layer B（Must-Not）+ Layer A（Why）を先に渡す |
| **役割別ナビ** | 「あなたへのナビ」3行で即座に自分事化できる設計 |
| **詳細の扱い** | AGENTS.md へのリンクで委譲 |
