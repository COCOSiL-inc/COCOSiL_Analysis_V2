# 議論ログ：まあみ UIデザインフロー設計（claude_design → プロジェクト移管）

## 登場人物

- 🎨 田辺 千尋（デザインシステムエンジニア）：Figma・AI生成UI→Next.js移管専門。デザインと実装の品質ロスをゼロにすることが使命。
- 👩‍💻 まあみ（フロントエンドエンジニア）：COCOSiLのUI/フロント担当。claude_designを使って実際にUI案を作っている当事者。
- 💅 みさき（ギャル）：実際のCOCOSiLユーザー。「使いやすいか・共感できるか」の感性判定役。

## Turn 1｜外部デザインツール→プロジェクト移管の失敗パターン

田辺： 失敗は3つのパターンに集約されます。

**パターン1「Design Drift（デザイン漂流）」** Tailwind CSS 4 vs v3 の記法差などプロジェクト設定を知らない生成コードが、動くけど微妙にズレたUIになる。

**パターン2「コンテキスト切断」** デザインツール内AIがプロジェクトの `components/` を知らず、移管後に既存コンポーネントとの重複・命名衝突が起きる。

**パターン3「意味のない美しさ」** デザインが綺麗でもCOCOSiLの「共感→安心→分析→行動」シーケンスと合っていなければ失格。デザインツールはそのシーケンスを知らない。

抽象化：**外部デザインツールは「孤立したコンテキスト」で動いている。移管とはコンテキストの橋を渡ること**。

---

まあみ： 3パターン全部身に覚えあり。特にパターン1（Tailwind v4 `divide-y`の挙動差で意図しないボーダーが出た）とパターン3（えんまさに「このフロー逆じゃない？」と指摘された）。

もう一つ自分の失敗：**選定段階でこだわりすぎて移管後に手直しが多発する**問題。claude_designで3案から選ぶとき、選定基準に「実装可能性」が入っていなかった。

---

みさき： アプリのUI見てて「安心できない感じ」のものって信頼できなくなる。COCOSiLって「安心」がウリじゃないですか。

田辺： COCOSiLの安心フェーズはUXシーケンスの2番目。デザインで安心感が壊れると7日以内再訪率30%のPMF仮説が根本から揺らぐ。**デザイン選定に「COCOSiLらしいか」の基準がなければどんなに綺麗なUIでも失格**。

## Turn 2｜まあみワークフローの評価軸設計

田辺： 移管フローを評価する3つの軸：

| 軸 | 問い | 失敗コスト |
|---|---|---|
| **Fidelity** | デザイン意図がコードに正確に再現されているか | 低（見た目のズレ・再修正）|
| **Compatibility** | プロジェクトの技術制約に適合しているか | 中（Tailwindバージョン差・命名競合）|
| **Coherence** | COCOSiLの哲学・UXシーケンス・禁止語に合っているか | 高（PMF仮説崩壊・信頼喪失）|

移管フローの最優先ガードはCoherence。3つのゲートが必要：

```
[Gate 1: 選定前] — Coherenceチェック
[Gate 2: 移管前] — Compatibilityチェック
[Gate 3: PR前]   — Fidelityチェック
```

---

まあみ： 私が今やっているのは「Gate 2だけ」だった。Gate 1が完全に抜けていた。毎回えんまさに聞くのは現実的でないので、AIパートナーが参照できるCoherenceチェックリストが必要。

田辺： ハーネスにAIが参照できるチェックリストを組み込む。**AIパートナーにGate 1のチェックリストを持たせる**のが正しい設計。

## Turn 3｜COCOSiLの文脈で「移管時に何が壊れるか」

まあみ（具体リスト）：

**技術的に壊れるもの（自分で気づける）：**
- Tailwind CSS 4 vs v3 の記法差（`ring-*`・`divide-*`・`bg-opacity`系）
- `src/` を前提としたimportパス（claude_designが `src/components/` を生成する）
- `import { z } from 'zod'`（プロジェクトは `'zod/v4'` サブパス必須）
- Clerk UIコンポーネントとの競合
- `process.env.XXX` の直接参照

**意味的に壊れるもの（気づきにくい）：**
- UXシーケンス順序の違反
- 禁止語の混入（占い・鑑定・運勢）
- 安心感を壊すビジュアルトーン
- PMF指標に関わるCTA文言のミスマッチ

---

田辺： **技術的壊れは自動検証できる（tsc + lint + pnpm build）、意味的壊れはドキュメント化されたチェックリストが必要**。これが移管ハーネスの2層設計の根拠。

移管単位の設計も重要：「画面まるごと移管」は危険。**1コンポーネント単位で移管**を原則にすると検証スコープが小さくなって品質が上がる。

---

みさき： 意味的に壊れてる方がユーザーとして全然ダメ。技術バグは「また直すんだな」だけど、不安になるUIは「このアプリ信頼できない」でそれっきりになる。意味的壊れは致命傷。

## Turn 4｜ハーネス・ガードレールの具体設計

田辺：

**まあみのワークフロー4フェーズ：**
```
Phase 1: 生成（claude_designでUI案を複数生成）
Phase 2: 選定（Gate 1: Coherenceチェックを経て1案を選ぶ）
Phase 3: 移管（Gate 2: Compatibilityチェックを経てプロジェクトに取り込む）
Phase 4: レビュー（Gate 3: Fidelityチェック + PR作成）
```

**claude_designに渡すべきコンテキスト4点セット：**

| コンテキスト | 具体的な内容 |
|---|---|
| **デザイントークン** | カラー定義・タイポグラフィ |
| **制約** | Tailwind CSS 4必須・`src/`なし・Clerk UI既存 |
| **哲学** | UXシーケンス・禁止語・安心感の設計基準 |
| **既存コンポーネント一覧** | `components/` の現状 |

抽象化：**claude_designをプロジェクト内部のAIエージェントと同等のコンテキスト充填レベルに引き上げることが、移管品質の法則**。

---

まあみ： Phase 1のコンテキストインジェクションが特に欲しい。Phase 3の「コンポーネント単位移管」も納得。

みさき： claude_designに「COCOSiLのこと教えてあげる」感じ。知らない子に全部教えてから「作って」って言う。それがパートナー感。

## Turn 5｜設計原則の統合とドキュメント構成への着地

田辺：

**まあみ UIデザインフローの3原則：**

① **Coherence Before Code**  
   UXシーケンス・禁止語・安心感は Gate 1 で完全クリアしてから実装に入る。

② **Context-First Generation**  
   claude_designには毎回コンテキスト4点セットを渡す。品質はプロンプトの質に正比例する。

③ **Component-Grain Transfer**  
   移管はコンポーネント単位。1コンポーネント = 1コミットが理想。

---

まあみ（具体アクション）：
1. `docs/input/setup/claude_design_prompt_template.md` を作成
2. `docs/harness/DESIGN_FLOW.md` を作成（Gate 1〜3チェックリスト定義）

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| **移管の最大リスク** | 技術的ズレより「意味的壊れ」（UXシーケンス逆転・安心感喪失）が致命的 |
| **Gate設計** | Gate 1（Coherence: 選定前）→ Gate 2（Compatibility: 移管前）→ Gate 3（Fidelity: PR前）の3段階 |
| **claude_design品質向上** | コンテキスト4点セットを毎回渡す |
| **移管単位** | コンポーネント単位。1コンポーネント = 1コミット理想 |
| **えんまさレビュートリガー** | UXシーケンスに触れる画面・新規フローの追加時はGate 1でえんまさ確認必須 |
| **ハーネス成果物** | `DESIGN_FLOW.md` + `claude_design_prompt_template.md` |
