# 議論ログ：TASK-** Issue・ドキュメント自動生成スキルの設計

## 登場人物

- 🧑‍💼 **佐々木（開発ハーネス・自動化設計）**：GitHub Actions・ワークフロー自動化歴8年。開発体験（DX）改善が専門。
- 👩‍💻 **田村（プロダクト管理・ドキュメント設計）**：PdM歴7年。要件定義書→実タスク分解・Issue管理設計が専門。
- 💅 **みさき（ギャル）**：実際のCOCOSiL開発チームユーザー。えんまさ役でタスク振り・確認をする側。

---

## Turn 1｜現状の症状を観察→根本問題の診断

**佐々木：** TASK-INDEXにはタスクが一覧されているのに、GitHub IssueとTSKファイルが完全に手動作成になっています。既存のTSK-DB-001・TSK-UI-001を確認しましたが、どちらも丁寧に手書きされている。TASK-INDEXには未着手タスクが20件以上あって、全部手動でやると最低5〜10時間の単純転写作業になる計算です。

表層的に見れば「チケット作成が面倒」ですが、根本はそこじゃない。**「要件定義書の知識と実作業チケットの間に橋がない」——これが本質的な問題です。** 要件は `§4.1〜§4.10` に詳細に書いてあるのに、TSKファイルを作るたびに担当者がゼロから要件書を読み直して解釈している。これは「文脈の再発見コスト」問題と診断できます。

**田村：** 佐々木さんの診断に補足します。このプロジェクトは **「要件書の質は高いが、実行層への変換機構が欠如している」** 状態です。`cocosil_v2_detailed_requirements_specification.md` は機能ごとに丁寧に書かれている。でも実際の開発者がタスクを始めるとき、毎回この詳細定義書を読み直して「自分の担当箇所を解釈する」作業が発生している。これは**知識の再解釈コスト**であり、しかも解釈のズレがバグの種になる。

このスキルが解くべき本質的な問題は「要件定義書の知識を実作業チケットに転写するコスト」であり、「チケットを楽に作る」ではない。

**みさき：** え待って、つまり今って「やることリスト（TASK-INDEX）」はあるけど、実際の「IssueチケットとTSKファイル」は毎回えんまさが手書きしてるってこと？それって地味にめっちゃしんどくないですか〜？要件書読んで、内容まとめて、テンプレ埋めて...全部手動なんですか？

## Turn 2｜選択肢の発散—評価軸と具体案の配置

**田村：** 評価軸を3つ立てて、選択肢を整理します。

| 軸 | 選択肢 |
|---|---|
| **①自動化の範囲** | A: TASK-INDEX全体をバッチ処理 / B: 指定タスクを個別生成 / C: 両対応（/start-task 統合） |
| **②入力の深さ** | P: TASK-INDEXのみ参照 / Q: TASK-INDEX＋要件定義書の該当§ / R: Q＋git log（実装状況） |
| **③出力の品質** | α: テンプレートを機械的に埋める / β: 要件書から引用して充実させる |

**佐々木：** **C（両対応）＋ Q（TASK-INDEX＋要件定義書参照）＋ β（要件書引用）** を強く推します。`/start-task` のStep 2.6がすでに「TSKファイルが存在しない場合は作成するか」を確認している。ここにこのスキルを呼び込めば、既存ワークフローに摩擦ゼロで組み込まれる。αの「テンプレだけ埋める」スキルを作っても3ヶ月後に「これ使えない」となる経験を私は4プロジェクトで見てきました。βが必須です。

**みさき：** え、Cって「タスク始めますって言ったら自動でIssueもTSKも全部作ってくれる」ってこと？それめっちゃいいじゃないですか〜！てかαとβって何が違うんですか？どっちも「チケット作る」じゃないんですか？

**佐々木：** αだと「TSK-F3-001：統合レポートUI」とタスク名が入るだけ。βだと「F3は共感チャット完了後に開放、Vercel OGで画像生成、受け入れ基準は○○、Gate 2（えんまさ承認）必要」まで入る。前者は「何をするか不明なチケット」、後者は「明日から動けるチケット」です。

## Turn 3｜実需を具体化→抽象パターン化→優先度決定

**みさき：** 私がえんまさ役でタスク振るとき一番困るのって「担当者に何を伝えればいいか迷う」ことなんですよね。だからIssueに最初から「なぜやるか（要件定義書のどこ）」「何をやるか（受け入れ基準）」「誰がやるか（担当ガード）」が入ってれば超助かる〜！あと「Gate 2（えんまさ承認）必要かどうか」も最初からわかると一番うれしい！

**田村：** みさきさんの実需を抽象化すると **「コンテキスト転写の完全性」** という要件です。

```
🔴 必須：
  - TSK ID（TSK-[分類]-[番号]）
  - タスク名・目的（1〜2文）
  - 参照要件（詳細定義書の該当§番号 + 直接引用）
  - 担当者（AGENTS.md §7 の担当ガード）
  - 受け入れ基準（機能完了の定義）

🟡 推奨：
  - 依存タスクの一覧
  - 壊れ方3パターンの該当判定
  - Gate 1/Gate 2 チェックの要否

🟢 任意（担当者が埋める）：
  - 実装メモ・設計メモ
```

**佐々木：** 「受け入れ基準」について補足します。要件書から引用するだけでは不十分で、AGENTS.md §7 の「機能完了の定義（型チェック通過＋ビルド成功＋えんまさ動作確認の3条件）」をテンプレに固定で含める必要があります。これを省くと「ビルドが通ったから完了」という禁止パターンが復活します。

## Turn 4｜技術実装の法則と制約の抽出

**佐々木：** スキルの実装で最重要な「読み順」を法則として定義します。

```
① docs/TASK-INDEX.md
    → 対象タスクのTSK IDと参照要件セクションを取得

② docs/output/requirements/cocosil_v2_detailed_requirements_specification.md
    → 該当セクション（§4.x）を Read

③ AGENTS.md §7
    → 担当ガード（Layer/担当者/Gate要否）を確認

④ docs/output/tasks/_TEMPLATE.md
    → TSKファイルのテンプレート構造を確認

⑤ 生成実行
    → gh issue create → TSKファイル書き出し → TASK-INDEX更新
```

法則：**「参照先を確認してから生成する（Read Before Write）」が絶対原則。**

**田村：** リスクを一つ指摘します。要件定義書が更新されてTASK-INDEXと乖離したとき、どちらを正とするか問題が起きます。推奨は「要件定義書を正とし、差分検出時はユーザーに確認を求める」。`spec-sync` スキルと役割が被る部分があるので連携インターフェースを定義すべきですが、初版はまず「読んで生成する」が確実に動くことを優先します。

**みさき：** え待って、「TSKファイル作ったらTASK-INDEXも自動で更新してくれる」のが一番うれしいんですけど、それって自動でやってくれるんですか？手動で二ヶ所更新するの、マジで忘れそう〜。私えんまさとして「TASK-INDEX見たら空欄だった」ってなったことあって、それが一番困ったんですよね。

**田村：** その指摘が本質を突いています。「生成 → TSKファイル書き出し → TASK-INDEX更新 → Issueリンク記入」を**アトミックに（1回の実行で）**完結させることが絶対条件です。片方だけ更新して終わるスキルは3ヶ月で誰も使わなくなります。

## Turn 5｜設計原則の統合と具体的次の一手

**田村：** 5ターンの議論を設計原則3箇条に集約します：

```
① Context-Complete Generation（文脈完全生成）
   タスクIDを入力したら「なぜ・何を・誰が・Gate要否」が揃ったチケットを出力する。
   要件書引用なしの部分的生成は行わない。

② Read Before Write（先読後書）
   生成前に必ず TASK-INDEX → 要件定義書(§4.x) → AGENTS.md の順で読む。
   この読み順を省略しない。

③ Atomic Update（アトミック更新）
   Issue作成 + TSKファイル生成 + TASK-INDEX更新を1回の実行で完結させる。
   中途半端な生成状態を残さない。
```

**佐々木：** 原則を受けて、明日から動けるスキル実装の具体アクション：

1. `.claude/skills/task-issue-generator/SKILL.md` を新規作成
2. `/start-task` の Step 2.6 からこのスキルを呼び出す連携を追加
3. AGENTS.md §9 のスキル使い分けガイドにトリガーワードを追記
4. TASK-INDEXの「未着手タスク（§4.3〜§4.10）」を一括生成（スキル完成後の最初の実用ラン）

**みさき：** なんか最初「スキルを作るって具体的に何？」て感じで全然ピンとこなかったんですけど〜、話してたら「タスク名言ったら全部揃ったチケット出てくる魔法コマンド」ってことですよね？「Read Before Write（読まずに書くな）」はめっちゃ感覚でわかります〜！えんまさとして「中途半端なTSKファイル来た」が一番まずいので、アトミック更新マジ大事〜！

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| スキル名 | `task-issue-generator` |
| 配置場所 | `.claude/skills/task-issue-generator/SKILL.md` |
| 起動方法 | `/start-task` Step 2.6 統合 ＋ 単独呼び出し両対応 |
| 入力形式 | TSK ID / タスク名 / `--all`（未着手一括） |
| 読み込み順序 | TASK-INDEX → 要件定義書(§4.x) → AGENTS.md → _TEMPLATE.md |
| GitHub Issue | `gh issue create` で自動生成（ラベル・担当者・参照§付き） |
| TSKファイル | `docs/output/tasks/TSK-***.md` を _TEMPLATE.mdベースで充実生成 |
| TASK-INDEX更新 | IssueリンクとTSKファイルパスを同一実行で記入（アトミック） |
| 設計3原則 | Context-Complete Generation / Read Before Write / Atomic Update |
| spec-sync連携 | 要件書更新検出時は差分表示してユーザー確認（拡張フェーズ） |
