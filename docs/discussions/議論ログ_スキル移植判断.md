## Turn 1｜現状診断：COCOSiLのハーネスに何が足りないか

**西村：** まず棚卸しから入りましょう。COCOSiLの現状のハーネス構成を見ると、**protect面**（AIに壊させない防護）はしっかり押さえられています。`prevent-destructive-command.js`によるLayer1ブロック、環境変数の保護、RLS無効化防止——これはR2リスクとして適切な最低限が揃っている。

一方で **produce面**（AIが生産的に動くための支援ツール）が手薄です。`cocosil-domain`がドメイン知識を持ち込み、`language-design`が禁止語を管理し、`requirements-doc-creator`がPRD作成を支援する。しかしこれらは全部「**何を作るか**」を決める上流フェーズ寄りのスキルです。「作りながら検証する」「作った後に片付ける」フェーズのスキルが一本もない。

**梅田：** 同意します。これを開発フローの時間軸で整理すると空白が浮き彫りになります：

| フェーズ | 現在のカバレッジ | 空白 |
|---|---|---|
| ① 目標確認（実装前） | requirements-doc-creator（重い） | 軽量なゴール固めがない |
| ② 実装中 | cocosil-domain・language-design | コード変更の検証支援なし |
| ③ PR作成 | cocosil-work（実装コマンド） | PRサマリー・セキュリティレビューなし |
| ④ ハーネス保守 | skill-improver | 健全度の継続監視なし |

これはスタートアップ初速フェーズでよく起きる問題です。プロダクト保護設計は先行して入れたが、日常的な開発サポートが後回しになっている。3名チームで「えんまさ＝意味設計・まあみ＝フロントエンド・ヒラメ＝バックエンド」という分業構成だと、**「誰が横断的な検証をやるか」が宙に浮きやすい**。

**みさき：** ん〜、なんか私がCOCOSiL使うときの感覚とちょっと似てるかも。診断機能はあるのに「で、結果どう使えばいいの？」ってフォローがない、みたいな——要するに、ハーネスも「守ること」はあるけど「毎日の実装を支援する道具」がまだない、ってことですか？

**西村：** みさきさん、的確な翻訳です。ハーネス設計の言葉に直すと **「Protection-first、Productivity-later症候群」** です。最初のセットアップで保護系hookを充実させると満足感があるので、Productivityツールが積み上がらないまま実装フェーズに突入してしまう。COCOSiLはまさにそのタイミング——v2実装開始直前——にいる。**今が補完するベストウィンドウです。**

## Turn 2｜7候補の仕分け：採用基準マトリクスで整理

**梅田：** 候補7つを評価します。私が使う軸は2つ：**「今すぐ使えるか」**（フェーズ適合）と **「チームコストが低いか」**（維持負荷）です。

```
             ← チームコスト高 ─────── チームコスト低 →

今すぐ使える  ┌────────────────────┬─────────────────────────┐
              │ harness-health-    │ pr-draft-summary        │
              │ improver           │ security-sensitive-     │
              │ code-change-       │ change-review           │
              │ verification       │ goal-grill              │
              └────────────────────┴─────────────────────────┘

まだ早い      ┌────────────────────┬─────────────────────────┐
              │ kecku-harness-     │ docs-sync               │
              │ bootstrap          │                         │
              └────────────────────┴─────────────────────────┘
```

**西村：** この配置に補足します。チームコストの主因は「**移植後のカスタマイズ量**」です：

| スキル | 汎用スキルとしての完成度 | COCOSiL固有の追加作業 |
|---|---|---|
| **pr-draft-summary** | ★★★★★ | なし。7行のミニマル構成 |
| **security-sensitive-change-review** | ★★★★☆ | Supabase RLS・Clerk認証の項目追加 |
| **goal-grill** | ★★★★☆ | `docs/goals/`ディレクトリ追加のみ |
| **code-change-verification** | ★★★☆☆ | pnpm前提・AGENTS.mdとの整合確認が必要 |
| **docs-sync** | ★★★☆☆ | COCOSiL固有のdocs/構造を参照先として追記 |
| **harness-health-improver** | ★★☆☆☆ | `source-registry.yml`を新規作成する必要あり |
| **kecku-harness-bootstrap** | ★☆☆☆☆ | `disable-model-invocation: true`のスクリプト型。ハーネスは既に完成済みで用途なし |

**みさき：** えー！kecku-harness-bootstrapって名前が一番インパクトあるのに評価が最下位なんですか（笑）なんで？

**梅田：** （笑）これは「ゼロからハーネスを作る道具」なんです。でもCOCOSiLはもうハーネスが動いてる。既に家が建ってるのに「家の建て方マニュアル」を移植してもしょうがない、という話です。

## Turn 3｜チームの実需と「使われなくなるスキル」のリスク

**みさき：** 梅田さん、ちょっと聞いていいですか。えんまさとヒラメとまあみが毎日使うって考えたとき、「使うシーンが頭に浮かぶ」やつはどれですか？私、道具って多すぎると全部使わなくなる経験あって〜。スキルを増やすのも同じだと思うんですけど。

**梅田：** 核心的な問いです。私が30件の失敗プロジェクトから学んだ鉄則は「**道具は使われないと存在しないのと同じ**」。具体的に各メンバーの使用シーンを想像してみます：

**えんまさ（意味設計）が使いそう：**
- `goal-grill` → 新機能のゴールを固める前に。今は requirements-doc-creator より軽い選択肢がない
- `pr-draft-summary` → PRのdescriptionを書くとき

**ヒラメ（バックエンド）が使いそう：**
- `security-sensitive-change-review` → Supabase migration・RLS変更のレビュー前に
- `code-change-verification` → APIルート変更後の検証手順の確認に

**まあみ（フロントエンド）が使いそう：**
- `pr-draft-summary` → PRを出すたびに
- `code-change-verification` → コンポーネント変更後の最小限検証確認に

**西村：** この分析から「使用シーンの厚み」を抽象化すると3パターンに分類できます：

**Pattern A：毎PR使う（生活インフラ型）**
→ `pr-draft-summary`、`security-sensitive-change-review`（Supabase変更時）

**Pattern B：機能着手時に使う（プランニング型）**
→ `goal-grill`、`code-change-verification`

**Pattern C：ハーネス保守時に使う（インフラ保守型）**
→ `harness-health-improver`、`docs-sync`

Pattern Cは「困ったときに使う」系なので、場所を把握していれば今すぐ全移植しなくていい。逆にPattern A・Bは今すぐないと「手動でやる」か「やらない」の2択になる——**そちらのコストの方が高い**。

**みさき：** あーそれわかります！うちも「タスク管理ツール5個入れたら全部中途半端になった」ことあって〜。てかPattern Aって「ないとPR出すのがめんどくさくなる」やつですよね。それ早く入れた方がよくないですか？

## Turn 4｜over-harnessing の罠：スキルを増やすことのリスク

**西村：** みさきさんの直感は正しい。ただ「追加することのコスト」を正直に話しておかなければいけません。

スキルを増やすことには **3つの膨張リスク** があります：

**① コンテキスト肥大化リスク**
`.claude/skills/`に置いたSKILL.mdは毎回のプロンプトに読み込まれます。`goal-grill`のSKILL.mdは253行。現在5つのSKILL.mdが既に存在する状態で、`harness-health-improver`の220行も加わると、文脈コストが線形に増加します。

**② トリガー曖昧化リスク**
`goal-grill`と`requirements-doc-creator`は機能的に重なる部分がある——「軽い実装ならgoal-grill、PRDが必要ならrequirements-doc-creator」という使い分けが自明でないと、毎回人間が判断することになる。

**③ 陳腐化リスク**
`harness-health-improver`は`source-registry.yml`という別ファイルを参照設計で参照している。COCOSiLにそのファイルがない状態で移植すると「なんとなく動くが微妙」な状態になる。

**梅田：** このリスクを踏まえると、採用基準の原則が導き出せます：

> **「3ヶ月以内に3回使う確信がないスキルは今期移植しない」**

この数値根拠は私の経験則で、スタートアップの意思決定サイクル（2週間スプリント × 6本 = 3ヶ月）内に習慣化しないツールは定着しない、という観察です。この基準で7候補を再判定：

| スキル | 3ヶ月3回達成の確信 | 理由 |
|---|---|---|
| pr-draft-summary | ✅ 確実 | PRは週1〜2回出る |
| security-sensitive-change-review | ✅ 確実 | Supabase変更は月3〜5回は起きる |
| goal-grill | ✅ 高い | 機能着手のたびに使う（月2〜3機能） |
| code-change-verification | 🟡 条件付き | AGENTS.mdとの重複解消が前提 |
| docs-sync | 🟡 条件付き | APIが安定するまで使う機会が少ない |
| harness-health-improver | 🔴 低い | source-registry.ymlを作らないと動かない |
| kecku-harness-bootstrap | 🔴 ゼロ | COCOSiLでは用途なし |

**みさき：** harness-health-improverって名前かっこいいから使いたくなるけど、動かすために別のファイルも先に作らないといけないって、それコスパ悪すぎ〜！「準備が大変な道具は使わなくなる」の典型じゃないですか？

**西村：** まさに。それを法則化すると **「セットアップコストは使用確率と反比例する」** です。source-registry.ymlを自力で作るコスト（30〜60分）+ 動作確認コストを考えると、即時採用は見送りが正解。ただし将来`HARNESS_HEALTH.md`にOpen Gapが5件以上蓄積したとき、そのコストは一瞬で回収できる——つまり「**今は不採用、将来候補**」という位置づけです。

## Turn 5｜設計原則の確立と最終採用リスト

**西村：** 5ターンの議論を集約して、COCOSiLのスキル移植に適用すべき設計原則を打ち立てます：

---

**COCOSiLスキル採用の3原則**

**① Fill Gaps, Not Laps（空白フェーズを埋める、重複フェーズには入れない）**
現行スキルが「上流（目標設定・ドメイン言語）」に偏っているので、「実装中・PR作成・保守」フェーズを補う。しかし同一フェーズのスキルを2本入れると選択コストが生まれる。

**② Zero Prerequisite Rule（動かすための前提が今のリポジトリにないものは移植しない）**
`source-registry.yml`・`docs/harness/`等、「先に別ファイルを作らないと動かない」スキルは採用基準を下げる。移植前に前提が自己充足していること。

**③ Always On Track（毎PR・毎着手で自然に使われる設計にする）**
「困ったときのために入れる」スキルはPattern Cに分類し、今期は見送る。Pattern A（毎PR）・Pattern B（毎着手）から優先的に埋める。

---

**梅田：** この原則に基づく最終決定：

| スキル | 判定 | 移植時の変更点 |
|---|---|---|
| **pr-draft-summary** | ✅ **即時採用** | 変更不要。そのままコピー |
| **security-sensitive-change-review** | ✅ **即時採用（カスタマイズ）** | Supabase RLS・Clerk認証・Next.js API routeの保護項目を追記 |
| **goal-grill** | ✅ **即時採用（軽微調整）** | `docs/goals/`ディレクトリを確保。requirements-doc-creatorとの使い分けをAGENTS.mdに1行追記 |
| **code-change-verification** | 🟡 **採用（AGENTS.md整合確認後）** | 既存検証コマンドとの重複を解消してから |
| **docs-sync** | 🟡 **3ヶ月後に再評価** | APIが安定してドキュメントが充実したタイミングで |
| **harness-health-improver** | 🔴 **将来候補（今期見送り）** | HARNESS_HEALTH.mdのOpen Gap 5件超過が移植トリガー |
| **kecku-harness-bootstrap** | 🔴 **不採用** | ハーネスは既成。用途なし |

**みさき：** なんか最初「7個どうするか」って考えたら頭パンクしそうだったけど、最終的に「今すぐ3つ、様子見1つ、あとは不要」ってスッキリした〜！てか `pr-draft-summary` って変更なしでそのままコピーできるって、コスパ最強じゃないですか？えんまさもヒラメもまあみも、PRを出すたびに使えるってことですよね。それ今日入れた方がいいと思う〜！
