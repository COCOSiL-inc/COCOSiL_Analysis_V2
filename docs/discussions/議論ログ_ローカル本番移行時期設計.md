# 議論ログ：ローカル→本番移行時期設計

**日付**: 2026-05-04  
**実行モード**: Agent Team Mode（独立エージェント × 3）  
**お題**: COCOSiL V2 の実装ロードマップをもとに、ローカル確認→本番デプロイの移行時期と判断基準を設計する

## 登場人物

- 🧑‍💻 **木村 誠（DevOpsアーキテクト）**：Vercel/Supabaseを使ったプロダクション投入歴8年。「インフラの環境差は本番に出るまで見えない」が口癖。
- 👩‍🔬 **岡田 理恵（SREエンジニア）**：SRE歴7年、計測基盤設計が専門。「計測できない指標で判断するな」派。
- 💅 **みさき（ギャル）**：IT素人だけど感性が鋭い。実際のWebアプリユーザー。「難しいことはわかんないけど使えるかどうかはわかる」。

## Turn 1｜初回本番デプロイの判断基準

**木村：** 結論から言います。「ローカルで確認済み」は本番GOの条件にはなりません。V2のスタックを見ると、RLS（Row Level Security）、Clerk JWT、Vercelエッジネットワークの3つは、ローカル環境では本番相当の動作を保証できません。特にRLS + Clerk JWTの組み合わせは、.env.localとVercelのEnvironment Variablesで微妙な設定差が出やすく、ローカルでは通るがプレビューで壊れるパターンが8割の案件で発生します。

**岡田：** 木村さんの指摘に補足します。「ローカルで通ったのにプレビューで壊れた」を防ぐ最小条件は3点です。①`pnpm typecheck`（型整合性）②`pnpm build`（Next.jsビルド）③えんまさがVercelプレビューURLで手動確認（UXシーケンスの実流確認）。この3条件を「アトミック確認ループ」と定義し、全通過まで本番マージを禁止にする。問題は③の「えんまさが何を確認するか」の観点が今まで曖昧だったことです。

**みさき：** え待って、つまりプレビューURLって何ですか？本番と違うサイトが出るってこと？

**木村：** 正解です。VercelはブランチをpushするたびにユニークなプレビューURLを発行します。`feature/phase-a-telemetry`ブランチなら`feature-phase-a-telemetry-xxx.vercel.app`のような固有URLが出る。インフラは本番と同一ですが、URLが違うのでデータは本番に混入しません。

**みさき：** あ、それなら安心ですね〜！「本番ではないのにVercelで動く」ってやつか。めっちゃ便利じゃないですか。

---

## Turn 2｜Phase別のローカル/本番使い分け

**岡田：** Phase A（PostHog + events_telemetry）は「本番ユーザーなしに意味がない計測」が本質です。5種イベント（`chat_phase_transition`など）はリアルユーザーの行動から取るから意味がある。ローカルで開発者が操作したデータはノイズにしかなりません。だから**Phase Aは本番投入がゴール**。計測基盤を作っただけで満足して本番に出さないのが最も危険なパターンです。

**木村：** Phase B（pgvector + RAG）は逆で、ローカルでかなり進められます。pgvectorのembeddingロジック、RAG検索の精度確認、セッション終了フックの動作確認——これらはseed dataで代替できます。ただし岡田さんが言ったように、感情的重みのある本物のチャットログでないと「気づきの要約」の品質を正確に評価できない。Phase Bの本番投入条件は「Phase Aで50人以上の行動ログが蓄積した後」です。

**岡田：** Phase C（Autogenesisループ）については補足します。prompt_versionsテーブルとVercel Cronバッチは、本番データなしには開発・ユニットテストは可能ですが、eval（改善スコアの判定）は本物のセッションデータがないと意味をなしません。Phase B完了後、100人超のデータが揃ってから初めてPhase Cのevalが現実の指標になる。

**みさき：** つまり、Phase Aを早く本番に出せばそれだけPhase B、CのGOが早くなるってこと？なんか連鎖してますね〜。

**木村：** その通りです。Phase Aを遅らせると全部後ろにずれる。「ローカルで完成度を上げてから本番に出す」という発想がV2においては致命的に遅い。Phase Aはプレビュー確認が取れた段階で即本番投入すべきです。

## Turn 3｜Vercelプレビューデプロイの活用

**木村：** プレビューURLで確認すべき最重要事項は2点です。①RLS + Clerk JWTが本番同一構成で動いているか（`supabase.from('events_telemetry').insert()` が403を返さないか）②Vercelエッジネットワークのキャッシュがリクエストに干渉していないか。特に②はローカルでは完全に見えません。Next.js App RouterのServer Componentsはデフォルトでキャッシュされますが、そのキャッシュ無効化設定の確認はプレビュー環境が初めての実機確認になります。

**岡田：** えんまさの確認フローを設計するとこうなります。Layer A（機械）→ Layer B（プレビュー）→ Layer C（本番）。Layer Aは`pnpm typecheck && pnpm build`が通ることで機械が保証。Layer BはVercelプレビューURLで「腑落ち体験が壊れていないか」をえんまさが手動確認。Layer Cはmainブランチへのマージで自動デプロイ。重要なのは「Layer B確認なしにLayer Cへ進まない」という原則を守ることです。

**みさき：** えんまさって何を見ればいいんですか？コードとか見てもわからないですよね？

**岡田：** えんまさはコードを見なくていいんです。チェックリストは3点のみ。①UXシーケンス（共感→安心→分析→行動）の順序が守られているか②診断結果が正しく表示されるか（INFJ等のMBTIタイプが一致しているか）③「腑落ち感」がある——つまり「これ私のことだな」と感じられるか。最後の1点は機械に判断できません。これがえんまさ確認の存在意義です。

**みさき：** あ、それなら私でもできますよ〜！てかそれ普通のユーザー目線じゃないですか？めっちゃ大事〜。

## Turn 4｜DB分離の判断基準と本番投入タイミング

**木村：** Phase B→Cの移行で必ず問題になるのが「いつDBを物理分離するか」です。私が8年の経験から導いた法則を提示します。**「データが判断ループの入力になった瞬間に物理分離する」（木村の法則）**。Phase AのPostHogイベントは「観察する」だけなので同一DBでいい。Phase Bのuser_insights_memoryも「セッション内の参照」なので論理分離（RLS + previewフラグ）で十分。しかしPhase Cのprompt_versions + eval_resultsは「AIがプロンプトを改善する判断ループの入力」になる。このデータが本番のユーザーデータと同じDBに混在すると、Autogenesisがユーザーデータを学習データとして参照するリスクが生まれます。Phase C起点で物理DB分離が必須。

**岡田：** 木村さんの分析を本番投入タイミングに繋げます。**「計測に必要な母数・環境が揃った瞬間が本番投入点」（岡田の法則）**。Phase Aは本番ユーザーがいないと計測できない→今すぐ本番に出す。Phase Bはembeddingの精度評価に50人分のチャットログが必要→Phase A完了後。Phase Cはevalの安定に100人分のデータと30日間のスコア推移が必要→Phase B完了後。この「母数条件」を定量化することで、感情的な「まだ早い/もう出せる」議論をなくせます。

**木村：** 補足します。Phase A/Bは「論理分離」で実装コストを最小化できます。具体的には`events_telemetry`テーブルにRLSポリシーを設定し、previewセッションのデータには`is_preview: true`フラグを立てる。これで本番データとプレビューデータを同一DBに入れながら分析時に除外できます。実装1時間以内のコスト。

**みさき：** なんかデータの扱いって難しそうですけど、ユーザーとして気になるのは「自分のデータが何に使われるか」なんですよね。「ベータテストに使われます」って書いてあるより「このデータは分析にのみ使われ、AIの学習には使いません」の一行の方が安心できる。

## Turn 5｜設計原則の統合と具体的デプロイロードマップへの着地

**木村：** 5ターンの議論を「ローカル→プレビュー→本番の設計3原則」として集約します。

**① "Preview is Production, Minus Real Users."**  
プレビュー環境は本番コードを走らせる場。違いは「人間のデータが入るか」だけ。インフラ・RLS・環境変数は本番同一構成で動かす。「プレビューで試してから本番」ではなく「プレビュー＝本番の最終確認」という認識が正しい。

**② "Separate When You Measure."（岡田の法則）**  
計測に足る母数が揃うまでDB共有は許容。Autogenesis Phase CでA/Bテストを回す段階になったら物理分離を実行する。それ以前の分離はオーバーエンジニアリング。「いつ分離するか」は技術的判断ではなく「データが判断ループの入力になるか」という機能的判断。

**③ "Data Fate Visible, Always."（みさきの法則）**  
データが何に使われるかを一行で書かないUIは出さない。プレビュー段階から「このデータは本番に残りません」の一文を義務化する。ユーザーの安心感はベータラベルよりデータ扱いの透明性から生まれる。

---

今日から取るべき3アクション（技術側）：

1. **Vercelプレビューに `NEXT_PUBLIC_IS_PREVIEW=true` を追加** → UIバナーとデータ扱い表記を条件分岐で出す
2. **RLSポリシーに `preview_session` フラグを追加**（Phase B起点） → プレビューデータを論理隔離できる状態にする
3. **`lib/env` に `isPreview()` ヘルパーを1関数追加** → 全コンポーネントが環境判定を一箇所から参照できる状態にする

加えて、AGENTS.mdのアトミック確認ループに「プレビュー確認チェックリスト」を追加することがPhase A完了の定義になる。

**岡田：** デプロイロードマップを条件付きマイルストーン表として整理します。

| フェーズ | GO条件 | デプロイ対象 | DB状態 |
|---|---|---|---|
| **Phase A 本番** | typecheck pass + えんまさプレビュー確認 | 診断フロー・レポート表示 | 本番DB（共有許容） |
| **Phase B 本番** | Eval指標が30日間安定 + 内省スコア計測ロジック確定 | Autogenesis Evaluate層 | 論理分離（同一DB） |
| **Phase C 本番** | 判断ループへの入力データが月100件超 + えんまさSchema変更Approve | 自律改善パイプライン | 物理DB分離必須 |

**えんまさ確認フロー**：ローカルで `pnpm typecheck && pnpm build` が通った時点でPRを作成し、Vercelプレビューを発行。えんまさがプレビューURLでUXシーケンス（共感→安心→分析→行動）の実流を手動確認。「腑落ち体験が壊れていない」確認後にmainマージ→本番自動デプロイ。Phase B以降はこのフローに「内省スコアの数値レビュー」が加わる。岡田の法則に従い、計測母数が揃う前に次フェーズのGOは出さない。

**みさき：** なんか最初は「本番で使えない＝まだ怖い・未完成」って思ってたけど、結局「データの扱いが透明で、段階的に試せる環境があれば使える」ってことですよね。プレビューURLで実際に触れるなら、私はそのタイミングで使い始めます。
