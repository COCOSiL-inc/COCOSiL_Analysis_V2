# 議論ログ：まあみワークフロー一本道化設計

## 登場人物

- 🔧 岡田 純（ワークフローデザイナー）：人間×AIの協働ワークフロー専門。Lean/DevOps原則をAI協働に適用。「分岐は停止の予告」が口癖。
- 👩‍💻 まあみ（フロントエンドエンジニア）：DESIGN_FLOW.mdの実際の使用者。どこで止まるかを最もよく知っている当事者。
- 💅 みさき（ギャル）：実際のCOCOSiLユーザー。「このフロー直感で分かる？」判定役。

## Turn 1｜現在の DESIGN_FLOW.md で止まる場所の特定

まあみ（3つの止まり場所）：

1. **Gate 1の12チェックボックス問題** — UXシーケンス4項目・禁止語5項目・トーン3項目を毎回全部チェックする義務感で手が止まる。3案生成して全部に12チェックは心理的に重い。

2. **えんまさレビュートリガーの判断不能問題** — 「UXシーケンス全体に関わる新規フロー」とあるが、まあみが作るものはほぼ毎回新規フロー。ほぼ毎回えんまさに聞くことになり、返答待ちで止まる。

3. **テンプレートのメンテ義務問題** — コンポーネント追加のたびにテンプレートを更新する義務。先延ばしが発生し、古い情報で生成することになる。

岡田：意思決定ポイントを数えると停止点が分かる。現フローの停止確率の高い地点は「Gate 1（まあみが全チェック）」と「えんまさトリガー判断（まあみが判断できない）」の2つ。

抽象化：**フローの停止点の数 = 判断委譲できていない責務の数**。

みさき：チェックリストを全部まあみがやるなら、AIがやればいいじゃないですか？結果を受け取るだけなら動ける。

## Turn 2｜「止まらない」フローの設計軸

岡田（3軸）：

**軸1：判断とアクションを分離する**  
まあみがやるべきは「選定」と「進む判断」のみ。チェック・コマンド実行・ファイル更新はAIに委譲。

**軸2：ブロッキングをノンブロッキングに変える**  
えんまさへの「事前確認（同期）」→「PRレビュー（非同期）」に後ろ倒し。まあみは確認待ちをせずに実装を完了させ、えんまさをPRにアサインする。

**軸3：メンテナンス義務をゼロにする**  
「テンプレートを更新せよ」という記憶依存タスクは必ず腐る。AIが毎回 `ls components/` を実行して動的収集する設計に変える。

```
【一本道フロー】
Step 1: AIに「〇〇のUI生成して」と依頼
  → AIが components/ スキャン → テンプレートに自動充填 → claude_designに投げる
Step 2: AIに「Gate 1チェックして」と依頼 → OK/NG+理由を受け取る
Step 3: まあみが「どの案が好き？」で選定（唯一の意思決定）
Step 4: AIが /start-task → 移管 → Gate 2自動チェック（typecheck）
Step 5: まあみが pnpm dev で目視確認
Step 6: AIが /finish-task → PR → えんまさをアサイン（非同期）
```

まあみ：Step 2の「Gate 1チェックして」一言でAIが12項目チェックしてくれる設計が欲しかった。軸2（ブロッキング→ノンブロッキング）も目から鱗。えんまさ確認をPRに後ろ倒しにしていい。

みさき：Step 3「どの案が好き？」がシンプルでいい。デザイン選定が「好みで選んでいいのか、正しくて選ぶのか」が曖昧だったのも解消される。

## Turn 3｜えんまさ依存の非同期化とGate 1のAI委譲

まあみ（えんまさ確認が必要なケースの分類）：

**AIがNoGoを出して⚠️フラグを立てる（少数派）：**
- 診断結果の見せ方を根本変更する場合
- UXシーケンス全体を変更するフロー設計
- AI生成プロンプトの見た目に影響するUI変更

**AIがOKを出してPRに流す（大多数）：**
- 新規コンポーネントの追加（既存フローの中に入るもの）
- スタイル調整・リファクタリング
- 既存フローの一部コンポーネント

岡田：えんまさ確認が必要でも、**まあみの作業は止めない**。PRに `⚠️ えんまさ確認必要: <理由>` を記載してアサインするだけ。まあみは次のコンポーネントに動く。

抽象化：**「確認必要」= 「進行禁止」ではなく「PRにフラグを立てる」**。停止をフラグに変換することで一本道が維持される。

## Turn 4｜AIパートナーへの委譲設計

岡田：AIパートナーが DESIGN_FLOW.md を読んだとき、どう動くかを明示する。

```
❌ 現状（責務混在）:
「チェックリストを確認して、問題があればえんまさに連絡する」

✅ 改善後（責務分離）:
【AIが実行】Gate 1チェックを実行し、結果をまあみに報告
  → OK: 「Gate 1クリア。選定に進んでください」
  → NG軽微: 修正案を提示
  → NG重大: 「PRに⚠️フラグを立てます」と明示して進める

【まあみが実行】AIの報告を受けて選定する
```

まあみ：テンプレートが `docs/input/setup/` ではなく `docs/output/template/` に作られているのも修正が必要。リンク切れはAIの「サイレントな停止点」になる。

岡田：**ドキュメントの品質はリンクの正確性によって決まる**。リンクが正しくなければドキュメントは道標にならない。

## Turn 5｜設計原則の統合

岡田（3原則）：

① **Delegate to AI, Decide by Maami（委譲と決定の分離）**  
チェック・実行・参照はAIが行う。まあみが行うのは選定と目視のみ。

② **Flag, Don't Block（停止をフラグに変える）**  
えんまさ確認が必要でもまあみの作業は止めない。PRに⚠️フラグを立てて非同期化。

③ **Live Links Only（死リンクは停止点）**  
ドキュメント内リンクは常に正確。テンプレートパスのズレ・存在しないパスは即修正。

## ✅ 議論まとめ

| 項目 | 変更前 | 変更後 |
|---|---|---|
| Gate 1チェック | まあみが12項目を手動 | AIが実行し結果を報告 |
| えんまさ確認 | 事前確認（ブロッキング） | PRに⚠️フラグ（非同期） |
| テンプレート維持 | まあみが手動更新 | AIが毎回 `ls components/` で動的収集 |
| フロー図 | Phase + Gate（多段階分岐） | Step 1〜6（番号付き一本道） |
| まあみの作業 | チェック・判断・更新・確認すべて | 選定と目視のみ |
