## Turn 1｜現状診断：「ブランチ管理が面倒」の根本にあるもの

**木村：** COCOSiL V2の現状を整理すると、CIはtypecheck/lintが動いていて、CODEOWNERSで担当者の割り当ても明示されている。PRテンプレも完備。インフラとしては十分に整っています。

ただ、今回「ブランチの自動管理が欲しい」という要求が出てきたということは、何かが詰まっている。私が3〜5人チームで見てきたパターンだと、これは表面上は「ブランチが溜まる・消えない・命名がバラバラ」という症状ですが、根本は **「人間の手動操作を前提にした設計に、チームの作業リズムが追いつかなくなっている」** という問題です。

COCOSiL特有の状況として：えんまさ（意味設計）・ヒラメ（構造設計）・まあみ（UI）という3つの異なる作業レイヤーが存在する。レイヤーをまたぐ変更が少ない分、各人が並行してブランチを持つことが多くなる。結果、マージ後もブランチが消えずに残り、`git branch -r` が汚くなっていく。

**松本：** 木村さんの診断に同意しつつ、もう一段掘り下げると——ブランチ管理の手間は、実は **「作業開始コスト」と「作業終了コスト」の非対称性** から来ています。

ブランチを作るのは軽い（`git checkout -b`）。でも終わらせる——PRを出して、レビューを待って、マージして、ブランチを削除して——には複数のアクションが必要。人間は「終わらせる作業」を後回しにしやすい。これが積み重なるとブランチ墓地ができる。

自動化で介入すべきは「終了コスト」側が圧倒的に効果的です。作成自動化は後述しますが、まず「終わらせる側を自動化する」ことが先です。

**みさき：** わかる〜！PR出してマージされたのに、なんかブランチそのまま残ってるんですよね。で「これ消していいやつ？」ってなって結局誰も消さない、みたいな。てかブランチ名も `fix/hoge` とか `test-aaa` とかあって、何のブランチかマジわからないんですけど〜。それも自動でなんとかなります？

## Turn 2｜選択肢の発散：自動化の2軸で整理する

**松本：** まずフレームワークを立てましょう。ブランチ自動管理の施策は **「作成の自動化」** と **「削除の自動化」** の2軸で整理できます。加えて各施策に「GitHub native機能で実現可能か / GitHub Actions必須か」という実装コスト軸もある。

```
                  ┌─────────────────────────────────────────────┐
                  │          自動化の選択肢マップ                 │
  ┌───────────────┼────────────────────┬────────────────────────┤
  │               │  GitHub native     │  GitHub Actions        │
  ├───────────────┼────────────────────┼────────────────────────┤
  │ 削除の自動化  │ ① マージ後ブランチ  │ ③ 古いブランチの定期   │
  │               │   自動削除（設定1つ）│   検出・削除           │
  ├───────────────┼────────────────────┼────────────────────────┤
  │ 作成の自動化  │ ② (ほぼ不可)       │ ④ Issue作成 / ラベル付け│
  │               │                    │   時のブランチ自動生成  │
  └───────────────┴────────────────────┴────────────────────────┘
```

**木村：** 施策をリスク別に並べ直すとこうなります：

| 施策 | 効果 | リスク | 実装コスト |
|---|---|---|---|
| ① マージ後自動削除 | 🔴高 | 🟢低（マージ済みなので安全） | 🟢低（GitHub設定1クリック） |
| ② 命名規則の強制 | 🟡中 | 🟢低 | 🟡中（Branch Protection Rule） |
| ③ 古いブランチ定期削除 | 🟡中 | 🔴高（誤削除リスク大） | 🟡中（Actions必要） |
| ④ Issue連動ブランチ作成 | 🟡中 | 🟢低 | 🔴高（Issues活用前提） |

**みさき：** え待って、①のマージ後自動削除って設定1クリックなんですか？！それマジでやってないだけ？（笑）てかそれが一番効く気がするんですけど、今なんでやってないんですかね？あと②の命名規則の強制って、変な名前でブランチ作ると怒られる感じ？

## Turn 3｜要件の確定：COCOSiL3人チームで本当に必要なもの

**松本：** COCOSiLチームの実需を3人のレイヤーから整理すると：

```
えんまさ（意味設計）の実需：
  - docs/** や lib/prompts/** を長期で育てるブランチが多い
  - → 長期ブランチを「保護」する仕組みが必要

ヒラメ（構造設計）の実需：
  - app/api/** の feature ブランチは短命。マージしたら即消えてほしい
  - → ①が直接効く

まあみ（UI）の実需：
  - PR待ちが長いと古いブランチが溜まりがち
  - → ブランチ名に何のタスクか分かる識別子が欲しい
```

抽象パターン：**「短命ブランチの自動終了」と「長命ブランチの明示的保護」は、同じルールでは管理できない**。

**具体的優先度：**

🔴 **必須（今週実装）**：①マージ後自動削除（GitHub設定）+ ②ブランチ命名規則の文書化
🟡 **推奨（Sprint内）**：命名プレフィックス強制チェック（Actions）+ 長期ブランチの明示的リスト化
🟢 **任意（後回し可）**：Issue連動ブランチ自動作成、古いブランチ定期通知

**木村：** 長期ブランチの保護は `.github/PROTECTED_BRANCHES.txt` というリストファイルをリポジトリに置き、削除系ActionsはこのリストをSKIPする設計が現実的。

**みさき：** てかブランチ命名って、例えばどんな名前にするんですか？`feature/F1-mbti-form` みたいな感じ？えんまさって `docs/concepts-docs-update` って名前のブランチ今使ってますよね。これはルール的にOKなの？

## Turn 4｜技術深掘り：命名規則の設計と削除ガードの実装

**木村：** みさきさんの `docs/concepts-docs-update` は **プレフィックス/スラッグ** 形式で良い命名です。COCOSiL命名規則案：

```
プレフィックス設計：
  feature/<slug>    → 新機能開発（ヒラメ・まあみ用）
  fix/<slug>        → バグ修正
  docs/<slug>       → ドキュメント・仕様書（えんまさ用途が多い）
  chore/<slug>      → 設定・依存関係・CI変更
  refactor/<slug>   → リファクタリング

<slug> の推奨形式：
  - kebab-case のみ
  - 任意で issue番号プレフィックス：feature/42-mbti-result-page
  - 日本語禁止（git操作時のエンコード問題回避）
```

命名違反は「プッシュブロック」ではなく「PRコメントで警告のみ」が3人チームには適切。

**松本：** 削除の自動化を3レベルで設計する原則：

```
Level 1：確認なしで削除（完全自動）
  → 対象：マージ済みPRのブランチのみ
  → 実装：GitHub設定「Automatically delete head branches」

Level 2：通知ありで削除（自動だが可視）
  → 対象：90日以上更新なし かつ PROTECTED_BRANCHES.txt未記載
  → COCOSiLには今は不要（ブランチ数が少ない）

Level 3：要承認で削除（人間が判断）
  → 対象：長期ブランチ・mainへのforce push等
  → 実装：hookで既にブロック済み
```

COCOSiLに今必要なのは **Level 1だけ**。Level 2は6ヶ月後に判断。

**みさき：** え、Level 1って設定1クリックって言ってましたよね？！それだけで全然違くなりそう〜！てかLevel 3は既にhookでやってるんですね。じゃあ今ない穴はLevel 1だけ？

## Turn 5｜方針統合：COCOSiL V2ブランチ管理の3原則と具体アクション

**木村：** 設計原則を3つに集約します。

**【COCOSiL V2 ブランチ管理の3原則】**

> **① Merge Once, Delete Immediately**
> マージされた瞬間に自動削除。人間の意思決定を挟まない。
> → GitHub設定「Automatically delete head branches」ON

> **② Name First, Automate Second**
> 命名規則を先に文書化し、自動化はその後。
> → AGENTS.md §9 にブランチ命名規則を追記

> **③ Long-term Branches are Explicit, Not Default**
> 長命ブランチはデフォルトではなく「明示的に守るもの」として宣言する。
> → `.github/PROTECTED_BRANCHES.txt` に長期ブランチを列挙

**松本：** 今日できること（30分）:
1. GitHubリポジトリ Settings → General → 「Automatically delete head branches」をON
2. AGENTS.md にブランチ命名規則セクション追記
3. `.github/PROTECTED_BRANCHES.txt` 新規作成

Sprint内にやること（2〜3時間）:
4. PR作成時に命名規則違反を検出してコメントするGitHub Actionsを追加
5. CIの必須レビュー設定（Branch Protection Rules）をGitHubに追加

今はやらないこと:
- 古いブランチの定期削除Workflow（墓地が実際に問題になってから）
- Issue連動ブランチ自動作成（Issueの使い方が固まってから）

**みさき：** なんか最初「めっちゃ複雑な仕組み作るのかな」と思ってたけど、今日できる一番大事なことって設定1クリックだったんですね（笑）でも命名規則が決まったら「これどのブランチ？」ってならなくなるし、長期ブランチのリストがあったら誤削除も防げる。マジでシンプルなのが一番強い〜。

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| マージ後の削除 | GitHub設定「Automatically delete head branches」をONにする（手動1クリック） |
| 命名規則 | `feature/` `fix/` `docs/` `chore/` `refactor/` + kebab-case slug |
| 長期ブランチの保護 | `.github/PROTECTED_BRANCHES.txt` に明示的に列挙 |
| 命名違反の扱い | プッシュブロックではなく、PRコメントで警告のみ |
| 古いブランチ定期削除 | 今は実装しない。6ヶ月後に再検討 |
| Issue連動ブランチ作成 | 今は実装しない |
| hookとの整合 | `git push --force` 等は既存hookでブロック済み。追加対応不要 |
