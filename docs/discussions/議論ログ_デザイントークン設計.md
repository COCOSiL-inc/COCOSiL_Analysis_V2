# 議論ログ：デザイントークン設計

## 登場人物
- 🎨 **橋本（UIデザインシステム専門家）**: デザインシステム歴8年。SaaS系プロダクトのブランドカラー設計・トークン定義が専門。「一貫性は選択の削減から生まれる」が信条。
- 👩‍💻 **松田（フロントエンドエンジニア）**: Tailwind CSS/React歴6年。デザイントークンとコンポーネント間整合性の担保を専門とする。
- 💅 **みさき（ギャル）**: 26歳の実際のCOCOSiL候補ユーザー。職場の人間関係にモヤモヤして、性格分析に興味あり。

## Turn 1｜現状診断：「落ち着いた・誠実」という指定が生む問題

**橋本**: 現テンプレートの「ビジュアルトーン」を見ると、「落ち着いた・誠実・科学的な印象を優先する」という形容詞しかありません。私がSaaS案件で繰り返し見てきた失敗パターンです。AIは毎回違う「誠実な色」を選ぶ——あるとき `#2563EB`（ブルー）、次は `#0D9488`（ティール）、また次は `#6366F1`（インディゴ）。10コンポーネントができあがると10種類のブルー系が混在する。本質的には「毎回の生成でブランドが再定義される問題」です。

**松田**: 技術面でも補足すると、Tailwind CSS v4では `tailwind.config` のカスタムカラー拡張が変わり、CSS変数 + `@theme` ブロックが主軸になりました。つまりテンプレートに「`bg-[#6366F1]` を使え」と書くより、`bg-primary` のような意味的クラス名で統一するほうが実装整合性が高い。形容詞だけのテンプレートは、技術的にも「手がかりがない」状態です。

**みさき**: え待って、「誠実な感じ」って書いてあっても伝わらないってこと？私てっきりAIがなんとなく雰囲気読んでくれると思ってたんですけど〜。つまり「この色番号使って！」って具体的に書かないとダメってこと？

**橋本**: まさにそうです。形容詞は人間同士でも解釈が分かれます。AIには尚更そう。

## Turn 2｜選択肢の発散：何をテンプレートに入れるか

**橋本**: テンプレートに追加できるデザイン情報の候補を、優先度と共に整理します。

| カテゴリ | 内容例 | 優先度 |
|---|---|---|
| カラーパレット（hex + Tailwindクラス） | Primary/Accent/Neutral定義 | 🔴 必須 |
| UXフェーズ別カラールール | 共感=暖色、分析=クール、など | 🔴 必須（COCOSiL固有） |
| ニュートラル禁止ルール | `gray-*` 混入防止 | 🔴 必須 |
| タイポグラフィ（逸脱ルールのみ） | `font-semibold` / `leading-relaxed` | 🟡 推奨 |
| スペーシング基準 | 8px grid、`rounded-xl` 固定など | 🟡 推奨 |
| シャドウ | card shadow定義 | 🟢 任意 |

特に**UXフェーズ別カラールール**はCOCOSiL固有です。「共感フェーズのコンポーネントに分析フェーズのクールな色を使うな」という制約は他のプロダクトにはない。これを明示しないと、AIは全コンポーネントをブランドカラー一色で塗ります。

**松田**: タイポグラフィについては、Tailwindにはデフォルトスケール（`text-sm`, `text-base`, `text-lg` 等）があるので、逸脱ルールだけ書けばいいと思います。「Tailwindデフォルト＋逸脱ルールのみ記載」というアプローチで、テンプレートが重くなりすぎるのを防げます。

**みさき**: 「分析フェーズがクールな色」って発想、めっちゃおもしろい〜！確かに、悩みを打ち明けたのにいきなり冷たいデザインで分析結果来たらなんか距離感感じそう。共感フェーズは暖かい感じがいい！

## Turn 3｜COCOSiL固有の要件：感情フローと色の対応

**みさき**: 実際のユーザー目線で聞いていいですか？「共感→安心→分析→行動」って順番らしいですけど、それぞれどんな気持ちで見てると思います？私が悩み話してて「わかってもらえた」感は、なんかほっこりした暖かい感じなんですよね〜。で、診断結果は「ちゃんとした根拠があるんだ」って信頼できる感じがほしい。

**橋本**: みさきさんの感覚は正確です。これを原則化すると：

| UXフェーズ | 心理状態 | カラー方向性 | NGパターン |
|---|---|---|---|
| 共感（Empathy） | 「わかってもらえる」 | 暖色系（Amber/Warm） | 寒色・ビビッド |
| 安心（Safety） | 「ここは安全」 | 白〜薄いスレート | 強調色・警告色 |
| 分析（Analysis） | 「根拠がある」 | クール（Slate/Indigo） | 暖色・エンタメ系 |
| 行動（Action） | 「やってみよう」 | アクセント（Orange CTA） | くすみ過ぎ・暗い |

**松田**: 実装観点で補足すると、フェーズ別の色をCSSトークン名に反映させるとさらに強い。`--color-empathy`・`--color-analysis` のように。Tailwindの `@theme` ブロックに定義しておけば、テンプレート指示が「`bg-empathy` を使え」で済む。AIが勝手にhex値を変えるリスクが消えます。

**橋本**: ただ、CSS変数の定義はコードベース側の仕事で、今回のテンプレートはAI生成指示書。テンプレートにはhex＋Tailwindクラスの対応表を置いて、生成指示時に「フェーズに応じてこの表の色を使え」と書く形で十分です。

**みさき**: 「bg-empathy」って書き方、コード知らなくても意味わかる〜！これ超いい！

## Turn 4｜具体的なカラーパレット：ブランド制約を踏まえた設計

**橋本**: 具体的なパレットを提案します。COCOSiLの制約（スピリチュアルでない・科学的・25〜35歳向け）を踏まえると：

```
■ ブランドカラー
Primary:       #1E293B  slate-800   — 誠実・権威・深み（見出し・ボタン）
Primary-light: #334155  slate-700   — やや柔らかい誠実さ（サブ見出し）
Accent:        #6366F1  indigo-500  — モダン・信頼感・CTA以外の強調

■ UXフェーズ別カラー
Empathy:       #F59E0B  amber-500   — 暖かさ・寄り添い（共感フェーズ）
Safety:        #F8FAFC  slate-50    — 静けさ・安全（安心フェーズ）
Analysis:      #475569  slate-600   — 落ち着き・論理（分析フェーズ）
Action:        #F97316  orange-500  — 行動喚起・前進（行動フェーズ・CTAボタン）

■ ニュートラル
Surface-1:     #F8FAFC  slate-50    — ページ背景
Surface-2:     #F1F5F9  slate-100   — カード背景
Text-primary:  #0F172A  slate-900   — 本文テキスト
Text-secondary:#64748B  slate-500   — サブテキスト・補足
Border:        #E2E8F0  slate-200   — 区切り線・カード枠

■ 状態色
Success:       #0D9488  teal-600    — 完了・ポジティブ
Error:         #DC2626  red-600     — エラー
Warning:       #D97706  amber-600   — 警告
```

スピリチュアル感が出やすい紫系（violet / purple）は意図的に外しました。Indigoは見た目が青寄りで「科学的」の印象を維持できます。

**松田**: このパレットに加えて、ニュートラルの禁止ルールを1つ追加してほしい。「`slate-*` のみ使用、`gray-*`・`zinc-*`・`stone-*` は混入禁止」というルール。これらはほぼ同じに見えて色温度が微妙に異なり、混在するとデザインがじわじわ崩れます。私が関わった案件で、Slateで統一するだけで「なんかバラバラ感がなくなった」という事例が複数あります。

**橋本**: 完全に同意。私が支援した20社超で最も頻繁に発生するデザイン乱れの原因がこれです。ルール一行で防げる。

**みさき**: gray と slate って違うんですか！？超気づかなかった〜！でも「混入禁止」って言われたらAIも守ってくれそう。ルール一行で防げるの、めっちゃコスパいい。

## Turn 5｜統合：テンプレート書き換え方針と具体的次の一手

**橋本**: 議論を踏まえ、「COCOSiL デザイントークン3原則」を整理します：

**① Color as UX Signal（色はUXの信号）**
フェーズ（共感/安心/分析/行動）に色を対応させ、コンポーネントはフェーズ色に従う。装飾目的のアクセント多用は禁止。

**② Slate-Only Neutral System（スレート統一）**
ニュートラルは `slate-*` 系のみ。`gray-*`・`zinc-*`・`stone-*` は混入禁止。この一つで一貫性の90%が保たれる。

**③ Token-First, Not Hex-First（トークン名で指示）**
AIへの指示はトークン名（`bg-primary` / `text-secondary` / `bg-empathy`）で行う。hex値を直接書かせない。

**松田**: 具体的な追記内容は以下の通りです：
- ブランドカラーパレット（8トークン）
- UXフェーズ別カラー（4フェーズ × hex + Tailwindクラス + 心理的役割）
- 状態色（3色）
- カラー禁止ルール
- タイポグラフィ逸脱ルール
- スペーシング基準（8px grid、`rounded-xl` 固定）

**橋本**: これを入れることで、AIは「誠実な色を毎回自分で解釈する」ことをしなくなります。10コンポーネントを生成しても、すべてが同じトークン体系に乗る。

**みさき**: なんか最初「カラーコードを追加したいだけ」と思ってたんですけど、共感フェーズには暖かい色・分析フェーズにはクールな色って対応させる発想がめっちゃよかった〜！COCOSiLの「共感→安心→分析→行動」の順番がデザインにもちゃんと反映されてるって、超一貫性あって好きです。これ見て作られたUIなら信頼できそう！
