## Turn 1｜現状診断 — 「3層の哲学が美しすぎて、層間の翻訳経路が文書任せ」

**板垣 涼介：**
正直に言います。COCOSiL V2を読み込んで最初に出た所感は「哲学のレイヤーは芸術品レベル」ですが、構造上の致命的な特徴が1つあります——**3層構造（① 設計中枢=Layer 0 / ② Autogenesis Constitution / ③ ハーネス実装）がぜんぶ「文書」で繋がっている**こと。

具体的に数えました。設計思想を定義する文書は `COCOSiL設計中枢.md`（425行）+ `AGENTS.md` §7（約120行）+ `HARNESS_DECISIONS.md`（381行）+ `language-design-v1.md`（177行）+ 議論ログ26本。総量 **1万行以上のテキストでガバナンスを敷いている**状態です。

一方、これらの思想を実装に強制する**機械可読定義はゼロ**。`prevent-destructive-command.js` の `DESTRUCTIVE_PATTERNS` 配列（12パターン）だけが唯一機械が読めるルール。Autogenesis Constitution の「絶対不変リスト」（Policy / UXシーケンス / 言語設計 / 評価指標定義）も、Mutableリストも、すべてMarkdownテキスト。

**抽象的診断**：これは結局**「Constitution as Comment」問題**です。憲法は書かれているが、コードがそれを参照する経路がない。Phase C（週次Autogenesisループ）が立ち上がったとき、AIが生成したプロンプトが「UXシーケンス順序を守れているか」を判定する基準は、人間がMarkdownを読んで判断するしかない構造になっています。

---

**谷口 真理：**
板垣さんの診断に同意しつつ、**別角度から根本矛盾**を出します。

設計中枢運用議論ログ（2026-05-03）の Turn 5 で確立された3原則の第一が「**Documents Die, Vocabulary Lives.（文書は死ぬ。語彙は生きる）**」です。文書への参照ではなく口語使用が生命線、と明言されている。

ところが、その議論ログ自体を含めて、現状COCOSiL V2は**「文書を増やすことで設計を担保する」アプローチ**を実装し続けています。`docs/discussions/` には26本の議論ログが堆積しており、今この瞬間も新しい議論ログが生成されようとしている。**自己原則の第一原則と、運用実態が、最も大きく乖離している**——これは設計の自己矛盾です。

加えて、私がメンタルテックプロダクトで何度も見てきた構造的リスクが1つ：**Bus Factor = 1 問題**。設計中枢の解釈権、Layer 2コンテンツ層の承認権、Autogenesisの「Approve判断」、アトミック確認ループの「動作確認」、言語設計の更新権限——**5つすべてが「えんまさ1人」に集中している**。`HARNESS_DECISIONS.md` §6 では、Bootstrap PR問題の対処として「えんまさのAdmin bypass」を採用したが、これは **Bus Factor=1 を構造化**した結果でもあります。

> 🔍 **抽象的診断**：COCOSiL V2の整合性問題は2つに集約される。① **Constitution as Comment**（思想の機械可読定義の不在）、② **Vocabulary's Death by Document Inflation**（語彙化されるべき思想が文書増殖に逃げている）。Bus Factor=1 はこの2つの帰結。

## Turn 2｜矛盾の3軸整理 — どこから直すか

**板垣 涼介（評価軸の提示）：**
設計思想と実装ハーネスのドリフトを、私は **3つの軸**で整理します。これがCOCOSiL V2の「整合性ハーネス」を再設計する判断軸です。

| 軸 | 定義 | チェック観点 |
|---|---|---|
| **軸1：定義 vs 実装の距離** | 思想を定義する文書と、それを強制する機械の距離 | 思想がコード/設定/CIで参照されているか |
| **軸2：判定の独立性** | 同じ問いに対して複数の独立判定が走るか | 1人の判定で済んでいないか |
| **軸3：閉ループの完結度** | 「観察 → 評価 → 改善」サイクルが閉じているか | 実装したが運用ループが回っていない部分はあるか |

**谷口 真理（具体的マッピング）：**
板垣さんの3軸に、私が読み込んで発見した**整合性ギャップ8件**をマッピングします。

| # | ギャップ | 軸1 距離 | 軸2 独立性 | 軸3 閉ループ |
|---|---|---|---|---|
| **C1** | Autogenesis Constitution「絶対不変リスト」が文書のみ。コードから参照不能 | 🔴 大 | — | — |
| **C2** | PR template の設計中枢チェック5問を**実装者単独で記入**する形式 | — | 🔴 ゼロ | — |
| **C3** | `/start-task` の5問判定 と PR template の5問判定の**関係が未定義**（重複か独立判定か） | 🟡 中 | 🔴 ゼロ | — |
| **C4** | 禁止語彙チェック（言語設計v1）が**機械化されていない**。F1/F2のプロンプトが既に存在しているが回帰検出経路ゼロ | 🔴 大 | — | 🔴 不在 |
| **C5** | Phase A 観察基盤のmigrationとtelemetry実装はあるが、**APIルートからのイベント送信箇所が未挿入**＝データが流れない | — | — | 🔴 不在 |
| **C6** | PMF基準「7日再訪率30%」と Autogenesis評価指標「再言語化率・矛盾受容率」の**関係が未定義**。Sycophancy（迎合）が両指標を逆走させるリスク | 🟡 中 | — | 🔴 不在 |
| **C7** | 「えんまさ動作確認」がアトミック確認ループの第3条件＝**全PRのマージにえんまさ必須**。Bus Factor=1構造化 | — | 🔴 ゼロ | — |
| **C8** | 設計中枢運用落とし穴ログの「**割れ体験を作る**」というアクションの**実装メカニズム不在**。割れた事例を集計・通知する仕組みなし | — | 🔴 ゼロ | 🔴 不在 |

**抽象的軸**：致命度は「3軸すべてが赤」がトップ。**C2（PR template単独記入）**と **C8（割れ体験のメカニズム不在）**が絡み合って、設計中枢が「形骸化」する経路を作っています。

> 🎯 **抽象的優先度原則**：「Must条件を守る機構」の整合性を最優先で固める。Should条件・運用負荷の最適化は後。

## Turn 3｜最致命矛盾の深掘り — 自己審査の失効と「割れ体験」の不在

**💅 みさき（実需持ち込み）：**
ちょっと**C2のPR template** がほんとに気になるんですけど〜。私エンジニアじゃないけど、**ヒラメさんが自分でAPIを書いて、自分で「Q1〜Q5全部○です」って書く**んですよね？それって「俺は自分のコード合格点だ」って自分で言ってるだけじゃないですか？それでマジでチェックになるんですか？

**谷口 真理（パターン化）：**
みさきさん、認知心理学の核心を突いています。これは「**自己審査の失効問題**」と呼ばれる現象で、設計中枢運用ログの Turn 3 で柴崎氏（架空）も指摘していた論点です。

データを再掲します：
- **自己審査による「本来否定されるべき提案」の通過率：71%**（確証バイアス研究の累積メタ分析）
- 残り29%は「明らかに不適合」のケースのみ。微妙なケースはほぼ通過する。

そして、構造的に重要なのは——**現状のCOCOSiL V2のチェック構造が、この失効問題を3重に作っている**点です。

```
チェック1（実装前）: /start-task の Step 2.5 で AIが5問判定
  → 自己（実装着手者）に対する自己判定

チェック2（実装中）: /cocosil-work で担当ガード確認
  → 自己（実装者）が自己のスコープを確認

チェック3（実装後）: PR template に Q1〜Q5記入
  → 自己（PR作成者）が自己のコードを判定

レビュアーは CODEOWNERS に基づくが、5問は記入済み欄を「読む」だけ
```

**3回チェックがあるのに、判定者は全部同じ人**。設計中枢運用議論で言われた「えんまさとヒラメの独立判定 → 差分を見る」フローは、一度も発動しない構造になっています。

**板垣 涼介（メカニズム解剖）：**
これを **PMマトリクス**でさらに分解できます。設計中枢のリトマス試験紙が「機能する使い方」と「形骸化する使い方」の対比表（同議論ログより）：

| 使い方 | COCOSiL V2の現状 | 機能するか |
|---|---|---|
| 実装者が自己チェック | ✅ /start-task + PR template が両方これ | ❌ 確証バイアスで71%通過 |
| PRレビュー時に判定理由を1行書く | △ PR template はあるが、書く人と判定する人が同じ | △ 部分的 |
| えんまさが週次抜き打ち | ❌ 仕組みなし | — |
| えんまさとヒラメが独立判断 | ❌ 仕組みなし | — |
| 月次ドリフトチェック | △ HARNESS月次レビューはあるが、5問判定の差分集計はない | △ 部分的 |

**機能する4つの使い方のうち、現状実装されているのは「自己チェック」（無効化形態）のみ**——これは本当に致命的です。

> 🎯 **抽象的要件原則**：リトマス試験紙の本質的価値は「全員Yesを得る」ではなく「**判定の差分を可視化する**」点にある。差分が出ない=機能していない、と見るべき。

---

**💅 みさき：**
あ〜！つまり**「全員ハンコ押した」より「ここで意見が割れた」って事実の方が、設計中枢が生きてる証拠**ってこと？でもそれ、現状仕組みがないんですよね？あと、えんまさが全PRに「動作確認しました」ってチェック入れる必要ある（C7）のなら、結局えんまさが**全コードを見ることが仕事**になっちゃって、**判定の独立性も死ぬ**じゃないですか？「実装者だけ判定」より「えんまさが全部判定」の方がもっとヤバくない？

**谷口 真理：**
みさきさんの再指摘、極めて重要です。**C2 と C7 は連結している**。
- C2 を解消するために「えんまさにレビュー集中」させると、C7 を悪化させる
- C7 を解消するために「えんまさのレビューを減らす」と、C2 が悪化する

**この2つを同時に解く設計**が必要です。それが**「機械の床、人間の門」の Two-tier 化**——機械可読Constitutionが Layer 1 で大量チェックを処理し、えんまさは「機械が判定不可能なケース」だけに集中する構造。

## Turn 4｜技術深掘り — ハーネス設計の不変法則

**板垣 涼介（法則の抽出）：**
ここまでの議論から、COCOSiL V2の整合性ハーネスを支配する**3つの法則**を抽出します。

```
法則1: Constitution must be Code, not just Comment.
  思想を文書だけで定義すると、実装が思想を参照する経路が
  「人間が思い出す」しかなくなる。Phase C のAutogenesisループは
  「変更案がConstitution違反でないか」を機械判定できねば成立しない。
  → lib/constitution/ をシングルソース・オブ・トゥルースにすべき。

法則2: Self-check is for thinking, not for verification.
  自己チェックの価値は「書く行為が思考を強制する」点。
  「合格判定の根拠」にはならない（71%通過の研究）。
  → PR template の5問は「実装者欄」と「レビュアー欄」を物理分離し、
    両者が一致しないPRをCIが警告する設計が必要。

法則3: Observation without circulation is dead data.
  Phase A 観察基盤の migration / events.ts を実装しただけでは
  Autogenesis議論で診断した「観察の不在 = データは死んだ資産」と
  同型の症状が再生産される。
  → APIルートから capture 関数が呼ばれるまでが Phase A の完了。
    現状はゴール手前で止まっている。
```

**谷口 真理（COCOSiL固有のリスク追加）：**
板垣さんの3法則に、**COCOSiL固有のSycophancyリスク**を加えます。

PMF基準は「7日以内再訪率30%以上」。これは行動指標として正しいKPIです。しかし、Autogenesis議論ログ Turn 3 で吉田氏が指摘した「**自己理解の深化は三毒の逆指標**」——心地よい錯覚（貪：依存）が再訪率を押し上げる構造があります。

具体例：
- ユーザーAが「癒される」と感じている → 7日以内に何度も戻る → 再訪率↑
- ユーザーBが「腑落ちして少し不快になった」（痴の解消）→ 1週間内省 → 再訪率↓
- 表面的にはAの方が「PMF達成」に貢献しているように見える
- でも設計中枢的にはBが「無明を晴らした」状態

**つまり、PMF基準と内省指標の両方を「同時に」見ない限り、Autogenesisが「再訪率最大化」=「Sycophancy最大化」の方向に最適化してしまう構造的リスク**があります。法則4を追加します：

```
法則4: PMF metric and Reflection metric must be measured together.
  7日再訪率↑ AND 内省スコア↑ = 真のPMF
  7日再訪率↑ AND 内省スコア↓ = Sycophancy疑惑（要警戒）
  片方だけ見るのは設計中枢違反。
```

---

**💅 みさき：**
あー！分かった、つまり：
- **機械が判定できることは機械にやらせる**（禁止語・型・UXシーケンス順序とか）
- **判定が割れる可能性があるところは、最初から2人に判定させる**（実装者とレビュアー、または機械と人間）
- **データを集めるだけじゃダメで、流して見て改善するまで一周しないとデータは死ぬ**
- **再訪率と内省スコアは絶対セットで見る**

ってことですよね？なんか聞いてると、**COCOSiL自身がCOCOSiLのユーザーになる**みたいな感じですね〜（Autogenesis議論ログで言ってたやつ）。**自分のシステムを自分で観察して、反応じゃなく観察モードで判断する**——三毒（自分の設計の自己愛・恐怖・混乱）を増幅しないハーネス、ってことじゃないですか〜？

**板垣 涼介：**
みさきさん、それはもう**Turn 5 のサマリー**をすでに先取りしています。仏教用語で言うと「**ハーネスの止観構造**」——システムが自身の振る舞いを「止」めて、構造を「観」る装置。

> 🔑 **抽象的法則**：COCOSiL V2のハーネスは、それ自体が**設計中枢の3原則 ① Dispel ② From Reaction to Reflection ③ Self-Knowing for Better-Relating** を体現しなければならない。**ハーネスがハーネスのユーザーになる構造**——これが整合性の最終条件。

## Turn 5｜統合 — ハーネス再設計の3原則 + 明日からの具体アクション

**谷口 真理（設計原則）：**

---

### 【整合性ハーネスを生かす3原則】

> **① Constitution as Code, Not as Comment.（憲法はコードであり、コメントではない）**
> Autogenesis Constitutionの「絶対不変リスト」（Policy / UXシーケンス / 言語設計 / 評価指標定義）を `lib/constitution/index.ts` に **machine-readable な定数・型・スキーマ**として定義する。Phase C のAutogenesisループは、ここを `import` してチェックする。文書 `AGENTS.md §7` は「`lib/constitution/index.ts` を正とする」と明示し、ドリフト時はコードを正とする。

> **② Independent Judgment by Default.（独立判定をデフォルトにする）**
> PR template の5問チェックは「**実装者判定列**」と「**レビュアー判定列**」を物理分離する。両者が一致しないPRをCIが警告（マージはブロックしない、警告のみ）。`/start-task` 判定は「**初期見立て**」、PR template判定は「**実装後の再判定**」として役割を区別する。設計中枢運用議論の「割れることが機能の正常作動」を構造化。

> **③ Living Loop, Not Living Document.（生きた文書ではなく、生きたループを）**
> 「観察 → 評価 → 改善」の閉ループが回っていない実装は「死んでいる」と見なす。Phase A の `events_telemetry` migration が完了し、`lib/telemetry/events.ts` も実装済みなら、**APIルートからの送信** + **PostHog ダッシュボードの定例監視** + **改善仮説への接続**まで完了して初めて「Phase A完了」と定義する。途中段階の実装は「Phase A 30%完了」のように分母を明示する。

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| **整合性問題の根本診断** | ① Constitution as Comment（思想の機械可読定義不在） ② Vocabulary's Death by Document Inflation（語彙化されるべき思想が文書増殖に逃げている） |
| **特定された矛盾・抜け穴** | C1〜C8（うちCritical = C2 PR template自己審査 / C4 禁止語彙未機械化） |
| **設計原則①** | Constitution as Code, Not as Comment — `lib/constitution/` を単一の真実に |
| **設計原則②** | Independent Judgment by Default — PR templateを2列分離・差分検知の自動化 |
| **設計原則③** | Living Loop, Not Living Document — 観察→評価→改善が回らない実装は「死んでいる」 |
| **隠れたリスク（法則4）** | PMF再訪率と内省スコアは必ずセットで見る。Sycophancy防止は両指標の同時測定が必須 |
| **今週の最優先3アクション** | ① `lib/constitution/` 新規作成 ② PR template 2列分離 ③ `lib/prompts/__tests__/` 即時導入 |
| **C7 Bus Factor=1問題** | Constitution as Codeで機械判定範囲が拡大 → えんまさのレビューはコア判断のみに集中 → 自然に薄まる |
| **メタ原則** | ハーネスがハーネスのユーザーになる構造（設計中枢の3原則をハーネス自身が体現する） |
