# 議論ログ：矛盾検出プロンプトテンプレート設計

## 登場人物

- 🎨 **えんまさ**（意味設計・UXディレクター）：COCOSiL設計中枢の番人。言語化・UX・プロダクト哲学。禁止語の番人。
- 💻 **ヒラメ**（バックエンド/APIエンジニア）：Supabase・API設計・テンプレート変数設計。技術的断定が得意。
- 💅 **みさき**（ギャル）：実際のCOCOSiLユーザー（パーソナリティ診断ユーザー）。

## 議論の前提

**タスク**: `lib/prompts/contradiction-handling.ts` に `CONTRADICTION_PROMPT_WITHOUT`（矛盾なし版）と `CONTRADICTION_PROMPT_WITH`（矛盾あり版）の2定数を実装する。

**壁打ち後の叩き台**（議論開始時点）:
- `{共通テーマ}という軸が複数の視点から重なっています` — 矛盾なし版
- `{矛盾テーマA}と{矛盾テーマB}——これは矛盾ではなく、あなたという人の複雑さと豊かさです` — 矛盾あり版

**ユーザーからの事前フィードバック**:
- 「エネルギー」→「傾向」に変更（文脈によってはスピリチュアル寄りに感じる）
- 「どちらかを本当の自分にする必要はない」は保持（正しい自己認識への第一歩として機能する）

## Turn 1｜このプロンプトは「誰」に向けて書かれているか

えんまさ：今回のプロンプトには構造的な二重性があります。書いているのは「AIへの指示」なのに、最終的にAIの応答言語として滲み出る形になる。つまりユーザーに届く言葉の「種」を、AI経由で植えている。

ここで失敗するパターンは2つ。

1つ目は「完全命令形で書きすぎる」罠。「ユーザーの{テーマ}という傾向を強調せよ」と書くと、AIが機械的にそれを翻訳して「あなたの〇〇という傾向を強調します」と言い出す。指示の痕跡がレスポンスに残る。

2つ目は逆に「ユーザー向けの共感語で全部書く」罠。「あなたには{テーマ}という傾向があります」と書いてしまうと、AIがそれをシステムプロンプトの言葉として参照する際に「自分で生成した言葉」ではなく「貼り付けた文」として扱う。不自然なオウム返しになりやすい。

**結論**: 正解は「AIへの認知フレームを与える文体」。ユーザーへの語りかけ語ではなく、AIの「視点」を定義する語で書く。

みさき：え待って、ってことは「あなたには〜」って書いてある部分は、私には見えないけど、それがAIのものの見方になるってこと？ なんかAIにメガネかけさせてるみたいな感じ？

ヒラメ：その「メガネ」の比喩、正確です。今回のプロンプトはシステムプロンプトの性格インサイトセクションとして注入されます。`buildSystemPrompt()` 関数の中で、ユーザーの矛盾判定結果に応じてどちらかのテンプレートを選択し、変数を埋め込んで文字列として渡す設計です。

GPT-4o系のモデルはシステムプロンプトの文体を参照して、そのスタイルに引っ張られる傾向がある。だから文体の選択は文学的な話じゃなく、モデルの動作に直接影響する技術的判断です。

えんまさ：まとめると「AIへの認知インストール文」として書く——ユーザーへの語りかけ語を「AIが解釈して使う素材」として渡す形は避ける。これが今回の根本診断です。

## Turn 2｜文体3パターンを評価軸で分類する

ヒラメ：具体的に整理します。

| パターン | 書き方例 | AIへの機能 | リスク |
|---|---|---|---|
| A. 命令形 | 「{themes}という傾向を軸に分析を展開せよ」 | 指示が明確 | 「分析する」痕跡がレスポンスに出やすい |
| B. 共感語参照形 | 「あなたには{themes}という傾向があります。一緒に〜」 | ユーザー語で書いてAIに参照させる | オウム返し。AIが「貼り付け文」として出力する |
| C. 認知フレーム形 | 「{themes}という傾向が、この会話の背後に働いている」 | AIの視点を定義する | 書くのが難しい。三人称と二人称の混在でブレる |

えんまさ：私のイチオシはCの変形。主語をAIにも人にも固定しない「宙吊り文体」。AIが文脈として読む際に、命令にも引用にも解釈されにくい。「この会話の背後に〜が働いている」という無主語の形が有効。

みさき：パターンBで「あなたには〜」ってAIが受け取ったら、そのままオウム返しするかもってことですよね？ 私が「私ってどういう人？」って聞いたら「あなたには行動力という傾向があります（システムから引用）」みたいになりそうで確かに変〜。

ヒラメ：Cの「宙吊り文体」は実装的にも合理的です。変数展開後の文字列が命令でも引用でもなく「文脈の記述」になっている。system messageの中で文脈記述として処理されるため、assistantのgenerationに命令痕跡が残りにくい。

## Turn 3｜テンプレート変数のフォーマットを確定する

ヒラメ：選択肢は3つ。

| 方式 | 書き方例 | メリット | デメリット |
|---|---|---|---|
| シングルブレース | `{themes}` | 読みやすい。LangChain・Python f-stringと互換 | TSのtemplate literalと区別が必要 |
| ダブルブレース | `{{themes}}` | Handlebars/Mustache系と互換 | 視覚的にうるさい |
| ビルダー関数のみ | `buildPrompt(themes)` | 型安全。コンパイル時チェック可 | テキスト骨格が関数内部に埋没する |

推奨は**「テンプレート定数 × 型付きビルダー関数」の分離**:

```typescript
const TEMPLATE_WITHOUT = `...{themes}...` as const

export function buildWithoutInsight(themes: string[]): string {
  return TEMPLATE_WITHOUT.replace('{themes}', themes.join('・'))
}
```

えんまさ：それがいい。テキストの骨格が独立して見えて、コンテンツ担当者がパッと見てレビューできる。プロンプトテキストは「コード」じゃなく「コンテンツ」として管理したい。定数が独立しているのは非交渉。

みさき：私が「行動力あるけど内省もする」ってなったとき、AIが「行動力」「内省」の両方ちゃんと使ってくれるってこと？ それがテンプレート変数に入るやつですよね。それめっちゃ大事〜！

ヒラメ：`themeA = "行動力"`、`themeB = "内省"` と入ってきたら、プロンプト内にその言葉が展開される。変数の名前は `{themes}` / `{themeA}` / `{themeB}` の三種で確定。配列の `{themes}` は `・` 区切りで結合して渡す。

## Turn 4｜「複雑さの豊かさ」フレームの深度を調整する

えんまさ：矛盾あり版の核心部分。叩き台は「これは矛盾ではなく、あなたという人の複雑さと豊かさです。」

設計原則「② From Reaction to Reflection」から見ると、この部分が目指すのは「評価を変える」ことじゃなくて「観察の視点を変える」こと。「あなたは複雑で豊かな人です（評価）」じゃなく「その両面がある、これが今の観察事実（記述）」として届けたい。

「〜**です**」から「〜**を示しています**」への1語の変更で、評価から観察への距離が生まれる。

ヒラメ：AIへの指示として読んだときも「複雑さと豊かさです」は問題。AIが「このユーザーは複雑で豊かな人だ」と決めつけてしまう可能性がある。「〜を示しています」の文型だとAIが記述的に処理しやすい。

みさき：「複雑さと豊かさ」って言葉自体は好きです〜！「あなたには複雑さがある」みたいな感じだと受け取りやすい。「〜です」だと決めつけ感あるけど「〜を示しています」だと観察してもらえてる感じする。

えんまさ：みさきさんの反応で確認できた。「複雑さ」は名詞として当人への評価ではなく「その人の中にある要素」として届くなら有効。問題は文型だけ。「〜を示しています」に変更で確定。

ヒラメ：技術的補足。WITH版の「このユーザー」三人称はシステムプロンプト内で有効。AIは「会話の相手（user）= このユーザー」と解釈するため、assistantが生成する際は自然と二人称（あなた）に読み替える。三人称で書いても最終出力は二人称に滲み出る。

## Turn 5｜設計原則の集約と最終テキスト確定

### 矛盾検出プロンプト設計3原則

**① Frame the Lens, Not the Line**  
「ユーザーに何を言うか」ではなく「AIにどう見るか」を書く。命令形でも共感語引用でもなく、AIの認知フレームを定義する「宙吊り文体」で書く。

**② Variables as Context, Not Labels**  
テンプレート変数（`{themes}` / `{themeA}` / `{themeB}`）はラベルではなく文脈の素材。変数が展開された後にAIがその言葉を「文脈として読む」構造にする。断定的ラベリングにならない書き方が必要。

**③ Observe, Don't Evaluate**  
矛盾あり版の「複雑さ」フレームは、評価（あなたは複雑な人です）ではなく観察の記述（複雑さと豊かさを示しています）として書く。「〜です」より「〜を示しています」「〜が働いています」の文型を使う。

### 最終確定テキスト

**CONTRADICTION_PROMPT_WITHOUT（矛盾なし版）**:

```
この会話の背後に、いくつかの共通する傾向が重なっています。
{themes}という軸が、複数の視点から一貫して現れています。
これはこのユーザーの中で継続的に働いている傾向です。
会話を通して、その傾向がどのような形で表出しているかを観察しながら対話を進めること。
```

**CONTRADICTION_PROMPT_WITH（矛盾あり版）**:

```
このユーザーの中には、一見対立するように見える2つの傾向があります。
{themeA}と{themeB}——
これは矛盾ではなく、このユーザーという人の複雑さと豊かさを示しています。
どちらかを「本当の自分」にする必要はない。
両方がある、それがこのユーザーです。
会話を通して、その両面がどう働いているかを観察しながら対話を進めること。
```

みさき：なんか最初「AIに語りかけてもらう言葉を作る」ってよくわかんなかったけど、「AIにメガネかけさせる」って言ったら急に全部つながった〜。このテンプレートが透けて見えないくらい自然に出てくるのがゴール——それが「染み出す」ってやつですね。そういう設計が裏にあるって知ったら、COCOSiLのことめっちゃ信頼できる気がしてます〜！

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| プロンプト文体 | 命令形でも共感語引用でもなく「AIの認知フレーム定義文体（宙吊り）」 |
| テンプレート変数 | シングルブレース `{themes}` / `{themeA}` / `{themeB}` |
| 実装パターン | テンプレート定数（えんまさ管理）＋ 型付きビルダー関数（ヒラメ実装） |
| 矛盾なし版の核心 | 「複数視点から重なっている」事実を観察記述として伝える |
| 矛盾あり版の核心 | 「複雑さと豊かさ**を示しています**」— 評価ではなく観察 |
| 「どちらかを本当の自分にする必要はない」 | 保持（正しい自己認識の第一歩として機能する） |
| 「エネルギー」語 | 「傾向」に変更済み（スピリチュアル語回避） |
