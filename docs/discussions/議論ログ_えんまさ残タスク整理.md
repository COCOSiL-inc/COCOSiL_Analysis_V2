## Turn 1｜現状の棚卸し——えんまさのタスクは「何層」あるのか

**前田：** まず整理しましょう。現在のリポジトリを見ると、えんまさが担当すべき作業は3つの性格がまったく異なるレイヤーに散らばっています。

**① プロダクトテキスト設計（コンテンツ生成）**：
F1.4の哲学提示テキスト、F2.4の4体系根拠テキスト、F3.2「安心」フェーズのAIサブテキスト、F3.3しっくりきたマーカーのUIテキスト——これらは「原稿を書く」作業です。

**② AIプロンプト設計（構造的設計）**：
F4の5フェーズチャットプロンプト（Phase 1〜5）、F6シェアカードのタイプ名生成プロンプト——`lib/prompts/`に入るコードとして設計するものです。現時点で`onboarding.ts`と`contradiction-handling.ts`は実装済み。ただし**F4の5フェーズプロンプトはまだ存在しない**。

**③ ハーネス判断・承認作業（ガバナンス）**：
`harness-redesign-proposal-2026-05-05.md`の「えんまさのレビュー待ち」ステータス、`lib/prompts/__tests__/`の禁止語彙テスト導入（Action 3）の起動判断、毎週のKPIモニタリング。

この3層が混在しているのが現状です。「コンテンツを書く人」と「AIシステムを設計する人」と「意思決定をする人」がえんまさ1人に集約している——これが診断です。

**早瀬：** 前田さんの診断に補足すると、②の「5フェーズチャットプロンプト」は単なる原稿執筆ではなく、**フェーズ遷移の判定ロジックをプロンプト内のシステム指示で制御する**という構造的設計を含みます。`onboarding.ts`の設計原則（「Blank Slate, Warm Door」「Scaffold the Openness」）を見ると、えんまさはすでにプロンプトをコードとして書く能力を示しています。しかし5フェーズは複雑度が1段上で、各フェーズの**入口条件・出口条件・状態変数**を定義しないとAIが適切に遷移を判断できない。これは設計仕様書+プロンプトコードの両方を同時に書く作業になります。

**みさき：** え待って、つまり今えんまさがやらないといけないことって「文章書く」だけじゃないってこと？コードっぽいものも？それめっちゃ大変じゃないですか〜。あと「フェーズ遷移の判定ロジック」ってつまり、チャットしてるときにAIが「このユーザーはもうフェーズ2に進んでいいな」って判断する部分ですよね？

**前田：** 正確です。みさきさんが言った通り。そしてそのロジックを「AIが自律判定できる言語で書く」のがえんまさの仕事——これが①と②の境界線です。

## Turn 2｜タスクの重み付け——どれが最大ボトルネックか

**早瀬：** 優先度を軸で整理します。**ヒラメとまあみが開発着手前**という前提を加えると、「えんまさのアウトプットを待たないと他の二人が動けない」タスクと「後から入れられる」タスクが分かれます。

**ブロッキングタスク（えんまさのアウトプット = 他二人の着手条件）：**

| タスク | なぜブロッキングか |
|---|---|
| **F4 5フェーズチャットプロンプト設計** | ヒラメのAPIルート実装の型・フェーズ状態管理の定義がこれに依存。まあみのチャットUI設計もPhaseの状態数が確定しないと画面設計できない |
| **F3.2「安心」フェーズ AIサブテキスト** | まあみのUI実装で「テキストの長さ・改行位置・表示タイミング」が変わる。原稿なしでUIを作ると後から崩れる |
| **F1.3 ウェルカム対話の最終確認** | `onboarding.ts`は実装済みだが、えんまさが「これで動作確認OK」と言わないとF1はクローズできない |

**ノンブロッキングタスク（後から入れられる）：**
- F1.4 哲学提示テキスト（LPに追記するだけ）
- F2.4 4体系根拠テキスト（コンポーネントの「？」アイコンの中身）
- F3.3 しっくりきたマーカーUIテキスト
- F6 タイプ名生成プロンプト（Phase 2以降）

**ガバナンス作業：**
- `harness-redesign-proposal`のレビュー承認——これは実は**今すぐできる半日作業**で、承認すれば`lib/prompts/__tests__/`の禁止語彙テストが導入でき、以降のプロンプト変更が機械的に守られる。つまり**承認が早いほど自分のプロンプト設計ミスが自動検知される**。

**前田：** 早瀬さんの整理に加えて、F4の5フェーズプロンプトは**Phase 1（Day 1〜21）のクリティカルパスの中のクリティカルパス**です。5フェーズのうちPhase 1「傾聴」とPhase 2「共感」だけでも先に出すと、ヒラメが先行実装できる。全フェーズ完成を待つ必要はない。

**みさき：** あー！「全部できてから渡す」じゃなくて、「フェーズ1と2だけ先に渡す」もできるんですね〜。なんかそっちの方が現実的っぽい！てか5フェーズって結構多くないですか？私がAIとチャットしてて「今どのフェーズだ」とか意識する？

**早瀬：** みさきさんが気づいた通りで、ユーザー側にフェーズは見えません。AIの内部状態管理です。ただ、みさきさんの「フェーズを意識しない」という感覚がまさに正しい設計目標で——**会話の流れとして自然にフェーズが変わっていることを、えんまさのプロンプトが担保する**。これが難しいところです。

## Turn 3｜えんまさの仕事量の実態——何時間相当か

**前田：** 実作業ボリュームを正直に試算します。

| タスク | 推定作業時間 | 主な難しさ |
|---|---|---|
| F4 5フェーズプロンプト（Phase 1〜5） | **8〜16時間** | フェーズ遷移条件の言語化・各フェーズ内の応答原則設計 |
| F3.2「安心」フェーズ AIサブテキスト | **2〜4時間** | 「承認・脱判定化・招待」の3要素を100字以内で表現（超高密度） |
| `harness-redesign-proposal` レビュー | **1〜2時間** | レポートを読んで「承認/条件付き承認/差し戻し」を決定 |
| F1.3 `onboarding.ts` 動作確認 | **1〜2時間** | プレビューで動かして「腑落ち」できるか主観確認 |
| F2.4 4体系根拠テキスト（4体系×200字） | **2〜4時間** | 哲学的根拠を一般向けに翻訳する作業 |
| F1.4 哲学提示テキスト（200字以内） | **1〜2時間** | LPコピー設計（言語設計スキルを適用） |
| `lib/prompts/__tests__/` 禁止語彙テスト | **2〜3時間** | VitestでPrompt as Codeのテストを書く（AIと協働可） |

**合計推定：17〜33時間**——Phase 1の21日間でフルコミット3〜4日相当の意識的な時間確保が必要。

**早瀬：** F4の5フェーズプロンプトは**書いて終わりではなく、動かして反復改善する作業**が必ず発生します。プロンプト設計8時間 + 反復検証4〜8時間が現実的な数字。

**みさき：** え、やば！じゃあえんまさって結構タイトなんですね。てか「3人体制」って言ってるけど、えんまさが詰まったら全員止まっちゃう感じじゃないですか？`coherence-report`でも Bus Factor=1 って書いてありましたよね？なんかこれ設計的問題な気がするんですけど、整理すべきタスクの話と分けて考えた方がよくないですか？

**前田：** みさきさん、鋭い。「Bus Factor=1 構造の是正」は中長期課題として別議論。今日のゴールは「えんまさが次の2週間でやること」の視界を明確にすること。構造的問題は別途。

## Turn 4｜タスクの依存関係ツリー——どの順で着手すべきか

**早瀬：** 依存関係を技術的に整理します。ヒラメ・まあみが動けるようになるためにえんまさが先に完了すべき順序：

```
Week 1（Day 1〜7）：ブロック解除最優先

① harness-redesign-proposal レビュー承認（1〜2h）
   └→ lib/prompts/__tests__/ 禁止語彙テスト導入が可能になる
      └→ 以降のプロンプト設計が「テスト付きで安全に書ける」状態になる

② F4 5フェーズプロンプト Phase 1「傾聴」+ Phase 2「共感」先行リリース（4〜6h）
   └→ ヒラメが F4 API設計（フェーズ状態型定義）に着手できる
   └→ まあみが チャットUI骨格のプロトタイプを開始できる

③ F3.2「安心」フェーズ AIサブテキスト（2〜4h）
   └→ まあみが 安心フェーズ画面のUIデザインに着手できる

Week 2（Day 8〜14）：コンテンツ充填

④ F4 Phase 3「深掘り」〜Phase 5「行動提案」（4〜8h）
   └→ Phase 1/2実装中にヒラメが型定義を固める
   └→ 残フェーズを渡せばAPI実装を完結できる

⑤ F1.3 onboarding.ts 動作確認（1〜2h）
   └→ F1クローズ

⑥ 残コンテンツ（F2.4・F1.4・F3.3）は Day 8〜14 の隙間で処理
```

**この設計で法則として成立するのは：**
**"Unblock First, Fill Later"** — ブロッカーを最初に解除し、コンテンツ充填は後続に回す。

**前田：** `lib/prompts/__tests__/` の禁止語彙テストは**AIと協働で半日で書ける**タスク。`drift.test.ts`のパターンをほぼコピーして禁止語彙チェックを追加するだけ。ハーネス承認と同日に完了できます。

**みさき：** 「Phase 1と2だけ先に渡す」ってすごくいい発想ですね〜！「5フェーズ全部できてから」って思い込んでたかも。でもフェーズって途中から変えたくなることありません？例えば「傾聴フェーズの応答が機械的すぎる」ってなったら、API実装後に書き直せる？

**早瀬：** できます。プロンプトを `lib/prompts/` の `.ts` ファイルとして管理しているので、**プロンプトの中身の変更はAPIルートを変えずに独立して行えます**。ヒラメはフェーズのID（`phase: 'listening'`）に対して応答を返す構造にすればよく、えんまさはそのIDに対応するプロンプトを後から差し替えられます。これが Prompt as Code の強みです。

## Turn 5｜えんまさへの処方箋——3原則と具体的な次の一手

**早瀬：** 全議論を踏まえ、えんまさのタスク管理を支配する設計原則を3つ集約します。

### えんまさ作業の3設計原則

**① Unblock First, Fill Later.**
ヒラメ・まあみの着手を妨げるタスクを最優先。コンテンツの質より着手できる素材の存在が先。

**② Phase-Sliced Delivery, Not All-at-Once.**
5フェーズプロンプトは「全フェーズ完成」を待たず、Phase 1-2を先行リリース。API設計と並走する。

**③ Harness Review is an Investment, Not a Burden.**
`harness-redesign-proposal`の承認作業は負担ではなく、以降の設計ミスを自動検知するインフラへの投資。先に終わらせると後が楽になる。

**前田：** 明日からえんまさが取るべき具体アクションを5項目：

**Day 1（今日・明日）：**
1. `docs/output/decisions/harness-redesign-proposal-2026-05-05.md` を通読し、コメント付きで承認 or 差し戻し（1〜2時間）
2. F4 Phase 1「傾聴」プロンプトの初稿を `lib/prompts/chat-phase1.ts` として作成

**Day 2〜3：**
3. F4 Phase 2「共感」プロンプト `lib/prompts/chat-phase2.ts` 作成 → ヒラメに「Phase 1-2 Ready」を共有
4. F3.2「安心」フェーズ AIサブテキスト起草（承認・脱判定化・招待の3要素、100字以内）→ まあみに共有

**Day 4〜5：**
5. `lib/prompts/__tests__/banned-words.test.ts` を `drift.test.ts` パターンで作成

**みさき：** なんか「何から手をつけていいか分からない状態」が「明日これだけやる」に整理されてすっきりしました〜！えんまさってプロダクトの「魂」を設計してる人じゃないですか。その人が何をすべきか見えにくい状態ってもったいないですよね。てかPhase 1プロンプト初稿、AIと一緒に議論しながら書けますよね？`blank slateで温かく迎える`ってもう`onboarding.ts`に書いてあるから、そこから続きのフェーズへの橋渡しをどう書くか考えるだけで済む気がする〜。

**前田：** みさきさんの直感は正しくて、`onboarding.ts`の最後に「移行の合図（2往復後にブリッジ文）」が既に定義されています。つまり**F4 Phase 1はそのブリッジ文の「渡り先」を定義する作業**なんです。入口は既にある。
