## Turn 1｜COCOSiLのリスク地図を描く：現状診断

**松田：** まず事実の整理から入ります。COCOSiLのスタックを見ると、Next.js App Router + Supabase + Clerk + OpenAI Agent Builderという構成ですね。MVPフェーズで2人チーム（えんまささんがフロント、ヒラメさんがバック）。この構成でAIエージェントを積極活用している。

ここで重要な観察が3点あります。

**①Supabase＋Clerkは「破壊コストが高い組み合わせ」です。** Supabaseのmigrationを誤って走らせると本番DBが死にます。私が支援した12社中3社で、AIエージェントが`supabase db reset`を実行しようとしたインシデントがありました。Clerkの設定誤操作も同様——JWTの設定を変えるとユーザー全員がログアウトします。

**②2人チームのロール分担は「AIへの権限委譲の境界線」になります。** えんまさ＝フロント、ヒラメ＝バックという分担があるなら、それをハーネスに反映させるべきです。「誰のAIが何を触っていいか」を明示しないと、フロント担当のエージェントがSupabaseのRLS設定を書き換えるという事故が起きる。

**③「診断コンテンツ」の著作権リスクがある。** 60動物占術のナレッジベースをAIで自己構造化する計画がありますよね。AIエージェントが外部APIから情報を取ってきて勝手に書き込む——これは今の段階でルールを明記しておかないと後で大問題になる。

**結論：COCOSiLはR2とR3の境界線上にいます。** ユーザーデータ（Supabase）＋認証（Clerk）＋AI診断コンテンツがある時点で、単純なR2では甘い。でも課金機能はまだないのでR4ほどではない。「R2+」か「軽量R3」として設計するのが正解です。

---

**中川：** 松田さんの診断に同意しつつ、フロントエンド側の視点を補足します。

Next.js App Routerを使っているなら、`next build`と`next lint`と`tsc --noEmit`の3コマンドが必ず実在するはずです。これはCOCOSiLの開発を止めない最低限の検証コマンドとして確定できます。

ただし気になるのはテストの有無です。MVPフェーズのスタートアップでunit testを書いてるチームは少ない。もしCOCOSiLにもテストがないなら、**HARNESS_HEALTH.mdに最初から「テストコマンドなし」と記録しておく**のが誠実な導入です。存在しないコマンドを`AGENTS.md`に書いてしまうと、エージェントが架空のコマンドを実行して詰まる。

**中川：** そうです、みさきさん、それで合ってます。

## Turn 2｜設計アプローチの選択肢：フルスタックか、分割か

**中川：** ハーネス導入のアプローチとして3つの選択肢が考えられます。軸は「単一repo vs 分割」と「リスク重心をどこに置くか」です。

| アプローチ | 構成 | メリット | デメリット |
|---|---|---|---|
| **A: fullstackテンプレート1本** | `AGENTS.md` 1本でフロント・バック共通 | シンプル、管理ラクで2人チームに最適 | フロント/バックの権限分離が弱い |
| **B: frontend+backendを別AGENTS.md** | `src/`以下にフロント用、`supabase/`以下にバック用 | 権限を細かく分離できる | 管理コスト2倍、MVPフェーズには過剰 |
| **C: fullstack+path-scoped rules** | A をベースに `.claude/rules/*.md` でスコープ分け | 必要な分離だけ追加できる | 少し構成が複雑 |

**COCOSiLには現時点でAが最適だと思います。** 理由は2つ——MVPフェーズで管理コストを増やすべきではない、そして2人チームなら口頭での調整が効く。ただし `AGENTS.md` のProtected areasセクションに「Supabase migrations」「Clerk設定」「.env系ファイル」を明示的にリストアップする。

**松田：** 中川さんのA案に同意しますが、1点追加します。`protect.env`の保護について——Supabaseは`.env.local`だけでなく、`supabase/.env`も存在するケースがあります。Next.jsプロジェクトは`.env`系ファイルが複数個所に散らばりやすい。hookのブロックパターンに`cat .env`や`cat .env.local`の両方を含めるのが重要です。

それと、もう一つ判断が必要なのが**Skillの選択**です。kecku-standard-harnessのSkillラインナップをCOCOSiLの開発フローに当てはめると：

```
🔴 必須：kecku-harness-bootstrap（導入）、code-change-verification（検証）
🟡 推奨：pr-draft-summary（PRはGitHub使ってる前提）、docs-sync（診断コンテンツの更新）
🟢 任意：security-sensitive-change-review（R3相当の変更時のみ）、harness-health-improver（月次）
```

**みさき：** てか、Skillって何個も入れたらいいってもんじゃないんですか？多い方が安全じゃないの？

**松田：** いい質問です。実は逆です。Skillが増えすぎると「どのSkillが発動すべきか」をAIエージェントが間違えるんです。似た名前のSkillが3つあると、間違ったSkillを呼ぶ誤発火が増える。COCOSiLは今、必須2本でいい。

**みさき：** あ〜、料理で「調味料多すぎると味がわかんなくなる」みたいな感じですね〜！

**中川：** そのたとえ、すごく正確です（笑）。

## Turn 3｜COCOSiL固有の要件を洗い出す：現場の実需から原則へ

**みさき：** ちょっといいですか、私 COCOSiLのユーザーになりたい側の人間として思うんですけど——MBTIの診断結果とかって、めっちゃパーソナルな情報じゃないですか。私のMBTI結果をAIがこっそり書き換えたりしてたらめっちゃ嫌なんですけど〜。それってハーネスで守れるんですか？

**中川：** みさきさん、それ本質的な点を突いてます。これはSupabaseのRLS（Row Level Security）ポリシーの話にもなるんですが、ハーネスでできることは「AIエージェントがRLSポリシーを変更するSQLを実行しようとしたらブロックする」という部分です。具体的に言うと：

```javascript
// prevent-destructive-command.js に追加すべきパターン
const DESTRUCTIVE_PATTERNS = [
  /DROP\s+TABLE/i,
  /TRUNCATE/i,
  /ALTER\s+TABLE.*DISABLE.*ROW/i,   // RLS無効化
  /CREATE\s+POLICY.*ALL/i,           // 全権限ポリシー追加
  /supabase\s+db\s+reset/i,          // DB初期化
  /supabase\s+migration\s+repair/i,  // マイグレーション強制修復
];
```

これでユーザーの診断データにアクセスできるRLSポリシーをAIエージェントが変えようとした瞬間にブロックできます。

**松田：** ここで重要な原則を抽象化します。COCOSiLの「守るべきもの」を整理すると、3層構造になります：

```
Layer 1 ── インフラ層（壊れると全ユーザー影響）
  → Supabase DB schema / RLS policy
  → Clerk JWT設定 / redirect URL
  → .env系すべて

Layer 2 ── コンテンツ層（壊れると信頼が失われる）
  → MBTI/西洋占星術/動物占い/六星占術のナレッジデータ
  → AI診断プロンプトテンプレート
  → ユーザーの診断履歴

Layer 3 ── 開発層（壊れても修復可能）
  → UIコンポーネント
  → APIルート
  → 一般的なビジネスロジック
```

**この3層をAGENTS.mdのProtected areasに明示することが、COCOSiL固有のハーネス要件の核心です。**

**みさき：** あ〜！つまり「AIが触っていい棚と触っちゃダメな棚を分ける」ってことか〜！Layer 2の診断コンテンツはまさに触られたくないやつですよね、マジで〜。

**中川：** そして優先度をつけると：

🔴 **必須（Day 1に入れる）**：Layer 1の保護 + `prevent-destructive-command.js`
🟡 **推奨（Week 1以内）**：Layer 2の保護ルール + `HARNESS_HEALTH.md` にテスト欠如を記録
🟢 **任意（月次レビュー時）**：`harness-health-improver` でアップデート追跡

## Turn 4｜MVPフェーズのリスクと「やりすぎない」法則

**松田：** 少し逆説的な話をします。COCOSiLは今MVPフェーズで、開発速度が最優先です。ハーネスを入れすぎると開発が遅くなる。これが私が見てきた失敗の中で一番多いパターンです。

例えば `rm` を全部ブロックすると、`rm -rf node_modules` すら通らなくなる。正しいパターンは：

```javascript
// ❌ ダメな例（広すぎる）
/\brm\b/i

// ✅ 正しい例（危険なケースだけ）
/rm\s+-rf\s+[^n]/i   // rm -rf は要確認（node_modulesは除外）
/rm\s+-rf\s*$/i      // rm -rf [引数なし] は絶対ブロック
```

**「最小限のフェンスで最大限に守る」——これがCOCOSiLのMVPフェーズにおけるhook設計の法則です。**

**中川：** 同意します。そしてもう一つ、COCOSiLに特有な設計判断があります。**えんまささんとヒラメさんで開発フローが違う**という点です。

えんまささん（フロント）は v0 や Claude で UIを生成するワークフローを使っている。この場合、AIが生成したコンポーネントを`src/components/`に直接書き込む操作が多い。これはむしろ**許可すべき操作**です。

一方ヒラメさん（バック）は Supabase の設定変更が発生する。こちらは慎重にチェックが必要。

つまり**同じリポジトリでも、フロントエンドとバックエンドの「触ってOKな操作」が違う**んです。

**みさき：** え、てかそれって「えんまさとヒラメさんで担当が違うから、AIへの命令も違う」ってこと？なんか会社の権限設定みたいですね〜。部長は全部できるけど平社員は閲覧だけ、みたいな。

**松田：** その比喩は正確ですが、COCOSiLの場合は「役割で分ける」より「操作の性質で分ける」方がシンプルです。**操作の可逆性で権限を決める**——Gitでrevertできる変更はAIに委ねてOK、できない変更（DBスキーマ変更・認証設定・.env変更）は必ず人間が承認する。これがCOCOSiLのハーネス設計の基本原則です。

**みさき：** あ〜それは確かに。誰がやってるかよりも何をやってるかで判断する方が分かりやすい〜。

## Turn 5｜設計原則の確定と「明日からの3ステップ」

**松田：** 5ターンの議論を整理して、COCOSiLのハーネス設計原則を3つに集約します。

### 【COCOSiLハーネス設計の3原則】

**① Reversibility-First（可逆性ファースト）**
AIエージェントが触っていい範囲はGitでrevertできる操作まで。DBスキーマ・RLS・Clerk設定・.envは全て「要人間承認」としてhookでブロックする。

**② Minimum Fence, Maximum Speed（最小フェンス・最速開発）**
MVPフェーズの開発速度を守るため、ブロックリストは「絶対アウトな12パターン」に絞る。疑わしいものは全ブロックではなく、HARNESS_HEALTH.mdに記録して月次レビューで判断する。

**③ Layer-Aware Protection（3層保護）**
インフラ層（DB/Auth/env）→ コンテンツ層（診断データ・プロンプト）→ 開発層（UI/API）の3層で保護水準を分ける。Layer 1・2はブロック+人間承認、Layer 3はAIに委任。

---
