## Turn 1｜現状認識：Gamma依存の「本当の問題」を診断する

**矢野：** コスト面だけ見ると「Gamma API月2万円 vs gpt-image-2 Batch $525」で移行一択に見えます。でも私が注目するのは別の数字です。Gammaに依存している状態は**「コストの問題」ではなく「意思決定権の問題」**です。GammaがAPIポリシーを変更すれば、レポートのデザインが突然変わる。クレジット制度が変われば予算計画が崩れる。実際、私が支援したSaaS2社でGammaのフォーマット変更により、フロント実装を緊急修正する事態がありました。「月2万円払い続ける問題」の本質は、**自社コアプロダクトのアウトプット品質を外部プラットフォームに人質に取られている**という構造問題です。

**福田：** デザイナー視点では別の根本問題があります。Gammaのテンプレートは「プレゼン映え」に最適化されています。一方、COCOSiLのレポートが果たすべき役割は**「腑落ち体験の提供」**です。プレゼン映えは視線を引きつけて感嘆させるデザイン。腑落ちは「あ、これ自分のことだ」という認識の解像度を上げるデザイン。Gammaのテンプレートに乗り続ける限り、設計中枢「無明を晴らす、装飾しない」を体現したレポートは作れない。これは**「デザインの手段的依存」から「哲学的依存」に変質してきている**、と診断します。

**みさき：** レポートってどうやって「読む」想定ですか？スマホ？印刷？私MBTIの診断結果をLINEで友達に送るとき、いつもスクショするんですよね〜。Gammaで作っても画像で作っても、最終的にはJPEGじゃないですか。「どう使われるか」が先に決まってないと、どのツールが向いてるかも変わりそう〜？

**矢野：** みさきさんの指摘、めちゃくちゃ重要です。「LINEで送るためにスクショ」は、**「レポートの消費単位が画像1枚」**であることを意味します。Gammaの「複数スライド」という強みがそもそも活きないユースケースです。

**福田：** つまり診断すると：**① 外部依存による意思決定権の喪失、② GammaとCOCOSiL哲学の根本的ミスマッチ、③ 「複数スライド」という強みが実ユーザーの使い方に対応していない**。この3つが「月2万円」という数字に集約されています。

## Turn 2｜選択肢の発散：評価軸を立て、3アーキテクチャを配置する

**福田：** 評価軸を整理します。

| 評価軸 | 説明 |
|---|---|
| **文字信頼性** | 日本語テキストの崩れない度合い |
| **腑落ち表現力** | 「自分のことだ」という感覚を生む設計の自由度 |
| **実装コスト** | 工数・スキル要件・保守負荷 |
| **ランニングコスト** | MAU1,000人規模での月額推定 |
| **COCOSiL哲学適合** | 「装飾しない・無明を晴らす」との整合性 |

| アーキ | 文字信頼性 | 腑落ち表現力 | 実装コスト | 月額コスト | 哲学適合 |
|---|---|---|---|---|---|
| Vercel OG (Satori) | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | ~$10〜50 | ⭐⭐⭐⭐ |
| gpt-image-2単体 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ~$525(Batch) | ⭐⭐⭐ |
| ハイブリッド | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ | ~$200〜300 | ⭐⭐⭐⭐⭐ |

gpt-image-2の「哲学適合 ⭐⭐⭐」は、「映える」と「腑落ちする」は別の設計思想であり、プロンプト設計を誤ると装飾が過剰になるリスクがあるため。

**矢野：** 「gpt-image-2単体は実装シンプル」はHappy Path限定の評価です。Vercelで60秒タイムアウトを解決するには①Fluid Compute設定、②生成完了通知UI、③失敗時のリトライ設計が必要。Vercel OGは同期・即時生成なのでこの問題が存在しない。

**みさき：** 「腑落ち表現力」って「これ私だ！ってなる感じ」ですよね？MBTIの結果画像、INFJっぽい色使いや雰囲気がわかるの、めっちゃわかる〜。あれってgpt-image-2が得意ってこと？

**福田：** 「INFJっぽい色使い」を**「象徴的視覚化」**と呼んでいます。Vercel OGはコードで作るので象徴的配色をデザイナーが意図的に設計できる。gpt-image-2はAIが自律的に解釈するので毎回異なる表現になりうる。COCOSiLのブランド一貫性を守る観点では一長一短です。

## Turn 3｜実需・要件の確定：「シェアしたい」の解剖から優先度を決める

**みさき：** 私が性格診断の結果をSNSにシェアするとき、何をシェアするかって言うと「自分のタイプ全体の画像」じゃなくて、**「自分の特徴の中で共感できた一文」**なんですよね。「INFJは完璧主義で、自分のビジョンに妥協しない」みたいなやつ。画像全体じゃなくてテキストの一部分じゃないですか。レポートのデザインより「刺さる言葉が書いてあるかどうか」の方が大事ってことにならないですか？

**福田：** パターン化すると：**「診断レポートの主役はデザインではなく、ユーザーが自分との一致を感じる核心ワード」**。グラフィックの美しさは「読む気にさせる入口」でしかない。これは要件の優先度に直結します：

| 要件 | 優先度 | 根拠 |
|---|---|---|
| テキストの正確な描画 | 🔴 Must | 核心ワードが崩れると「伝わる」が完全に死ぬ |
| UXシーケンス（共感→安心→分析→行動）の視覚的表現 | 🔴 Must | COCOSiL哲学の体現・7日再訪率に直結 |
| 象徴的デザイン（体系ごとの視覚的個性） | 🟡 Should | 「INFJっぽい」感を演出するが必須ではない |
| アニメーション・インタラクション | 🟢 Nice | 現フェーズでは不要 |

**矢野：** 「テキストMust」が確定したことで、アーキテクチャ選択は自動的に絞られます。**gpt-image-2の99%精度は「月5,000枚 × 1% = 50枚の文字崩れ」を意味します**。画像はCDNキャッシュに乗った後は修正できない。核心ワードが崩れた状態でキャッシュされたユーザーは間違ったレポートを受け取り続けます。Vercel OGはReact + CSSなので物理的に文字崩れが発生しない。「テキストMust」の要件に対してはVercel OGが唯一の完全解答。

**みさき：** 映えより信頼感でしょ、って思います〜。綺麗なレポートで文字が崩れてたら、診断結果自体を疑っちゃう。

## Turn 4｜技術深掘り：段階的移行の「法則」と崩してはいけない制約

**矢野：** 私がSaaS6社の移行で学んだ**「依存コスト削減の3段階法則」**があります。

**第1段階 — 機能等価（差し替え）：** Gammaで作っていたものと機能的に同等品を自社で作る。デザイン改善はしない。工数：スプリント1〜2本。

**第2段階 — 品質向上（最適化）：** 等価置換が安定したらCOCOSiL固有のデザインを詰める。UXシーケンスを視覚的に体現したレイアウト設計。工数：スプリント2〜3本。

**第3段階 — 差別化（強化）：** gpt-image-2を「背景・アイコン生成専用」として部分導入。文字はVercel OG、ビジュアルのみAI生成のハイブリッド化。工数：スプリント2〜3本。

この段階設計の意義は**「リスクの分離」**。第1段階が失敗してもGamma APIに即戻れる。第3段階でgpt-image-2が問題を起こしてもVercel OGレイヤーが担保している。

**福田：** 重大な補足。**「第1段階の等価」はGammaと同じデザインではなく、ユーザーへの診断結果の画像デリバリーが止まらないことという機能的等価**です。GammaのテンプレートをコピーするとGammaの設計思想のコピーになる。デザインはゼロから、COCOSiLのUXシーケンスに従って設計する必要があります。

そして崩してはいけない制約：**文字領域はVercel OGが100%担保する設計は絶対に崩さない**。「核心ワードが崩れたレポートがキャッシュされる」リスクは許容できない。

**みさき：** え、最初から「COCOSiLのデザインで作る」必要があるんですね〜。「とりあえずGammaのコピー」じゃだめって、意外と深い話だ〜。

**矢野：** 「ゼロから設計」は必ずしも「工数が増える」ではないです。Vercel OGはReactで書けるので、COCOSiLのデザインシステム（Tailwind + デザイントークン）をそのまま使える。GammaのテンプレートエンジンのUIに合わせる方が難しかった。

## Turn 5｜方針統合：設計3原則と明日からの具体アクション

**福田：** COCOSiL向けのレポート生成設計原則を3つに集約します。

### 🧭 COCOSiL レポート設計3原則

**① Text is Sacred（文字は聖域）**
レポートの核心価値はテキストの「核心ワード」にある。文字の描画は画像生成AIに任せない。Vercel OG（Satori / React）が文字レイヤーを100%担保する。1%の文字崩れも許容しない。

**② Insight Over Impression（印象より洞察）**
「映える」デザインより「腑落ちするレイアウト」を優先する。gpt-image-2を使う場合も、背景・アイコンなど「文字を持たないビジュアル要素」のみに限定する。装飾は「無明を晴らすか？」の問いに答えられないものは入れない。

**③ Ship First, Enhance Later（まず届ける、後から磨く）**
3段階移行（等価→品質向上→差別化）を守る。gpt-image-2はフェーズ3以降のオプション。Gamma APIはフェーズ1完了・えんまさ確認後にFeature Flagで切り替え。

**矢野：** 明日からの具体アクションに降ろします。

**フェーズ1（スプリント1本・2週間）：**
1. `app/api/reports/generate/route.ts` に `@vercel/og` を使ったエンドポイントを新設
2. 4体系の診断結果データを受け取り、`1024×1792`（縦長）のPNGを返す実装
3. Supabase Storage の `user-reports` バケットへの保存パイプライン構築
4. 既存のGamma APIとFeature Flagで並列稼働（Gamma APIはすぐに廃止しない）
5. `pnpm typecheck && pnpm build` 通過確認

**フェーズ2（スプリント1〜2本）：**
6. Vercel OG内で「共感→安心→分析→行動」の4セクションレイアウトを設計
7. 各体系のデザイントークンを `lib/constitution/` に追加
8. えんまさによるプレビューURL確認（Gate 2 必須）

**フェーズ3（スプリント2〜3本）：**
9. gpt-image-2で体系別の象徴背景画像を非同期生成（Inngest JobQueue経由）
10. Vercel OGの文字レイヤーの背景にgpt-image-2生成画像をSharpで合成
11. `quality: low` でプロンプト検証 → `quality: high` で本番生成

**みさき：** 最初「GammaをAIに変えるだけ」みたいな話かと思ってたら、「まずVercel OGで文字をちゃんと届ける」→「COCOSiLらしいデザインに磨く」→「最後にgpt-image-2で映えを足す」って順番なんですね〜。「まず信頼を作ってから映えを追加する」って、COCOSiLが「装飾しない」って言ってる順番と同じな気がしてきた！テキストが正確なことが信頼の土台だって、めっちゃ腑落ちしました。

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| **第1選択アーキテクチャ** | Vercel OG (Satori) — 文字レイヤーの完全担保 |
| **gpt-image-2の位置づけ** | フェーズ3以降・背景/アイコン専用（文字非担当） |
| **Gamma APIの廃止タイミング** | フェーズ1完了・えんまさ確認後（Feature Flag切り替え） |
| **最優先要件** | テキスト信頼性 > 象徴的デザイン > コスト削減 |
| **移行フェーズ数** | 3段階（等価→品質向上→差別化） |
| **崩してはいけない制約** | 文字領域はVercel OGが100%担保。gpt-image-2に文字を持たせない |
| **Vercelタイムアウト対策** | フェーズ3でInngest非同期化（フェーズ1〜2は不要） |
| **月額コスト見込み** | フェーズ1〜2: ~$10〜50 / フェーズ3: ~$200〜300（ハイブリッド） |
| **Gate 2 必要タイミング** | フェーズ2のレイアウト設計・フェーズ3のプロンプト設計時 |
