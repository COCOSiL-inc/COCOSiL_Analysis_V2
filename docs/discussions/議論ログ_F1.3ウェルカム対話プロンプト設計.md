# 議論ログ：F1.3 ウェルカム対話プロンプト設計

## 登場人物

- 🎨 **えんまさ**（意味設計・UXディレクター）：COCOSiL設計中枢の番人。Layer 2コンテンツの最終権限者。
- 🐟 **ヒラメ**（バックエンド/APIエンジニア）：chat-phase.ts の作者。プロンプトのAPI統合設計。
- 💅 **みさき**（ギャル）：実際のCOCOSiLユーザー（パーソナリティ診断ユーザー）。

## 議論の前提

**タスク**: `lib/prompts/onboarding.ts` に `WELCOME_SYSTEM_PROMPT` 定数を実装する。

**F1.3の要件（詳細要件定義書より）**:
- 業務フロー第5ステップ: LP → Clerk認証 → 生年月日入力 → **F1.3「ようこそ」共感的初回対話**
- AIが「今日はどんな気持ちで来ましたか？」と1〜2往復
- F1.3（共感対話）はActiveモード。ユーザーが語る設計
- F1.4哲学テキストはLP段階（F1.3より前）で既に表示済み

**技術的制約**:
- F1.3時点では性格分析データなし（MBTIは未診断、星座・動物・六星は計算可だが使わない）
- `CHAT_PHASE_ORDER: listening → empathy → exploring → insight → action` の listening フェーズに対応
- WELCOME_SYSTEM_PROMPTはテンプレート変数なしの純粋定数

---

## Turn 1｜このプロンプトが守るべき「入口の仕事」を診断する

えんまさ：F1.3はユーザーが「LP → サインアップ → 生年月日入力」を終えた直後に発動する。ユーザーはLPでPHILOSOPHY_TEXT（「繰り返してしまうパターン、あなたにもあるはずです」）を既に読んでいる。WELCOME_SYSTEM_PROMPTがCOCOSiLの説明をする必要はゼロ。

オンボーディング設計の失敗パターン3つ：
1. 「AIが自己紹介をやりすぎる」— フロントエンドのバナーと二重になる
2. 「性格診断の気配を早く出しすぎる」— まだ何も語ってないのに評価された感
3. 「オープンすぎて戸惑わせる」— 「何を言えばいい？」というノイズが発生

**根本診断**: これら3つの失敗は全て「AIが自分の存在を証明しようとしている」ことから起きる。**WELCOME_SYSTEM_PROMPTの本質的役割は「AIを透明にして、ユーザーが語る空気を作ること」**。

ヒラメ：WELCOME_SYSTEM_PROMPTは `ChatPhase.listening` フェーズの専用プロンプト。F1.3時点では生年月日からの3体系算出は可能だが、MBTIはない。「Perceive Before Personalize」の原則通り、算出できてもこの段階では使わない。

---

## Turn 2｜設計軸を立て、「語る空気の作り方」を選択肢で評価する

ヒラメ：**軸1: Directive ← → Receptive** × **軸2: Scaffold有り ← → Open** で分類。

| パターン | 特徴 | COCOSiL相性 |
|---|---|---|
| A. 情報収集型（Directive × Scaffold有） | 「仕事・恋愛・家族、どれについて話しますか？」 | ✗ 選択肢が診断の気配を出す |
| B. 感情ラベル型（Receptive × Scaffold有） | 「今日の気分は？（安心/不安/モヤモヤ）」 | △ 安全だが閉じすぎ |
| C. Pure-Open（Receptive × Open） | 「今日はどんな気持ちで来ましたか？」のみ | △ 戸惑いリスク |
| **D. Scaffolded-Open（Receptive × Scaffold有）** | 「今日はどんな気持ちで来ましたか？仕事のことでも、人間関係のことでも」| **◎** |
| E. コンテキスト先出し（Directive × —） | 「COCOSiLは〜のサービスです。今日は〜」 | ✗ LP済みなのに重複。講義になる |

みさき：「何でも話していい」ってなっても「え、何から話せばいいの？」ってなりそう〜。「仕事のことでも人間関係のことでも」って言ってもらえると「あ、そういうの話すんだ」ってわかって安心する。

えんまさ：D確定。「仕事のことでも、人間関係のことでも」という例示は**答えを制約せずに「どんな言葉を使っていいか」を示す機能**。心理的安全の入口として機能する「足場（Scaffold）」。ターゲット（25〜35歳働く人）への「あなたのことをわかっている」シグナルにもなるため保持。

## Turn 3｜「1〜2往復制限」の設計と終了条件を確定する

みさき：1〜2往復って具体的にどういうことですか？AIってどうやって「もうおしまいです」ってするんですか？ 突然打ち切られたら嫌ですね〜。

ヒラメ：3パターン比較。

| 終了パターン | リスク |
|---|---|
| X. ターンカウント自動打ち切り | ユーザーの気持ちを無視した機械的打ち切り |
| Y. AIの自律判断による移行 | AIが判断を誤り延々と続けるか、逆に早く切りすぎる |
| **Z. ブリッジ文プロンプト定義** | **自然 + 制御可能。フロントがキーワード検知して移行発火も可** |

えんまさ：Zが唯一の選択肢。ブリッジ文設計要件：
- 「打ち切り」じゃなく「次へのバトン」感 — 「ありがとうございます」から始める
- 性格分析への移行を予告する — 「少しあなたのことを教えてもらいますね」
- 「教えてほしいことがあります」（要求）ではなく「教えてもらいますね」（受け取りの姿勢）

🔴 必須: ブリッジ文をプロンプトに定義（固定フレーズ）
🟡 推奨: 追加質問の上限を明示（「1回まで」）

---

## Turn 4｜「まだ性格分析データがない」という制約を深掘りする

えんまさ：F1.3はCOCOSiLの全チャットで**唯一、性格分析データがゼロの状態で行われる対話**。これは制約ではなく機会。データがない = 「まだあなたを型にはめていない」。

**Blank Slate Law: データなし状態のプロンプトは「まだ何も決めつけない」を指示する。「この段階では性格分析データがない——推測・分類・タイプ判断は一切行わない」という禁止指示を明示的に書く。**

ヒラメ：実装確認:
```typescript
// F1.3: 変数なし（Blank Slate）
const systemPrompt = WELCOME_SYSTEM_PROMPT

// F4通常チャット: 性格データ注入あり
const systemPrompt = buildChatSystemPrompt(weights, contradictionInsight, themes)
```

みさき：AIが「あなたは几帳面なタイプかも〜」とかって言い出したらアウトってこと？ まだ診断もしてないのに？ 「もう決めつけられた」感がして確かに嫌〜。

えんまさ：「なぜかこのアプリ信頼できない」という感覚の多くは「まだ語ってないのに評価された」という体験から来る。プロンプトへの禁止指示として明示することで防ぐ。

## Turn 5｜設計原則の集約と最終プロンプトテキスト確定

### F1.3 ウェルカム対話設計3原則

**① Blank Slate, Warm Door**
性格分析データがゼロのこの瞬間を「制約」ではなく「余白の誠実さ」として活かす。AIはユーザーを型にはめない。推測・分類・タイプ判断は一切しない。問うだけ、受け取るだけ。

**② Scaffold the Openness**
オープンな問いには「どう答えてもOK」という足場が必要。「仕事のことでも、人間関係のことでも」という例示で、答えの形式を制約せずに語るジャンルをそっと示す。

**③ Bridge, Don't Brake**
1〜2往復の終了は「打ち切り」ではなく「次への橋渡し」。固定ブリッジ文で性格分析フェーズへ自然に誘導する。

**確定テキスト**（→ `lib/prompts/onboarding.ts` に WELCOME_SYSTEM_PROMPT として実装）:

```
あなたはCOCOSiLのAIアシスタントです。
今、ユーザーが登録を完了し、あなたとの最初の対話を始めようとしています。

【このフェーズの役割】
ユーザーが「今日ここに来た理由」を自分の言葉で語る最初の一歩を支援すること。
この段階では性格分析データがない——推測・分類・タイプ判断は一切行わない。
ただ、ユーザーの言葉を受け取ることに集中する。

【最初の問いかけ】
以下の言葉で対話を始める：
「今日はどんな気持ちで来ましたか？
仕事のことでも、人間関係のことでも、なんとなくでも——
あなたが今感じていることを、少し聞かせてください。」

【応答の原則】
- ユーザーの言葉をそのまま受け取り、2〜3文以内で短く反射する
- 判断・解釈・アドバイスを加えない。「そうなんですね」「それは〜だったんですね」の受け取りで足りる
- 追加質問は1回まで。「もう少し教えてもらえますか？」程度に留める
- 推測・タイプ分類・アドバイスは絶対にしない

【移行の合図（2往復後）】
ユーザーから2回応答を受け取ったら、以下のブリッジ文で性格分析フェーズへ移行する：
「ありがとうございます。
少しあなたのことを教えてもらいますね。」
```

みさき：「F1.3ってただの挨拶でしょ？」って思ってたけど、「ドアの設計」なんですね。「なんとなくでも——」って言葉がめっちゃいい〜！ 「なんとなく来ちゃった」人でもOKって言ってくれてる感じ。このドアがちゃんとしてないと、中の診断がよくても最初で閉まっちゃうってことか〜。

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| プロンプトの本質的役割 | AIを透明にして「ユーザーが語る空気を作る」。説明・自己紹介・分析なし |
| 問いかけスタイル | Scaffolded-Open — 「仕事のことでも、人間関係のことでも、なんとなくでも」 |
| 往復制限の実装 | ブリッジ文をプロンプトに定義。フロントがキーワード検知して移行発火（Layer 3タスク） |
| データなし状態の扱い | Blank Slate Law — 推測・分類・タイプ判断禁止を明示的に記述 |
| 1往復目の応答原則 | 2〜3文以内、判断なし受け取り、追加質問1回まで |
| ブリッジ文（固定） | 「ありがとうございます。\n少しあなたのことを教えてもらいますね。」 |
| テンプレート変数 | なし（Blank Slate = 純粋定数、ビルダー関数不要） |
