**②腐敗速度**（コンセプト変化についていけなくなる速度）

```
ファイル                     AIへの影響  腐敗速度  更新コスト
README.md                     高         低        中

cocosil-domain/SKILL.md       最高       高        中
```

## Turn 4｜更新リスクと守るべき制約

**ヒラメ：** 更新時に注意すべき制約を3つ言います。

**① AGENTS.mdはL2保護対象に準ずる**  
`AGENTS.md`はAIエージェントの行動規範そのものです。書き方を間違えると、エージェントが誤った判断をする。特に「Protected Areas」（Section 7）の記述は変更前にえんまさのレビューが必要です。

**② cocosil-domain/SKILL.mdのdescriptionフィールドは検索キーワード**  
このフィールドはAIがスキルを自動検索するときに使われます。「いつ読み込むか」の判断基準がここにある。変えすぎると読み込まれなくなる。具体的に言うと、現在の`description`に「要件定義書: docs/input/requirements_input/mvp-requirements-v1-3.md」という参照が入っていますが、`docs/input/concepts/`へのリファレンスもここに追加すべきです。

**③ README.mdのドキュメントリンクが現実とずれている**  
現状の`README.md`は`[コンセプト資料](docs/concepts/コンセプト資料.md)`と書いていますが、実際のパスは`docs/input/concepts/コンセプト資料.md`です。リンク切れが発生しています。更新のついでに修正必須。

**えんまさ：** ヒラメの指摘に加えて、言語リスクも言います。

AGENTS.mdを書き直す際、**「占い師」「運勢」が紛れ込むリスク**があります。とくにプロダクト説明の文脈で4体系を紹介する際、「動物占い」という正式名称をUI向けに誤用しやすい。Section 0（新設するプロダクト哲学）では必ず「4体系パーソナリティ分析」と表現する必要があります。

もう一つ：`lean-canvas-v3.md`にある「3年後定性ゴール：ユーザーから『生きるのがラクになった』」という表現は強力ですが、**KPIでも開発ガイドでもない**。READMEに入れるなら「プロダクトビジョン」セクションとして明確に区分する必要があります。

**みさき：** え、「動物占い」って正式名称なんですか？知らなかった〜！でも確かにアプリで「動物占いの結果は子守熊です」って出てきたら、私「占いじゃん！」ってなりますよね。それって設計ミスじゃないですか〜？

**えんまさ：** そうです、みさきが正確に指摘してくれました。だからこそ**UIに出す文言と内部コードの名称を分ける**ルールが存在しています。でもそのルールが`language-design/SKILL.md`だけに書いてあって、`AGENTS.md`にも「なぜ分けるか」の説明がない。これが今回追記すべき核心の一つです。

## Turn 5｜方針統合・ドキュメント更新の設計原則

**えんまさ：** ここまでの議論を踏まえて、ドキュメント更新の設計原則を3つに集約します。

---

**【COCOSiLドキュメント更新の3原則】**

**① Philosophy First（哲学先付け）**  
技術ルールの前に「なぜか」を1〜3行置く。エージェントはルールの意図を知ることで例外判断の質が上がる。特に AGENTS.md Section 0（新設）と Section 6（禁止語の理由）に適用。

**② Concept-to-Code Traceability（コンセプト→コードの追跡性）**  
`docs/input/concepts/` の仕様が実装ルールに反映されているか追跡できる構造にする。具体的には：`cocosil-domain/SKILL.md` の description フィールドにコンセプトファイルへの参照を追加し、`AGENTS.md` の技術スタックに Clerk・OpenAI・Gamma API を追記する。

**③ Guard Rail Semantics（ガードレールに意味を与える）**  
禁止語・保護エリア・UXシーケンス固定のルールには必ず「このルールを破ったらどうなるか」を付記する。例：「共感フェーズスキップ禁止 → PMF仮説の7日再訪率30%が成立しなくなる」。

---

**ヒラメ：** 原則を受けて、実施した具体アクション5項目：

1. **AGENTS.md**: Section 0「プロダクト哲学（3行）」新設 + Section 3に Clerk/OpenAI/Gamma API を追記 + Section 6の禁止語に理由を追加
2. **cocosil-domain/SKILL.md**: `description` に `docs/input/concepts/` への参照追加 + ビジネスモデル（PMF仮説・損益分岐）セクション追加 + Gamma API 依存の警告を追記
3. **README.md**: ヘッダーにプロダクトビジョン追記 + ドキュメントリンクを正しいパスに修正（`docs/concepts/` → `docs/input/concepts/`）
4. **language-design/SKILL.md**: ブランドフレーズ（「自分を知って、ラクになる」等）と"使い分けトーン例"を追記
5. **すべての更新は現ブランチ `setup/v2-init` で実施。変更は追記・修正のみ（既存ルールの削除禁止）**

**みさき：** なんか最初に「READMEが開発者にしか最適化されてない」って話から始まって、結局「ドキュメントに魂を吹き込む」みたいなテーマになりましたよね〜。めっちゃ納得感ある〜！特に「なぜ共感が先か」がどこにも書いてないってやつ、私もちゃんと説明されたら納得するから、AIにもそれ伝えたら絶対品質上がりますよね。あと「動物占い」って言葉問題、マジ大事だと思うのでドキュメントにしっかり書いといてほしいです！
