## Turn 1｜COCOSiLのAIコーディング脆弱点の診断

**🔐 橋本：**
最も危険なレイヤーはAPI Route（Layer 3）×認証なしDB書き込みの組み合わせです。最大の問題は `getSupabaseClient()` がServer ComponentではなくAPI Routeで呼ばれていること。Clerk認証のJWTがRLSに渡っていない可能性が高く、「誰でも書き込める」か「Supabase anon keyで全許可」になっている危険性があります。これはIDOR（認可不備）の典型構造で、統計上AIコードの1.91倍発生します。

現行ハーネスが見落としているのはここです。Layer 1のhookは `DROP TABLE` や `git reset --hard` を正しくブロックしますが、**「動作するが危険なコード」はブロックできません**。静的パターンマッチで防げるのは既知の破壊的コマンドだけ。RLSバイパスは構造的な問題であり、hookを素通りしてCIにも引っかかりません。

私がCTOだった当時、まったく同じ構造（anon key + server-side insert）で退会済みユーザーが他人のデータを上書きし続けていたインシデントを経験しました。気づいたのは3ヶ月後です。

**⚙️ 村上：**
COCOSiL V2の構成を見て、3点の危険信号を指摘します。

**1. えんまさの「承認ゲート」が単一障害点**：Layer 2（AIプロンプト・診断ロジック）の承認がえんまさ1人に集中しています。バス係数1の典型で、承認が形骸化するリスクもあります。「見ればわかる気がするから早く承認する」という劣化です。

**2. テスト未整備下でのAIスループット過多**：G1（テスト基盤）未解消のまま、AIエージェントがLayer 3を自律的に変更できる状態は危険。Comprehension Debtが静かに積み上がります。

**3. Hookブロックは「事後防御」に過ぎない**：「緩やかに壊れるコード」（型安全の崩壊・仕様ドリフト）は検知できません。

**💅 みさき：**
あの〜、ちょっと聞いてもいい？「AIが書いたコードはバグが多い」って話、つまり**私の診断結果が間違ってる可能性があるってこと？** マジ困るんだけど〜！「あなたはこういう性格です」って言われて信じて、職場の人間関係まで変えようとしてたのに、それがバグのせいで全然違う結果だったりしたら、なんか余計消耗しない？

## Turn 2｜ヒラメ vs えんまさのレビュー責任の設計軸

**🔐 橋本：**
15年のセキュリティ経験から分類します。

**ヒラメが担う「動作検証」チェックリスト（5点）：**
1. **IDOR防御**：`auth.uid()` がRLSポリシーに存在するか、`user_id = auth.uid()` の条件が全CRUD操作に適用されているか
2. **JWT連携**：Clerk JWTのSupabase統合が `supabase.auth.getUser()` 経由で行われているか（`request.headers`直読み禁止）
3. **型安全の境界**：APIルートの入力値がZodスキーマで検証されているか、`unknown`→型アサーションの強制キャストがないか
4. **N+1・漏洩クエリ**：ループ内でSupabaseクライアントを呼び出していないか、`select('*')`で不要カラムを露出していないか
5. **エラー境界**：Supabaseのエラーオブジェクトをそのままクライアントに返していないか（DBスキーマ情報の漏洩）

**えんまさへの「承認言語化」の渡し方：**「ヒラメ署名済みチェックリスト」をPRに添付します。えんまさが判断するのは**技術的正しさではなく意味的正しさ**——「このレスポンス構造で共感フェーズを表現できているか」「このエラーメッセージで三毒を増幅していないか」です。

私がVibe Codingで本番DBを吹き飛ばした事例の教訓：**技術者が「動く」と言った瞬間にプロダクトオーナーが「完成」と思うのが最も危険**です。ヒラメの署名は「安全に動く」の宣言であり、えんまさの承認は「COCOSiLとして正しい」の宣言——この二重構造を明文化することで、レビューの責任が曖昧になりません。

**⚙️ 村上：**
**「関心分離型デュアルゲート」** が有効です。橋本さんが指摘したIDOR問題と、えんまさのバス係数1問題は、実は同じ根にあります。「技術的妥当性」と「意味的妥当性」が同じ承認者に集中していることが構造的な原因です。

- **Gate 1（ヒラメ）**: 技術的適合性チェック。型整合・RLS確認・禁止語彙テスト通過の機械的確認。「なぜ通ったか」をPRコメントに明文化する義務を課す
- **Gate 2（えんまさ）**: 意味的妥当性のみ。UXシーケンス・設計中枢との整合・腑落ち体験の主観確認に**絞る**

この分離により、えんまさの認知負荷は「技術細部を読む」から「哲学との乖離を嗅ぎ取る」に純化されます。Gate 1を規律化するとGate 2のレビュー時間が60〜70%削減されます。

G1未解消との接続：Vitestが動く前はGate 1が"口頭の誓い"にすぎません。**G1解消をGate 1制度化の前提条件として明示的にタスク化**することを提案します。

**💅 みさき：**
ちょっと待って、「担当を分ける」って言うけど、えんまさって結局コード見れるの？見れないなら「使ってみてOKそう」ってノリで承認してない？それって実質チェックゼロじゃん。忙しい日に「とりあえずOK」ってなったら私がバグ踏むんだけど🙃

## Turn 3｜現行ハーネスの穴と優先修正箇所

**🔐 橋本：**
**穴の特定（3点）：**

1. **IDOR（`app/api/**`）**: `GET /api/diagnoses/[id]` でClerk JWTの `sub` とDBの `user_id` を照合するコードが存在しない場合、他ユーザーの診断データを `id` 推測で取得できます。Layer 3は「AI委任OK」なので、hookも制度も止めません。

2. **えんまさ承認の形骸化**: PRチェックリストは「記入必須」ですが、空欄・コピペでもマージできます。`lib/prompts/` 変更時の禁止語チェックが手動なら、忙しい日の「とりあえずOK」は技術的に防げません。

3. **Constitution as Codeの実行不能状態**: `lib/constitution/banned-words.ts` が存在してもテスト未整備（G1未解消）のため、`drift.test.ts` が走らない。Constitution as Codeが**「コードとして機能していない」**状態です。

**共通パターン：「宣言はあるが実行が人間依存」**。hookは操作をブロックできますが、「危険な不作為」（照合コードを書かない、テストを書かない）は検知できません。

**最初に直すべき1点**：`drift.test.ts` を含むG1解消を最優先します。Constitution as Codeは現状「コメントと等価」であり、えんまさ承認もプロンプト禁止語チェックも、その上に乗っている全制度が機能不全のままです。

**⚙️ 村上：**
みさきさんの指摘は本質を突いています。「コードを見ない人が実効的にレビューする」唯一の方法は、**レビュー対象をコードから出力に変える**ことです。PRに「before/afterの実際のAI応答例3件」を必須添付にする。えんまさなら「この返答で腑落ちするか」は判断できます。

G1未解消のヒラメチェックリストの欠陥は「再現性の保証なし」です。チェックリストは人間の記憶に依存する手続きであり、テストなしのチェックリストは「今日は動いた」の記録であり、「明日も動く」の保証ではない。

Layer 2とLayer 3の境界については、`app/api/route.ts` の中に診断ロジックのインライン実装が入り始めた瞬間にLayer 2相当になります。この「漏れ出し」パターンは `pnpm lint` で検知できません。「`app/api/` 内でのビジネスロジック直書き禁止」をhookかArchitectureテストで強制する必要があります。

**💅 みさき：**
「哲学的整合性って結局、私が使って『なんか違う』って思う感覚のことじゃないの？それってえんまさじゃなくて、実際に使う私が判断すべきじゃん。えんまさが哲学で守れなかったとき、誰が私を守ってくれるの？」

## Turn 4｜「理解負債」防止の具体的メカニズム

**🔐 橋本：**
具体的に3箇所を解剖します。

**`app/api/` の認可ロジック**: ヒラメが `auth()` を呼んだことは確認できますが、「なぜこのエンドポイントは `userId` ではなく `orgId` で絞るのか」という判断根拠がコードに存在しません。AIが生成した `if (!userId) return 401` は動作しますが、ヒラメがRLS設計書を読まずに通過させたとき、IDORは静かに生き続けます。

**`lib/prompts/` の AI応答制御**: プロンプト文字列はレビュアーが「何を言っているか」は読めます。しかし「なぜこのフェーズで共感ではなく安心から入るのか」という止観構造との接続が失われると、次のPRで順序が崩れても誰も気づけません。禁止語テストは構文を守りますが、意図は守りません。

**Supabase RLSポリシー**: `auth.uid() = user_id` という1行が正しく見えます。ただしJWT内の `sub` がClerkの `userId` と一致している前提が**コード外の暗黙知**です。これが崩れるのはClerk設定変更時で、ヒラメではなくインフラ変更担当者が踏みます。

**法則：「動作の根拠を、動作と同じファイルに置け」**——これが理解負債の唯一の処方です。`assertInvariant(clerkUserId === rlsUserId, 'JWT-RLS mismatch')` という実行可能な型制約として埋め込む。`validateUxSequence(phases)` という関数が止観の順序制約を持つ。理解をドキュメントに逃がした瞬間、それは読まれないコメントと等価になります。

**⚙️ 村上：**
「ユーザーが最終ゲート」は逆転させられます。**フィードバックログをGate 2入力に変換するループ**を設計します。みさきへの応答に「腑落ちしなかった」「違和感があった」ボタンを置き、週次でえんまさが5件をレビューする。これはUXシーケンス違反の検知を人間1人の主観から**統計的パターン**に昇格させる動きです。

理解負債防止は「**Pull型知識共有**」が有効です。毎週月曜15分、AIが書いたコードの中から最も複雑な関数1つをヒラメが選んでSlackに投稿、えんまさ・まあみが質問する。弊社クライアント（10名以下チーム）で導入後、3ヶ月でコードレビュー指摘件数が**約40%減**した実績があります。

「Whyを書く義務」の制度化で最大の変化は**PRの粒度が小さくなる**ことです。「なぜ」を1文で書けない場合、PRが大きすぎるシグナルになる。レビュー時間が平均50%短縮します。

**💅 みさき：**
え待って、それって私がフィードバックしたら本当に直してもらえるの？それならやる〜！でもさ、そもそも普通の人って「なんか変な診断」って気づけなくない？私みたいに感性強めの子ならわかるかもだけど、みんながみんなそうじゃないし…結局また私頼みになるじゃん笑

## Turn 5｜新ハーネス設計原則と明日からのアクション

**🔐 橋本：**

**Harness Design Principles — COCOSiL V2**

**① Executable Invariants Over Documents**
「宣言より実行」。UXシーケンス順序もJWT-RLS整合も、テストとして動かしてこそ保証。`drift.test.ts` が沈黙している間はConstitutionは絵に描いた餅。

**② Trust the Code, Not the Person**
人間のチェックリスト依存は穴になります。`assertInvariant(clerkUserId === rlsUserId)` のように、根拠をコードと同居させてください。

**③ Gate-Per-Concern, Not Gate-Per-Person**
「ヒラメが見る＝技術的正しさ」「えんまさが見る＝意味的正しさ」。関心を混在させるとどちらも曖昧になります。

**④ Pull Knowledge, Don't Push Blame**
障害は個人の失念ではなく設計の欠如として扱います。月曜15分の振り返りはその場として機能します。

**明日からヒラメが取る3つのアクション：**
1. `lib/constitution/__tests__/drift.test.ts` を実装してCIに乗せる（G1解消の起点）
2. `app/api/` の各ルートに `assertInvariant(clerkUserId === rlsUserId)` を1箇所ずつ追加する
3. PRテンプレートにヒラメ署名済み5点チェックリストを追記し、次のPRで試運転する

**⚙️ 村上：**

**えんまさへの3つの行動変容：**

1. **Gate 2の入力を「AI応答サンプル3件」に固定する。** PRに「このプロンプトで生成した応答サンプル」を必須添付。えんまさは「腑落ちするか」「三毒が増幅していないか」だけを問えばよい。レビュー時間を10分以内に収められます。

2. **フィードバック収集の母集団を「感性ラベル付き」で設計する。** スクリーニングアンケートに「ついカッとなる」「ぐるぐる考える」のような三毒ベースの自己記述を1問入れ、感性傾向を事前把握。感性バイアスを構造で中和します。

3. **`validateUxSequence()` を今週1本書く。** ドキュメントを先に更新してはいけません。コードが先、ドキュメントは追従。このテスト1本が「Constitution as Code」への最初の実効コミットになります。

**核心メッセージ：** 「承認の質は、承認者の能力ではなく、**入力の設計で決まります。**」

**💅 みさき：**
ねー、最初は「AIが間違えたら私が傷つくじゃん」って思ってたけど、コードがルールになってて、私みたいな感性強めの子がフィードバックできるって聞いて、なんか変わった。でも正直、最後に信じるのは自分の「なんか違う」って感覚だと思う。それが届く仕組みがあるなら、使ってみる価値あるかも。

## ✅ 議論まとめ

| 項目 | 現状 | 新方針 |
|---|---|---|
| **共通問題の根** | 宣言はあるが実行が人間依存 | Executable Invariants Over Documents |
| **ヒラメの責任範囲（Gate 1）** | 未定義（AI委任OKとしか書かれていない） | 5点技術チェックリスト署名＋PRテンプレート必須 |
| **えんまさの責任範囲（Gate 2）** | 全Layer 2承認（技術＋意味が混在） | AI応答サンプル3件で「意味的妥当性のみ」判断 |
| **Constitution as Code** | テストなし（コメントと等価） | G1解消でdrift.test.ts + validateUxSequence()を実行 |
| **理解負債防止** | 仕組みなし | Pull型知識共有（月曜15分）+ Why義務でPR粒度自然縮小 |
| **Layer 2/3境界の曖昧さ** | app/api/への診断ロジック漏れ出し検知なし | Architectureテスト or hookで直書き禁止を強制 |
| **ユーザーが最終ゲート問題** | えんまさ1人の主観に依存 | フィードバックログ→週次Gate 2入力に変換するループ |
| **最優先アクション** | — | G1解消（drift.test.ts実装・CI連携） |
