## Turn 1｜「どうせStripeでしょ」という直感が正しい理由と落とし穴を診断する

**木村：** 「エンジニア目線で正直に言います。今回の条件を見た瞬間に『Stripeで決まり』と思いました。ただそれを言うだけでは議論にならないので、なぜ他の選択肢が脱落するのかを具体的に説明します。」

「私が過去に見てきた失敗事例です。あるBtoCスタートアップがPay.jpで決済基盤を作りました。日本語サポートが楽で最初は快適。でも1年後にグローバル展開を考えたとき、**Pay.jpは海外カードの受け入れが事実上できず**、移行コストが新規実装の3倍かかった。もう1社はPaddleで始めて、Merchant of Recordの仕組みで確かに税務が楽でしたが、**法人格取得前に個人名でMoRを組むと、法人化後の切り替えで数ヶ月止まる**リスクを踏んだ。」

「これを抽象化すると、決済プロバイダ選定は**『今の開発速度』と『将来の移行コスト』のトレードオフ**ではなく、**『将来の移行コストが高すぎて事実上ロックインされる』問題**です。」

**中原：** 「Stripeは**個人事業主でも登録可能**で、法人化後に同アカウントのまま法人に切り替えられます。Pay.jpも個人事業主OKですが将来性に課題があります。PaddleはMoR方式で法人口座と事業者証明が必要で、設立手続き中のCOCOSiLには審査が通りにくい。**Stripe一択に近い**というのが財務リスクの観点からの結論です。」

**みさき：** 「え待って、私的には『カード登録するサービスのUIが怖いかどうか』がめっちゃ重要なんですけど！決済画面だけ急に英語になるサービス、信頼感ゼロですよね〜。COCOSiLって"自分を大事にしてくれるサービス"感あるじゃないですか？その決済画面が「あれ？」てなったら絶対萎えるんですけど。」

**木村：** 「みさきさんの指摘が本質を突いていて、Stripe Elements / Payment Elementは**日本語UIに完全対応**、COCOSiLのブランドカラーに合わせたCSSカスタマイズも容易です。」

## Turn 2｜評価軸を確立し、4候補を多次元でスコアリングする

**中原：** 「COCOSiLの条件に特化した5つの評価軸を立てます。」

| 評価軸 | 重み | 理由 |
|--------|------|------|
| **①技術統合コスト（60日制約）** | 高 | V2開発60日。実装速度が命 |
| **②サブスク管理の成熟度** | 高 | 4プラン×月額、プラン変更・解約・プロレーション |
| **③法人格問題への対応** | 高 | 設立手続き中。個人事業主での契約可否 |
| **④Clerk連携の品質** | 中 | ユーザーIDとサブスク状態の紐付けは設計の核 |
| **⑤将来性（B2B・海外展開）** | 中 | 3年後MAU 1万人・法人展開シナリオへの対応 |

**木村：** 「この5軸でスコアリングします（5点満点）。」

| プロバイダ | ①技術統合 | ②サブスク | ③法人格 | ④Clerk連携 | ⑤将来性 | **合計** |
|-----------|-----------|----------|--------|-----------|--------|---------|
| **Stripe** | 5 | 5 | 5 | 5 | 5 | **25/25** |
| Pay.jp | 4 | 3 | 4 | 2 | 2 | **15/25** |
| Paddle | 3 | 4 | 1 | 2 | 4 | **14/25** |
| Square | 2 | 2 | 3 | 1 | 2 | **10/25** |

「PaddleのMerchant of Recordが魅力的に見えますが、**プライシング変更の柔軟性が低い**。¥500→¥1,500→¥3,000という4プラン構造を後から変えにくい制約があります。COCOSiLのように初期から収益モデルを試行錯誤するサービスには向かない。」

**みさき：** 「Pay.jpって点数低いんですね〜。Clerk連携が2点ってどういうこと？私のカード情報が繋がらないみたいなこと？」

**中原：** 「Clerk連携の2点は『繋がらない』のではなく『自前で実装するコードが多くなる』という意味です。StripeはClerkの公式パートナーで、公式SDKが提供されています。Pay.jpは公式Clerk連携がなく、Webhook処理を一から書く必要があります。60日の制約下では**実装日数換算で2〜3日の違い**になります。」

## Turn 3｜「支払う側」の実需とプロレーション問題を要件に落とす

**みさき：** 「私、実際にサブスク使ってて思うんですけど、プラン変更するときが一番ストレスなんですよね〜。「ライトからスタンダードに変えたいんだけど、今月残りの日数分どうなるの？」ってなって調べないといけないのがめっちゃ億劫で。COCOSiLって『自分を大事にしてくれる』じゃないですか？その決済体験だけリッチじゃなかったら一気に萎えませんか？」

**木村：** 「みさきさんが言ったのは**プロレーション（日割り精算）**の問題です。Stripe Billingはプロレーションが標準機能で、アップグレード時に即時課金・ダウングレード時に次回請求から反映、を2〜3行のコードで実現できます。」

```typescript
// Stripe: プラン変更（プロレーション込み）
await stripe.subscriptions.update(subscriptionId, {
  items: [{ id: itemId, price: newPriceId }],
  proration_behavior: 'create_prorations', // 日割り自動計算
});
```

「Pay.jpでこれを実現しようとすると自前で日割り計算ロジックを書く必要があり、**バグが出やすく開発工数が3〜5倍**になります。」

**中原：** 「これは**『決済体験の品質 = プロダクト哲学の体現』**という原則です。COCOSiLの哲学がオンボーディング→共感チャット→レポートと丁寧に設計されているのに、課金フローだけ粗雑だと体験の一貫性が壊れます。**決済UXの粗さは離脱率に直結する**——ウェルネス系スタートアップで決済画面でのドロップオフが全体LTVを30〜40%毀損した事例を知っています。」

**みさき：** 「あと解約ってどれくらい簡単にできるかも気になります！解約しにくいサービスって、なんか「囲い込もうとしてる」感あって逆に嫌で課金すらしたくなくなるんですよね〜。」

**中原：** 「これはF8.1の**データオーナーシップ原則と対になる設計**です。Stripeのカスタマーポータル（Stripe Billing Portal）を使えば、ユーザーが自分でプラン変更・解約・支払い方法変更を完結できます。実装はほぼ設定のみ。この**『自分で管理できる透明性』がCOCOSiLの哲学と整合**しています。」

## Turn 4｜法人格問題の具体的回避策とClerk-Stripe連携の実装法則

**中原：** 「法人格の問題を実務レベルで整理します。」

「**パターンA：個人事業主としてStripe登録 → 法人化後に切り替え**。Stripeは個人事業主登録が可能で、後から法人アカウントに切り替えができます。切り替え時に顧客のサブスク情報を移行する作業が必要（Stripeサポートが対応）。法人化まで3ヶ月以内なら推奨。」

「**パターンC（推奨）：test_modeで全機能実装、法人設立後に即live_mode切替**。60日の制約下では最も現実的。開発期間中はtest_modeで完成度を100%にしておき、法人設立後にAPIキーを入れ替えるだけで本番稼働できます。」

**木村：** 「Clerk + Stripeの連携には**3つのデータの一致**を常に保つ設計が必要です。」

```
Clerk User（認証） ←→ Stripe Customer（課金） ←→ Supabase profiles（DB）

同期タイミング:
1. 新規ユーザー登録時 → Stripe Customerを自動作成 → profiles.stripe_customer_id に保存
2. サブスク変更時 → Stripe Webhook → Supabaseのプランフラグを更新
3. 解約時 → Stripe Webhook → Supabaseの制限フラグを即時更新

Webhook処理の法則:
- idempotency_key必須（重複処理防止）
- 失敗時のリトライはStripeが自動で3日間実施
- 処理順序の保証が必要（created_atで並び替え）
```

「この3点の一致が崩れると、**『課金はされているのに機能が使えない』または『解約済みなのに機能が使える』**という致命的なバグが発生します。」

**みさき：** 「え待って、Webhookって何ですか？つまりStripeが「払ってもらえたよ」ってCOCOSiLに知らせる仕組みってこと？」

**木村：** 「完璧な翻訳です。Webhookは『Stripeからの通知』です。課金成功・失敗・解約・プラン変更をリアルタイムでCOCOSiLのサーバーに通知して、自動的にユーザーの権限を切り替える仕組みです。」

## Turn 5｜決済プロバイダ選定の原則と60日実装アクション

**木村：** 「COCOSiL V2決済選定の3原則を集約します。」

```
【COCOSiL V2 決済設計 3原則】

① Stripe First, Migrate Never
   決済基盤の移行コストは新規実装の3倍以上。
   「将来の選択肢を残す」ために最初からStripeを選ぶ。
   Pay.jp/Paddleの「今の楽さ」は将来のロックインコストで相殺される。

② Transparent by Default（透明性をデフォルトに）
   Stripe Billing Portalでユーザーが自分でプラン変更・解約・履歴確認できること。
   「囲い込み感」のない決済体験がCOCOSiLの哲学と整合する。

③ Auth-Billing Consistency（認証と課金状態の一致）
   Clerk User ⟺ Stripe Customer ⟺ Supabase profiles の3点が常に同期している状態を保つ。
   Webhook処理のidempotencyを確保し、「課金されているのに使えない」を防ぐ。
```

**中原：** 「原則を60日のアクションに落とします。」

```
【60日 Stripe実装アクション】

Week 1-2（Day 1〜14）: 基盤構築（test_modeで完全実装）
  1. Stripe Dashboardにtest_modeで登録（個人事業主名義でOK）
  2. pnpm add stripe @stripe/stripe-js でインストール
  3. Stripe Customer作成 + Clerk User ID を metadata に記録
  4. 4プランのPrice（recurring）をStripe Dashboardで作成
  5. Webhook エンドポイント（app/api/stripe/webhook/route.ts）を実装

Week 3-4（Day 15〜28）: サブスク機能実装
  6. Stripe Payment Elementによる決済フォーム実装（Next.js App Router）
  7. プラン変更・解約・プロレーション処理の実装と単体テスト
  8. Stripe Billing Portal 設定（ユーザー自己管理機能）

Week 5（Day 29〜35）: 法人設立完了次第
  9. 法人名義でStripe本番アカウント切り替え
  10. 本番Webhookエンドポイントの設定・疎通確認

Phase 3（Day 43〜55）: 本番課金ローンチ
  11. test_mode → live_mode 切り替え（環境変数のキー入れ替えのみ）
  12. 最初の5人に手動でサブスク作成し、Webhook処理の動作確認
```

**みさき：** 「なんか最初『Stripeって外資系だし難しそう』って思ってたんですけど、日本語フォームもできるし、解約しやすくできるし、私が心配してた部分が全部解決されてるんですね〜！あと『Transparent by Default』って原則、めっちゃCOCOSiLっぽいじゃないですか！『自分の情報は自分のもの』って哲学とピッタリで、**COCOSiLの決済体験も哲学の一部になれる**って感じして最高だと思いました！」

## ✅ 議論まとめ：F7 決済プロバイダ選定

| 項目 | 方針 |
|------|------|
| **選定プロバイダ** | **Stripe** |
| **脱落理由（Pay.jp）** | Clerk公式連携なし・海外展開不可・プロレーション自前実装 |
| **脱落理由（Paddle）** | 法人格未取得での申込みリスク・プライシング柔軟性低 |
| **脱落理由（Square）** | サブスク管理が弱い・Clerk連携なし |
| **法人格問題の対策** | test_modeで全機能実装 → 法人設立後に即live_mode切替（パターンC） |
| **Clerk連携方式** | Clerk User ID を Stripe Customer metadata に保存、Webhookで3点同期 |
| **サブスク管理** | Stripe Billing（4プラン×月額・プロレーション自動・Portal自己管理） |
| **実装タイミング** | Day 1〜28でtest_mode実装完了、Day 29〜35で法人化後live切替、Day 43〜55でローンチ |
| **設計三原則** | Stripe First / Transparent by Default / Auth-Billing Consistency |
