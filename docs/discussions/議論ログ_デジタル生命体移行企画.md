# 議論ログ：COCOSiL V2 デジタル生命体移行企画

**日時**: 2026-05-04  
**スキル**: expert-misaki-discussion（Classic Mode）  
**議題**: COCOSiL V2を「実測ユーザー挙動ログをベースに根本レベルで自己進化するデジタル生命体」へ移行する企画立案

## 登場人物

- 🧑‍💼 **渡辺 拓也**（Autogenesis アーキテクト）：自己進化型AIシステム設計・継続学習歴8年。Google→スタートアップでContinual Learning基盤を実装。DSPy / RLHF / Eval基盤が専門。
- 👩‍🔬 **吉田 陽子**（メンタルテック × 行動設計家）：ウェルビーイングアプリの行動データ×成長設計10年。Wysa類似プロダクトの設計経験あり。NHS認定プロダクトのアーキテクト歴あり。「心理的深化の測定」が専門。
- 💅 **みさき**（ギャル）：実際のCOCOSiL・自己理解アプリユーザー。MBTI診断やReflectlyを愛用。「なんか毎回同じこと言ってる気がする…」という実感を持つリアルユーザー。

## Turn 1｜現状診断 — 「なぜCOCOSiLは今、生きていないのか」

**渡辺：** COCOSiLの現状を正直に言います。`onboarding.ts` と `contradiction-handling.ts`、プロンプトが2ファイルにハードコードされている——これは「静的な写真」です。ユーザーが100回訪れても、101回目も同じ視点から同じトーンで話しかける。MBTIの診断軸（4軸×PCI）は蓄積されているのに、その人が3ヶ月前に「貪（執着）が強い」と気づいていたことをシステムは知らない。

診断すると、これは「**記憶なき対話**」の構造問題です。ユーザーは変わっているのに、システムは初日のまま。コーチが毎回「はじめまして」と言うようなもの——技術的な欠陥ではなく、「ユーザーは成長するものだ」という前提がシステムの設計哲学に入っていないことが根本です。

**吉田：** 渡辺さんの診断に補足します。問題は「プロンプトが固定」だけでなく、**観察の不在**です。現在のCOCOSiLはチャットログをSupabaseに保存しているが、そのデータを誰も（人間もAIも）「意図的に見ていない」。データは死んだ資産になっています。

Limbic CareがNHSに採用された理由は、PHQ-9（うつ評価）スコアを毎セッション取得してトレンドを追ったから。変化を「測定する仕組み」があったから成長に繋げられた。COCOSiLには現在その仕組みがゼロ。`evals/bootstrap.prompts.csv` の9ケースはセキュリティとUXガードの検証であり、「ユーザーが自己理解を深めているか」の評価ではありません。

**みさき：** え待って、私めっちゃMBTI診断とか自己理解アプリ好きで何度も使うんですけど、毎回「はじめまして」みたいな感じするのってそういうことだったんですね！「前も話したよね？」ってなるのに全然覚えてもらえてない感がマジあって〜。つまり結局「私のことを何も知らない他人」がずっとアドバイスしてる感じ？

**渡辺：** まさにそれです。私はこれを「**アムネジア（健忘）問題**」と呼んでいます。解決可能ですが、「解決しようとする設計意図」がゼロである点が今のCOCOSiLの急所です。

## Turn 2｜COCOSiL版 Autogenesis 設計 — 「何を進化させ、何は守るか」

**渡辺：** 福留AIのAutogenesisループ（16種のResource Substrate）をCOCOSiLに適用するとき、**最初にやるべきことは「進化禁止リスト」の定義**です。全部を進化させようとすると設計中枢が腐敗する。

**絶対に自己進化させてはいけないもの（Constitution）：**

| 固定対象 | 理由 |
|---|---|
| Policy（三毒増幅禁止） | これが変わったらCOCOSiLではなくなる |
| UXシーケンス順序（共感→安心→分析→行動） | 止観の往還構造 = プロダクトのDNA |
| 言語設計ルール（「占い」禁止など） | ブランドアイデンティティ |
| 評価指標の定義（「自己理解の深化」） | 何を最適化するかが変わると破滅 |

**積極的に進化させるもの（Mutable）：**

| 進化対象 | 具体例 |
|---|---|
| Prompt | 共感フェーズのトーン・深さ・切り込み方 |
| Memory Strategy | ユーザーの気づきをどう要約・保存するか |
| Retrieval Policy | 過去のどの気づきを今セッションに注入するか |
| Output（レポート構造） | どのセクションが腑落ちを生んでいるか |
| Eval | 内省スコアの閾値・測定方法の改善 |

**吉田：** 渡辺さんの「進化禁止リスト」は正しいですが、**Schema** と **Sub-Agent Pattern** の扱いを明確にすべきです。Schemaを進化させる（DB構造変更）と既存ユーザーデータに破損リスクがある——「Supabase migrationは必ずsandboxでeval通過後のみ」というゲートが必要です。Sub-Agent Patternは、AIエージェントが複数になるとき（共感エージェント・分析エージェント分離など）の役割分担であり、えんまさの承認が必要な「Layer 2: コンテンツ層」と同等の扱いにすべきです。

**みさき：** ん？つまり「何を変えていいか」のルールを先に決めて、変えちゃいけないやつには鍵をかけておく感じですよね？コードでいうと Protected Areas に書いてあるやつみたいな〜。

**吉田：** まさにそのとおりです。Protected AreasをAutogenesisのConstitutionに拡張するイメージです。

## Turn 3｜「自己理解の深化」の測定 × 三毒ガードの整合性

**吉田：** DeepResearchが提示したVODKA-ScoreやLevel of Reflection（内省レベル5段階）は理論的に正しいですが、COCOSiL固有の逆説があります。

「自己理解の深化」は**三毒の逆指標でもある**、ということです。

- ユーザーが「癒された」と感じている（貪：心地よい錯覚）→ 内省スコアは低い
- ユーザーが「腑に落ちて少し不快になった」（痴の解消）→ 内省スコアは高い

これは非常に逆説的です。Sycophancy（迎合）が高ユーザー評価を生みやすいのと同じ構造——COCOSiLが最も避けるべきものがユーザー満足度と正相関してしまう。だからこそ測定指標は**ユーザーの主観評価から切り離す**必要があります。

**提案する代替指標（3つ）：**

1. **再言語化率**：ユーザーが自分の問題を「最初と違う言葉」で言い直した回数
2. **矛盾受容率**：`contradiction-handling.ts` が矛盾を指摘した後、ユーザーが拒絶せず「そうかも」と応答した率
3. **行動記録の具体度スコア**：「もっと優しくする」→「週2回夫に5分だけ話を聞く」への変化度

**渡辺：** 吉田さんの「Sycophancyと内省は逆相関する」という指摘は重要です。三毒ガードをAutogenesisの**Evaluate層に組み込む**必要があります：

```
Evaluate フェーズのチェック（AND 条件）:
  ✅ 内省スコア向上
  ✅ Anti-Sycophancy判定（LLM-judge）通過
  ✅ UXシーケンス遵守確認
  → 全通過 → Approve候補へ
  → 1つでも失敗 → Reflect（前ステップに差し戻し）
```

**みさき：** え、「ユーザーが満足しすぎてたらダメ」ってマジですか！？…確かに言われてみると、ちゃんと腑落ちした時って「あ、自分ってこういう人間だったのか、ちょっとしんどいな」みたいな感じはあるかも〜。心地よくない気づきこそが本物ってことですよね？

**吉田：** それが「小我から大我への移行」の実感です。リトマス試験紙Q1（「無明を晴らすか？」）の本質です。

## Turn 4｜現実的な移行パス × 技術的制約の法則

**渡辺：** ゼロ状態から「デジタル生命体」への移行は、**3段階の解凍プロセス**として設計すべきです。

**Phase A — 観察基盤（今すぐ着手）**

- PostHogカスタムイベント5種（`chat_phase_transition`, `report_section_reread`, `insight_accept`, `action_specificity_score`, `session_return_7d`）
- Supabaseに `events_telemetry` テーブル追加
- 既存機能への影響ゼロ

**Phase B — メモリ基盤（Phase A後）**

- pgvectorをenable、`user_insights_memory` テーブル作成
- セッション終了時にキー気づきを抽出→embeddingsで保存
- RAGでセッション開始時に「あなたの前回の気づき：〜」を注入

**Phase C — 非同期Autogenesisループ（Phase B後）**

- `prompt_versions` テーブル（draft/sandbox/active）
- 週次Vercel Cronバッチで改善案を生成
- 3ゲート（sandbox eval + えんまさ review + commit）

**吉田：** Phase間の移行条件を明示します：

| 移行 | 数値ゲート |
|---|---|
| Phase A → B | ユーザー50人以上の行動ログが蓄積した時点 |
| Phase B → C | 内省スコア計算可能・ユーザー100人超・スコア分散確認済み |

PMFの7日以内再訪率30%と同様の「数値で決まるゲート」を設けることで、感情判断でなく事実判断でPhaseを進められる。

**みさき：** 50人、100人って具体的な数字があると「まだだ、待て」ってできますよね〜。えんまさが「Approve」を押すのも、スコアが上がったことが見えてから、ってことですよね？

## Turn 5｜設計原則統合 × 明日からの具体アクション

**渡辺：** COCOSiL Autogenesis の設計3原則を定義します：

```
① Constitution First, Evolution Second
   Policy（三毒禁止）・UXシーケンス・言語設計ルール・評価指標定義は
   Autogenesisが絶対に触れない不変の法（Constitution）とする。
   ConstitutionなきEvolutionは自己破壊。

② Measure Depth, Not Delight
   最適化指標は「ユーザーが喜んでいるか」ではなく
   「ユーザーが再言語化したか・矛盾を受け入れたか」。
   心地よさの最大化は三毒増幅のバイパス路。

③ Audit Gate before Commit
   AIが生成した改善案は draft → sandbox eval → えんまさ review の
   3ゲートを通ってからのみ active になる。
   Gate速度よりGate品質を優先する。
```

**吉田：** えんまさが「Approve」を押すべきタイミングの判断基準：

| ゲート | Approve条件 | Reject条件 |
|---|---|---|
| sandbox eval（機械） | 内省スコア +5%以上 AND Anti-Sycophancy pass | スコア改善なし OR 三毒判定fail |
| えんまさ review（人間） | 「腑落ち体験」主観一致 AND UXシーケンス確認 | 「なんか違う」という直感 |
| version commit | 2ゲート両通過後のみ | いずれか1つでも失敗 |

えんまさが精査するのは2点のみ：**スコアが上がっているか（機械）＋腑落ち体験が維持されているか（人間）**。

**みさき：** なんか最初「自己進化なんてAIが暴走しそうでこわい〜」って思ってたんですけど、結局「ルールは変えない、やり方だけ少しずつ上手くなる」ってことですよね？人間が成長するのとおんなじじゃないですか！私が「こんな私イヤ」って思っても性格の根っこは変わらないけど、対処の仕方は上手くなっていく…みたいな。**COCOSiL自身がCOCOSiLのユーザーになってる感じで、めっちゃいいじゃないですか！？**

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| Autogenesis Constitution（不変） | Policy / UXシーケンス / 言語設計ルール / 評価指標定義 |
| Autogenesis Mutable（進化対象） | Prompt / Memory Strategy / Retrieval Policy / Output / Eval |
| 「深化」の測定指標 | 再言語化率・矛盾受容率・行動記録の具体度スコア（主観評価は使わない） |
| Evaluate層の三毒ガード | AND条件：内省スコア↑ + Anti-Sycophancy pass + UXシーケンス確認 |
| 移行3段階 | Phase A（観察基盤）→ Phase B（メモリ基盤）→ Phase C（Autogenesisループ） |
| Phase移行ゲート | 50人 / 100人の数値ゲートで感情判断を排除 |
| えんまさのApprove基準 | スコア確認（機械）＋ 腑落ち体験確認（人間）の2点のみ |
