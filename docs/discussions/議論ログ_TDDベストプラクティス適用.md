## Turn 1｜現状診断 — COCOSiL V2ハーネスとレポートのギャップを照合する

**高橋 俊介：**

このレポートのSection 5（ギャップ分析）をCOCOSiL V2の現状ハーネスに当てると、構造が見えます。現状の検証コマンドは `pnpm typecheck` / `pnpm lint` のみ——これはレポートで言う「決定論的な床」の最薄層しかない状態です。

レポートが「優先度：高」と分類するギャップを3つ、COCOSiL V2固有の文脈で読み替えます：

| レポートのギャップ | COCOSiL V2での具体例 |
|---|---|
| プロンプト回帰テストの欠如 | `lib/prompts/` の共感フェーズ変更が診断品質を落としても気づかない |
| ツール引数バリデーションなし | OpenAI APIへの呼び出し引数がLLM生成のまま渡り、実行時クラッシュ |
| LLMをブラックボックスで扱う | 同一ユーザー入力で異なる診断結果が返っても検知不能（壊れ方①） |

**根本診断：** COCOSiL V2は「静的解析で床を敷いた段階」であり、レポートのPhase 0（Quick Wins）すら未着手です。今週G9解消でプレビューURL + PRチェックリストを整備したことで「③UX断絶」の人間ループは確保できた。次に整備すべきは「①ロジック破綻」と「②状態不整合」の自動検知です。

---

**松田 綾子：**

高橋さんの診断に同意しつつ、COCOSiL V2特有の制約を加えます。

このプロダクトのLLM依存機能は2つです。**F2（統合レポート: Gamma API）** と **F3（共感AIチャット: OpenAI）**。レポートのテスト分類学に当てはめると：

```
F2レポート生成:
  - プロンプト構築 → 単体テスト可能（純粋関数）
  - Gamma APIレスポンス → レスポンスパーサーテスト可能
  - レート制限/障害時のフォールバック → 統合テスト対象

F3共感チャット:
  - 3フェーズ（共感→安心→分析）のフロー制御 → コンポーネントテスト対象
  - 「三毒増幅していないか」の判定 → 評価（Eval）対象
  - 禁止語彙の混入 → 単体テスト可能（文字列チェック）
```

**重要な気づき：** COCOSiL V2の「設計中枢 5問チェック」は事実上のEvalルーブリックです。GitHubがDeepEvalで導入した5次元採点に相当する。これを手動チェックから自動Evalに昇格させる設計が、このプロダクトに最も適した方向性だと思います。

## Turn 2｜選択肢の発散 — どのパターンをCOCOSiL V2に適用するか

**高橋 俊介（評価軸の提示）：**

COCOSiL V2への適用判断軸を2つ立てます。

**軸1：決定性** — テスト対象がLLMに依存するか（確率的）、しないか（決定論的）
**軸2：導入コスト** — 3人チーム・MVP段階での実装難易度

| パターン | 決定性 | 導入コスト | COCOSiL V2での適用先 |
|---|---|---|---|
| Pydantic バリデーション | 決定論的 | 低（1日） | 全APIルートの入出力型 |
| プロンプト構築関数テスト | 決定論的 | 低（1日） | `lib/prompts/` の禁止語彙・必須キーワード確認 |
| LLM-as-a-Judge Eval | 確率的 | 中（3日） | F3共感チャット3フェーズの品質評価 |
| コントラクトテスト（Pact） | 決定論的 | 高（1週間） | F2 Gamma API連携（Over Engineering） |
| シミュレーション環境 | 決定論的 | 高（1週間） | F3フロー全体のE2E（Sprint 3以降） |

**COCOSiL V2への推奨順：** まず「決定論的・低コスト」から始め、LLM Evalは `lib/prompts/` に実装が入ったタイミングで導入する。シミュレーション環境はOver Engineering——今は必要ない。

---

**松田 綾子（具体的手段の配置）：**

3つの段階に整理します。COCOSiL V2スタック（Next.js/Vitest/Supabase）で実際に動くものだけ並べます。

```
【Phase 0: 今すぐ（G1解消と同時）】
  pnpm add -D vitest
  対象: lib/diagnostics/ の計算ロジック（MBTI・六星占術）
  → 同一入力 → 同一出力 の決定論的保証
  → レポートの「壊れ方①ロジック破綻」を自動検知

【Phase 1: F3実装時（共感チャット）】
  lib/prompts/ に Vitest テスト追加
  - 禁止語彙が含まれていないか（"占い", "霊感" 等）
  - 共感フェーズのプロンプトに必須キーワードが存在するか
  → レポートの「プロンプト構築関数テスト」を適用

【Phase 2: F2/F3安定後（Eval導入）】
  pnpm add -D promptfoo
  設計中枢5問をEvalルーブリックとして自動評価
  → Q2「三毒を増幅していないか」をLLM-as-a-Judgeで自動採点
```

## Turn 3｜要件・範囲の確定 — 実需をパターン化して優先度へ

**💅 みさき（実需持ち込み）：**

実際のユーザーとして怖いのって3つあって〜。① 診断結果が毎回変わる、② アプリが「あなたは〇〇型です」って言った後に全然違う提案してくる（矛盾してる感じ）、③ 途中で「エラーが発生しました」ってなって診断が消える。この3つが起きたら絶対もう使わないですよね〜。

**高橋 俊介（パターン化）：**

みさきさんの3つ、実はレポートのテスト分類と完全に対応しています。

```
みさきの恐怖 → テストパターン → COCOSiL V2での具体実装

① 診断結果が毎回変わる
  → 壊れ方①ロジック破綻
  → Vitest: lib/diagnostics/ のunit test（同一入力→同一出力）

② 矛盾した提案が来る
  → 壊れ方③UX断絶（レスポンス構造がUIと噛み合わない）
  → PRチェックリスト「レスポンス構造が型定義と一致」（整備済み）
  → F3実装時: プロンプト構築テスト（フェーズ間の文脈引き継ぎ確認）

③ 途中でエラーになる
  → 壊れ方②状態不整合（Supabase RLS + Clerk JWT）
  → Vercelプレビュー + 手動ログ確認（G9解消で整備済み）
  → Phase 1でフォールバック実装テストを追加
```

**信頼の3条件（評価軸として定義）：**
- **一貫性**（同じ入力→同じ結果）
- **整合性**（言葉と提案が矛盾しない）
- **継続性**（途中で消えない）

**松田 綾子（優先度整理）：**

| 恐怖 | 対応するテスト | 優先度 | タイミング |
|---|---|---|---|
| 診断結果が毎回変わる | Vitest unit test（lib/diagnostics/） | 🔴 F2前に必須 | G1解消時 |
| 矛盾した提案 | プロンプト回帰テスト | 🔴 F3実装時に必須 | Phase 1 |
| 途中でエラー | フォールバックテスト + RLSログ確認 | 🟡 F2/F3統合後 | Phase 2 |
| 禁止語彙混入 | 文字列チェックテスト | 🔴 F3実装時に必須 | Phase 1 |

## Turn 4｜技術深掘り — COCOSiL V2固有の制約から法則を抽出する

**松田 綾子：**

COCOSiL V2固有の制約を踏まえた実装方針を3点示します。

**1. 設計中枢5問 → Evalルーブリック化（最重要）**

```yaml
# promptfooconfig.yml（Phase 2で導入）
prompts:
  - lib/prompts/chat-phase1-empathy.ts

providers:
  - openai:gpt-4o-mini  # 安価なモデルでEval

tests:
  - vars:
      userInput: "上司に怒鳴られてつらい"
    assert:
      - type: llm-rubric
        value: "Q2: 三毒（怒り・欲・混乱）を増幅する表現を含まないか"
      - type: not-contains
        value: "占い"
      - type: contains
        value: "共感"
```

**2. Supabase RLS + Clerk JWT の統合テストは「人間ループ」が最速**

レポートはTestContainersでの完全自動化を提案していますが、COCOSiL V2のチーム規模ではOver Engineering。Vercelプレビュー + えんまさの目視確認が費用対効果最大。Phase 2でSupabase branch機能が安定してから自動化を検討。

**3. Gamma API（F2）のフォールバックテスト**

```typescript
// vitest でモックする場合
import { vi } from 'vitest'
import { generateReport } from '@/lib/prompts/report'

vi.mock('@/lib/gamma', () => ({
  gammaClient: { generate: vi.fn().mockRejectedValue(new Error('Rate limit')) }
}))

test('Gamma API障害時はフォールバックレポートを返す', async () => {
  const result = await generateReport({ userId: 'test-user' })
  expect(result.fallback).toBe(true)
  expect(result.content).toContain('一時的にご利用できません')
})
```

**高橋 俊介（法則化）：**

COCOSiL V2に適用すべき普遍的法則は2つです。

```
法則1: 「決定論的なものは機械に、確率的なものは人間に」
  LLMが絡む部分（F2/F3の出力品質）を自動化しようとすると
  Eval環境の維持コストがチーム規模を超える。
  まず決定論的に検証できる部分（型・プロンプト文字列・計算ロジック）を
  機械に任せ、「この出力は仕様通りか」はえんまさが判断する。

法則2: 「プロンプトはコードと同格に扱う」
  lib/prompts/ の変更は型変更・API変更と同等のリスクを持つ。
  PRチェックリストに「プロンプト変更を含む場合は禁止語彙テスト追加」を
  追記することで、プロンプト回帰を最小コストで防げる。
```

## Turn 5｜方針統合・結論 — COCOSiL V2向け設計原則とロードマップ

**高橋 俊介（設計原則）：**

レポートの知見をCOCOSiL V2の哲学・制約・スタックに適応した設計原則を3つ定義します。

```
① Deterministic First, Eval Later（決定論的を先に、Evalは後から）
  プロンプト文字列・計算ロジック・型バリデーションを先にVitest化する。
  LLM出力の意味的品質評価（Eval）は lib/prompts/ が安定してから導入する。
  「今日テストできないものは今日テストしない」——Over Engineeringを防ぐ唯一の盾。

② Design-Center as Rubric（設計中枢をEvalルーブリックとして使う）
  Q1〜Q5の5問チェックはすでにEvalの採点軸として完成している。
  Phase 2で promptfoo を導入するとき、このルーブリックをそのまま
  promptfooconfig.yml に転記するだけで、COCOSiL固有のEvalが完成する。

③ Prompt as Code（プロンプトをコードと同格に扱う）
  lib/prompts/ の変更は必ずVitest文字列テストをセットで書く。
  「禁止語彙なし・必須フェーズキーワードあり」の2条件を最低限として。
  プロンプトの変更がCI（GitHub Actions）で検知されない状態を終わらせる。
```

**松田 綾子（ロードマップ・次の一手）：**

```
🔴 G1解消時（F2実装開始前）:
1. pnpm add -D vitest
2. lib/diagnostics/__tests__/ を作成
   - MBTI 16タイプの境界値テスト（同一入力→同一出力）
   - 六星占術の計算テスト（壊れ方①を自動検知）
3. ci.yml に test ジョブを追加

🟡 F3実装時（共感チャット）:
4. lib/prompts/__tests__/ を作成
   - 禁止語彙テスト: expect(prompt).not.toContain("占い")
   - 必須キーワードテスト: expect(prompt).toContain("共感")
5. PRチェックリストに「プロンプト変更 → テスト追加必須」を追記

🟢 F2/F3安定後（Sprint 3以降）:
6. promptfoo 導入（設計中枢5問をEvalルーブリックとして設定）
7. Gamma APIフォールバックテスト
8. Supabase branch + ステージング環境（壊れ方②の自動化）
```

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| 根本問題 | COCOSiL V2はPhase 0（Quick Wins）未着手。決定論的テストが皆無 |
| 設計原則①（Deterministic First） | 計算ロジック・型・文字列を先にVitestで自動化。LLM Evalは後から |
| 設計原則②（Design-Center as Rubric） | 設計中枢5問をpromptfooルーブリックとして転記するだけでEval完成 |
| 設計原則③（Prompt as Code） | `lib/prompts/` の変更にはVitest文字列テストを必ずセットで書く |
| G1解消時の実装 | Vitest導入 + `lib/diagnostics/__tests__/` 作成 + ciジョブ追加 |
| F3実装時の実装 | `lib/prompts/__tests__/` 作成（禁止語彙・必須キーワードテスト） |
| Sprint 3以降 | promptfoo + Gamma APIフォールバック + Supabaseステージング |
| Over Engineeringの境界 | Pact・TestContainers・シミュレーション環境は現時点で不要 |
