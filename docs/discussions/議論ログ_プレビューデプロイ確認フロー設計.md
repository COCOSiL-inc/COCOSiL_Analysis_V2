## Turn 1｜現状の対立：Layer B確認とは何を確認しているのか

**木村：** 岡田さんの「早期リリース論」、正面から受けます。私が主張している「Layer B完全確認」が何を指すのか、まず具体化します。Supabase + Clerk構成で本番障害の9割は「プレビューと本番で環境変数が違う」ことから生まれます。具体的には3点です。①`NEXT_PUBLIC_SUPABASE_URL`がプレビューで本番DBを向いている（RLSが本番ユーザーデータに効いてしまう）、②ClerkのJWT `issuer`がlocalhost想定のまま（プレビューURLで認証が通らない）、③SupabaseのJWT秘密鍵がClerk側と不一致（silent 401で診断結果が返らない）。これを「確認した」と言えるのは、**プレビューURLで実際にClerkログイン→診断完了→DBへの書き込み確認**までを通した場合だけです。つまりLayer B確認は「URLを開いて動いた」ではなく「RLS+JWT+書き込みの3ループが成立した」という意味です。

**岡田：** 木村さんの定義に同意します。ただし「3ループ確認」をえんまさに毎PR依頼するのは現実的じゃない。木村さんが挙げた3つの障害パターンのうち、①と③は**デプロイ設定の一回限りの作業**で解決できます。Vercelの「Environment Variables」でプレビュー環境に専用Supabase URLを固定すれば、プレビューが誤って本番DBを向くことはなくなります。問題は②のClerk `issuer`設定で、これだけがプレビューURLごとに変わる。Clerkの「Allowed Origins」にワイルドカード（`*.vercel.app`）を追加すれば、新規プレビューURLでも認証が通ります。すなわち**一回の設定作業**で、以後のすべてのPRプレビューで「ClerkログインできてSupabaseに書ける」状態を保証できます。

**みさき：** え待って、つまりプレビューURLを開いたら「普通に使える状態」にするための設定を、最初に一回やればいいってこと？それ最初からやっておけばよかったんじゃ？

## Turn 2｜設定の具体化：えんまさがプレビューで確認できる最小構成

**木村：** みさきさん、その「最初に一回やる」が何かを技術的に具体化します。Vercelダッシュボードで設定する環境変数は3種類あります——Production、Preview、Development。現状おそらくProductionにしか設定がない。えんまさのプレビュー確認を成立させるには、以下をPreview環境に設定する必要があります。

| 変数名 | Preview値の考え方 |
|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabaseの**ステージングプロジェクト**のURL（本番と分離） |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | ステージングプロジェクトのanon key |
| `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` | Clerk本番instanceのpublishable key（ステージング用instanceがあれば別） |
| `CLERK_SECRET_KEY` | 同上 |

加えてClerk管理画面で`Allowed redirect URIs`に `https://*-cocosil.vercel.app`を追加します。これでプレビューURLが毎回変わっても認証フローが切れません。この設定は**1回限り**、所要30分です。

**岡田：** 木村さんの設定リストは正確です。私が追加したいのは「ステージングプロジェクトの範囲」。Supabaseで新規プロジェクトを作るとコストが発生する懸念がありますが、**無料枠の第2プロジェクト**として作成し、migrationsだけを本番から流し込む方法が現実的です。`supabase db push --db-url <staging-url>`で本番スキーマを複製、RLSポリシーも同時に適用されます。これでえんまさは「本番と同じスキーマ・RLSが効いたDBで、ClerkのJWT認証が通るかどうか」をプレビューURLで確認できます。GO条件を「プレビューで診断1件完走できた」に絞れば、えんまさの確認は**3分以内**で完了します。

**みさき：** めっちゃ整理された！てか「ステージング」と「本番」ってDBが2個あるってこと？ユーザーのデータはどっちに入るんですか？

## Turn 3｜えんまさの確認フロー設計：3段階の抽象的要件原則

**木村：** みさきさんの本質的な質問に答えます。ステージングDBには**えんまさのテスト用データのみ**が入ります。本番ユーザーデータは一切混入しません。これを保証するのがVercelの環境変数分離です。では「えんまさの確認フロー」を3段階で定義します。

```
Stage 1: ローカル確認（開発者）
  └── pnpm typecheck && pnpm build が通ること
  └── 担当：ヒラメ（AIエージェント）

Stage 2: プレビュー確認（えんまさ）
  └── Clerk でログイン → 診断完了 → ステージングDBに結果が書かれること
  └── 所要時間：3分以内
  └── 確認ツール：プレビューURL（PR に自動付与）

Stage 3: 本番GO（えんまさ）
  └── Stage 2 通過後に「LGTM」コメント → main マージ許可
```

この原則は「ビルドが通れば完了」を禁じた現行ハーネス設計と整合します。**機械で確認できること（typecheck/build）は機械に委ね、人間にしか判断できないこと（腑落ち体験・UXシーケンス）だけをえんまさに渡す**——これが確認フローの設計原則です。

**岡田：** 木村さんの3段階設計に賛成。私が優先度をつけるとすれば、**今週中に実装すべき順序**はこうです。

🔴 P0（今日）：VercelのPreview環境変数にステージングSupabase URLを設定
🔴 P0（今日）：ClerkのAllowed redirect URIsにワイルドカード追加
🟡 P1（今週）：ステージングSupabaseプロジェクト作成 + migration流し込み
🟢 P2（次週以降）：PRテンプレートに「プレビューURL確認チェックボックス」追加

P0の2項目は**設定変更のみ、コード変更不要**です。これを今日やれば、次のPRからえんまさはプレビューURLでClerkログインができます。ただしステージングDBができるまではP0段階では本番DBを見てしまう——そこは木村さんの懸念通りです。

**みさき：** なんか「P0今日やる」ってなったの、めっちゃ具体的になった感じする！てか「腑落ち体験だけえんまさが確認」ってのはマジ大事な気がする〜。機械には「腑落ちしたか」は判断できないじゃないですか。

## Turn 4｜リスクと法則：RLSとJWT分離の不変制約

**木村：** Turn 3のP0→P1の移行に技術的リスクが1つあります。P0段階（ステージングDBがない状態）でプレビューが本番DBを向く期間は、**RLSが本番ユーザーデータに対して発動します**。これは「データが漏れる」という問題ではありません——RLSはClerkのJWTを検証してuser_idベースで行を制限するので、えんまさのテストアカウントには他ユーザーのデータは見えません。ただし**書き込みテストが本番DBに混入する**リスクがあります。診断1件分のダミーデータが本番テーブルに入る。これを防ぐには、ステージングDB完成まではプレビューで「診断完走テスト」はしない——読み込み確認（画面表示・認証通過）だけに留める、という運用ルールが必要です。

**岡田：** 木村さんのリスク指摘は正確ですが、**期間は最大3〜4日**です。SupabaseのステージングプロジェクトはUIから5分で作れて、`supabase db push`は30分で完了します。リスク受容の判断軸として、私はこう整理します——「本番DBへのダミーデータ混入」は技術的に除去できる（後でDELETE）、しかし「えんまさの確認フローが未整備のままリリースすること」は事業的に取り返せない。優先度は後者の解消です。ここで抽象法則として言えるのは：**「完璧な分離を待つより、リスクが取り戻せる順序でGOを出す」がスタートアップの確認フロー設計原則**です。

**みさき：** 「後でDELETEできるなら混入してもいい」ってなんか潔い判断だ〜。でも木村さん、「読み込み確認だけにする」って、えんまさが知らずに診断テストしちゃったらどうなるんですか？

## Turn 5｜設計原則の統合と今日からの具体アクション

**木村：** みさきさんの指摘で重要な運用上の欠落が明確になりました。「えんまさが知らずに書き込みテストをしてしまうリスク」を防ぐには、**PRコメントに確認フェーズを明示する**必要があります。「このPRはP0段階のため、ログイン確認・画面表示確認のみお願いします（診断完走テストはステージングDB整備後）」という一文をPRテンプレートに入れる。これで運用カバーできます。設計原則として3点に集約します。

> **① Separate by Environment Variable, Not by Code**
> RLS・JWT分離の信頼性は環境変数の分離で担保する。コードを変えなくても設定が正しければ環境は分離される。

> **② Human Confirms What Machine Cannot**
> typecheck / build / RLS適用は機械に委ねる。「腑落ち体験」「UXシーケンスの自然さ」だけをえんまさが確認する。3分以内に完了する確認だけをヒトに依頼する。

> **③ Stage Risk, Don't Eliminate It**
> 完全なリスクゼロを待つより、リスクが取り戻せる順序でフェーズを切る。P0→P1の移行期間はルールで補完する。

**岡田：** 原則を受けて、今日からの具体アクションを5項目で示します。

1. **今日（30分）**：Vercel管理画面 → Settings → Environment Variables → `Preview`タブに`NEXT_PUBLIC_SUPABASE_URL`等を本番値で一時設定（P0運用開始）
2. **今日（10分）**：Clerk管理画面 → Allowed redirect URIs → `https://*.vercel.app`を追加
3. **今週中（1時間）**：Supabaseで`cocosil-v2-staging`プロジェクト作成 → `supabase db push`でmigrationsを流し込み
4. **今週中（30分）**：VercelのPreview環境変数をステージングURLに更新（P1移行完了）
5. **次PR（10分）**：`.github/pull_request_template.md`に「プレビュー確認フェーズ（P0/P1）」チェックボックスを追加

アクション1+2だけで、**次のPRからえんまさはプレビューURLでClerkログインができる**。岡田の早期リリース論と木村のLayer B確認論の両方を満たす最短経路がこれです。

**みさき：** なんか最初「Layer B確認ってどれだけ大変なん」って思ってたけど、要は「設定2個を今日やる」だったんですね。それなら「確認できる環境を待つ」より「今日設定して次のPRから確認開始」のほうがマジ現実的〜！えんまさが3分でOK出せる状態、めっちゃ重要だと思いました。
