# COCOSiL Spec-Sync スキル

仕様変更をCOCOSiLのドキュメント群に正確・網羅的に伝播させるスキル。

## 設計3原則

**① Propagation over Perfection（伝播の網羅性を精度より優先する）**
漏れなく影響先を列挙し、「更新不要」の判断もAIが明示する。網羅性が高ければ人間が補正できる。逆は成立しない。

**② Layer-Aware Execution（Layer構造に従って実行範囲を変える）**
Layer 1（保護）は警告のみ、Layer 2（えんまさ承認必須）は提案のみ、Layer 3（AI委任可能）は自動更新まで実行する。

**③ Reverse Traceability（双方向の伝播を行う）**
「コンセプト → AGENTS.md → コード → ドキュメント」の双方向を追跡する。`lib/constitution/` とドキュメントが矛盾する場合、コードを正として警告する。

---

## Step 1：変更内容の入力と確認

ユーザーから以下の情報を受け取る（不明な場合は質問する）：

1. **変更内容**：「何を変えたか / 変えようとしているか」（自然言語 or ファイルパス）
2. **変更前の状態**：任意。なければスキルが現状ファイルを読み込んで判断する

---

## Step 2：変更パターンの自動判定

以下のマッピングで変更をP1〜P5に分類する。複数該当する場合は全て列挙する。

| パターン | キーワード・ファイルパス | 具体例 |
|---|---|---|
| **P1: 哲学更新** | `docs/input/concepts/COCOSiL設計中枢.md`、「設計3原則」「三毒」「無明」「PMF仮説」 | 設計中枢の原則追加・UXシーケンスの意味変更 |
| **P2: 禁止語変更** | `lib/constitution/banned-words.ts`、「禁止語」「使ってはいけない言葉」 | 新しい禁止語の追加・既存禁止語の削除 |
| **P3: UXシーケンス変更** | `lib/constitution/ux-sequence.ts`、「共感→安心→分析→行動」の順序変更 | フェーズ名変更・順序変更・フェーズ追加 |
| **P4: 機能スコープ変更** | 要件定義書・`docs/output/requirements/`、「機能追加」「機能削除」「スプリント変更」 | F1/F2/F3の仕様変更・新機能追加 |
| **P5: API/DB設計変更** | `app/api/`・`supabase/`・`lib/types/`・`lib/diagnostics/` | テーブル追加・APIエンドポイント変更・型定義変更 |

**判定が曖昧な場合：**
- P1 vs P3の境界：`lib/constitution/ux-sequence.ts` が変わるならP3、哲学文書のみならP1
- P4 vs P5の境界：要件定義書に先に書かれているならP4、コードが先行しているならP5
- 複数パターンが重なる場合：両方に分類して伝播先を合算する

---

## Step 3：影響ドキュメントの列挙

パターン別の影響ファイルを `references/propagation-matrix.md` から引く。
各ドキュメントの現在の記述をReadツールで読み込み、更新が必要かを判定する。

**「更新不要」の場合も明示すること**（みなして省略しない）。

---

## Step 4：Layer判定と実行範囲の決定

影響する各ファイルについて、以下のLayer判定を行う：

```
Layer 1（絶対保護）— 以下ファイルは変更禁止、警告のみ
  - supabase/migrations/**
  - .env / .env.local / .env.production
  - proxy.ts 内の clerkMiddleware 設定
  - lib/clerk/**
  → 出力：「⚠️ Layer 1保護対象：手動で対応してください」

Layer 2（えんまさ承認必須）— 提案を生成、実行は停止
  - lib/prompts/**
  - lib/data/**
  - lib/diagnostics/**
  - lib/constitution/**（Constitution変更はえんまさ承認後にコミット）
  - AGENTS.md §0（プロダクト哲学）
  - language-design/SKILL.md（禁止語・ブランド言語）
  → 出力：「🟡 Layer 2：以下の変更案を えんまさ に確認後、実行してください」

Layer 3（AI委任可能）— 更新案を生成し、実行まで行う
  - docs/output/requirements/**
  - docs/harness/**
  - cocosil-domain/SKILL.md（機能スコープ・技術スタック部分）
  - AGENTS.md §3/§5/§7（技術スタック・開発環境・保護エリア）
  - docs/README.md
  → 出力：「✅ Layer 3：自動更新可能」
```

---

## Step 5：更新案の生成

各影響ファイルについて以下の形式で出力する：

```markdown
## 変更種別：P[N] - [パターン名]

### ✅ 必須更新（自動実行可能・Layer 3）

#### `[ファイルパス]`
**変更すべきセクション：** §[セクション名] / [セクション見出し]
**現在の記述：**
> [現在の記述を引用]

**更新案：**
> [更新後の記述]

---

### 🟡 必須更新（えんまさ確認後に実行・Layer 2）

#### `[ファイルパス]`
**変更すべきセクション：** §[セクション名]
**更新提案：**
- [箇条書きで変更内容を提案]
承認者：**えんまさ**

---

### 📋 推奨更新（任意・影響度: 中）

- `[ファイルパス]`：[変更推奨の理由と概要]

---

### 🔒 不変確認（変更禁止）

- `[ファイルパス]`：[変更してはいけない理由]
  → 今回の変更との関係：[関係の説明]

---

### ✏️ 更新不要

- `[ファイルパス]`：[理由]（今回の変更と無関係）

---

### PRチェックリスト追記

**Gate [1/2]（[担当者]）追加チェック：**
- [ ] [追加チェック項目]
```

---

## Step 6：COCOSiL特有の整合性チェック

更新前後に以下の整合性を確認する：

### 禁止語チェック
更新案に以下の語が含まれていないか確認する（UIコピー・プロンプト・エラーメッセージのみ）：
- **禁止**：占い、鑑定、運勢、占い師、当たる、霊感、霊視
- コード内変数名・DBカラム名は例外

### UXシーケンス整合性チェック
更新内容が「共感→安心→分析→行動」の順序を維持しているか確認する。
順序を崩す変更提案は警告を出す。

### Constitution整合性チェック
`lib/constitution/` が存在する場合、更新後のドキュメントとの矛盾を確認する。
矛盾がある場合は「コードを正」として警告する。

## 他スキルとの連携フロー

```
P5（API/DB変更）を含む場合:
  先に security-sensitive-change-review スキルを実行してから spec-sync へ

P1/P2/P3（哲学/言語/UX変更）を含む場合:
  spec-sync で伝播先を列挙した後、
  変更の意味的妥当性が不明確なら expert-misaki-discussion で議論する

P4（機能スコープ変更）を含む場合:
  要件定義書の更新が大規模なら requirements-doc-creator と連携する
```

## 参照ファイル

## よくあるミスパターン（NEVER）

| ミス | 正しい対応 |
|---|---|
| 影響ドキュメントを列挙せず、1ファイルだけ更新する | 必ず propagation-matrix.md を参照して全影響先を列挙する |
| Layer 2ファイルをえんまさ確認なしに更新する | 提案を出して停止する。実行はえんまさの承認後のみ |
| 「更新不要」ファイルを記載しない | 「このファイルは今回関係ない」ことを明示する |
| COCOSiL設計中枢の変更をAGENTS.mdに反映しない | P1は必ずAGENTS.md §0とcocosil-domain/SKILL.mdを更新対象に含める |
| UIコピーに禁止語が混入する | 更新案生成前に禁止語チェックを実行する |
